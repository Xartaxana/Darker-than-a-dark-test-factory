# /triage — разбор падений прогона

Проведи падения через цепочку агентов.

1. Определи целевой прогон: из `$ARGUMENTS` (например `RUN-20260702-0130`) или возьми
   самый свежий `runs/RUN-*.md` со `status: NeedsTriage`.
2. Запусти **failure-analyst** (Task) — он вынесет вердикт по каждому падению
   (`APP_BUG/TEST_BUG/SITE_CHANGED/ENV_ISSUE/FLAKY`) и переведёт прогон в `Triaged`.
3. По вердиктам маршрутизируй (можно передать оркестратору или запустить точечно):
   - `APP_BUG` → **bug-reporter** (создать/прилинковать баг, без правок кода приложения);
   - `TEST_BUG`/`SITE_CHANGED` → **test-maintainer** (починить тест/селекторы/recordings);
   - `ENV_ISSUE` → сообщи пользователю про окружение (не вина приложения);
   - `FLAKY` → предложи карантин (`@pytest.mark.quarantine`).
4. Когда все пункты закрыты (баг заведён / тест починен / карантин) — прогон в `Closed`.
5. Покажи пользователю сводку: вердикты, заведённые баги, починенные тесты, что осталось.

Границы: никакой правки `app-under-test/`. Спорные случаи — `Blocked` для человека.
