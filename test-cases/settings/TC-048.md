# TC-048 — WebView dark mode требует холодного рестарта

## Предусловия
- Приложение запущено с чистыми данными, тема = Light, на вкладке Browse открыта
  страница AO3 (WebView содержимое видно светлым).

## Сценарий (Given-When-Then)

**Given** приложение в светлой теме, страница AO3 в WebView отображается светлой

**When** пользователь переключает тему на Dark в Settings и возвращается на экран
Browse **без** перезапуска приложения

**Then** содержимое WebView (сама веб-страница AO3) остаётся в прежней светлой
цветовой схеме — не темнеет мгновенно (нативные Compose-элементы вокруг WebView уже
тёмные — см. TC-047, но это не является предметом assert здесь)

**When** далее приложение принудительно останавливается (`adb shell am force-stop`) и
запускается заново

**Then** после холодного старта содержимое WebView отображается в тёмной цветовой
схеме (`ALGORITHMIC_DARKENING`/`FORCE_DARK` применены с начала загрузки страницы)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Исходная тема | Light |
| Целевая тема | Dark, выбрана без рестарта |
| Способ рестарта | `adb shell am force-stop` + `am start` |

## Заметки для автоматизации
- Двухфазный Given-When-Then оправдан тем, что это один связный сценарий про один
  и тот же переключатель — "не применяется сразу" и "применяется после рестарта" это
  две стороны одного наблюдаемого факта (таймингового контракта), не два независимых
  сценария; если ревьюер сочтёт нужным разбить на два файла — не возражать, но на
  взгляд test-designer это один тест-кейс с двумя точками наблюдения одного факта.
- Ссылается на известную хрупкость дизайна (app-under-test/CLAUDE.md checklist пункт
  3 "Kill and restart app in dark mode → pages load dark") — этот кейс формализует
  именно то ручное проверочное действие в тест-кейс.
- Не проверять `applyDayNight()`/`AppCompatDelegate` напрямую — только наблюдаемый
  визуальный результат страницы.
