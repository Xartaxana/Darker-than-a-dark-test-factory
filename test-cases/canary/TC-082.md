# TC-082 — Save filter button: инъекция + идемпотентность (live)

## Предусловия
- Приложение запущено с чистыми данными, режим **live**.
- Открыта форма AO3 Sort & Filter (search/tag browse), форма ещё скрыта
  (AO3 показывает её по клику toggle-ссылки).

## Сценарий (Given-When-Then)

**Given** приложение запущено live, страница с формой Sort & Filter
загружена, форма изначально скрыта

**When** пользователь раскрывает форму (клик по toggle-ссылке AO3, класс/
стиль `#work-filters` меняется — срабатывает `MutationObserver`)

**Then** сразу после submit-кнопки формы (`input[name="commit"][type="submit"]`)
появляется РОВНО ОДНА кнопка `[data-ao3-save-profile]` с текстом "Save filter"
**And** повторное скрытие/раскрытие формы (ещё один toggle, повторный вызов
`injectSaveFilterButton` через `MutationObserver`) НЕ создаёт вторую кнопку —
элемент остаётся ровно один

**Инвариант:** идемпотентность инъекции — свойство, верное для ЛЮБОГО числа
срабатываний `MutationObserver`/повторных вызовов `injectSaveFilterButton`
(guard `|| document.querySelector('[data-ao3-save-profile]')) return`), не
факт «кнопка появилась один раз при одном конкретном клике».

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Форма | реальная AO3 Sort & Filter (search/tag browse) |
| Селектор | `[data-ao3-save-profile]` (`framework/web/selectors.py::SAVE_PROFILE_BTN`) |

## Заметки для автоматизации
- Не дублирует TC-040 (функциональность: клик сохраняет `FilterProfile` с
  именем) — здесь предмет проверки СЕЛЕКТОР/идемпотентность инъекции кнопки,
  а не факт сохранения. TC-040 сам отмечает эту границу («это ближе к
  canary») — данный кейс закрывает её.
- **Маркер (изменено AT-BUG-026, 2026-07-22): `@pytest.mark.p1
  @pytest.mark.live`** (был `p0`). Приоритет демотирован P0 → P1 —
  подтверждённый qemu-краш (`0xc0000005`) под тяжёлым live-рендером этой
  страницы повторяемо ронял ВЕСЬ эмулятор в блокирующем `-m p0` прогоне
  (witness — `bugs/AT-BUG-024.md` D1-верификация fix-verifier 2/2,
  `bugs/AT-BUG-026.md`). Идемпотентный контракт остаётся детерминированно
  покрыт сиблингом TC-083 (replay, остаётся в `p0`) — риск-покрытие R-02 НЕ
  теряется; теряется только «каждый коммит обязан пережить LIVE-рендер этой
  тяжёлой страницы». Canary-ценность (дрейф реальной разметки AO3)
  сохраняется прогоном в p1/regression на своём расписании.
- **GPU-конфиг при прогоне ЭТОГО теста отдельно/в p1 (AT-BUG-021, решение
  Lead 2026-07-19, переоценено AT-BUG-026):** `AO3_EMU_GPU=host` показал
  0 крашей в узком соук-тесте именно этих 3 live-тестов (AT-BUG-021, 26
  вызовов), рекомендуется для точечного прогона ИМЕННО TC-082/078/080.
  AT-BUG-026 (2026-07-22) проверил `-gpu host` на ПОЛНОМ `-m p0` (46
  тестов, смешанный live+replay) — только 1 из 4 прогонов чистый; в одном
  прогоне устройство пропало БЕЗ характерной сигнатуры `0xc0000005` (Get-Device
  → NO DEVICE, но Windows Event Log — 0 событий Application Error за окно
  инцидента, позитивный контроль запроса подтверждён) — новый,
  неохарактеризованный класс, не тождественный swiftshader-крашу. Поэтому
  `host` НЕ применён как дефолт всей p0-сессии — только для точечных
  прогонов этого теста. Полный разбор — `bugs/AT-BUG-026.md`.
- Демотирован из p0 без прохода test-reviewer (правка приоритета — не
  переход статуса `schemas/test-case.schema.yaml`, `priority` — открытое
  поле enum, не гейтится `transitions.yaml`); ПОМЕЧЕНО для прохода
  test-strategist — демотирование канареечного P0 из блокирующего гейта
  меняет риск-покрытие R-02 на уровне стратегии, решение test-maintainer
  ограничено фактом «TC-083 детерминированно несёт тот же контракт»,
  окончательная оценка — за test-strategist/Lead.
- Сиблинг-кейс TC-083 — тот же контракт в replay
  (`sort_filter_form.mitm`, уже Verified в AT-BUG-006, где сам факт инъекции
