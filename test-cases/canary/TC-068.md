# TC-068 — Селектор work-блёрба + work id на живом листинге (live)

## Предусловия
- Приложение запущено с чистыми данными, режим **live**.
- Открыта реальная листинговая страница AO3 (search/tag browse), содержащая
  хотя бы одну работу.

## Сценарий (Given-When-Then)

**Given** приложение запущено live и открыта реальная листинговая страница
AO3 с ≥1 работой

**When** страница полностью загружается и bridge выполняет начальный проход
инъекции (`document.querySelectorAll('li[id^="work_"].work.blurb')`, стр.853
ao3_bridge.js)

**Then** каждый элемент, найденный этим селектором, имеет `id`, начинающийся
с `work_`, из которого извлекается непустой числовой work id (`li.id.replace('work_', '')`)
**And** количество найденных элементов ≥1 и совпадает с количеством видимых
work-заголовков (`h4.heading`) на странице (селектор не промахивается мимо
части работ и не ловит посторонние `<li>`)

**Инвариант:** контракт «каждый рендеренный work-элемент листинга адресуется
парой (селектор `li[id^="work_"].work.blurb`, извлекаемый work id)» верен для
ЛЮБОГО количества и состава работ на странице — это предпосылка всех
bridge-фич листинга (rate/note/tag-кнопки, фильтрация), не свойство одной
конкретной работы.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Страница | реальная листинговая страница AO3 (произвольный search/tag browse) |
| Селектор | `li[id^="work_"].work.blurb` (`framework/web/selectors.py::WORK_BLURB`) |

## Заметки для автоматизации
- Проверка через `execute_script` (получить массив `[id, count]` одним вызовом)
  дешевле, чем перебор Appium-локаторов по каждому элементу.
- Сверка "количество = количество h4.heading" — простая perf-дружественная
  проба того, что селектор не занижает/завышает набор (например, из-за
  рекламных/promoted блоков листинга AO3, которые НЕ являются `li.work.blurb`,
  но могут визуально выглядеть похоже).
- Маркер: `@pytest.mark.p0 @pytest.mark.live`.
- Сиблинг-кейс TC-069 — тот же контракт в replay-режиме.
