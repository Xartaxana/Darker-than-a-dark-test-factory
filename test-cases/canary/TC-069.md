# TC-069 — Селектор work-блёрба + work id на replay-листинге (replay)

## Предусловия
- Приложение запущено с чистыми данными, режим **replay**
  (`framework/data/recordings/listing_basic.mitm` — все 5 эталонных работ
  `framework/data/works.py::ALL`).
- Открыта записанная листинговая страница этой фикстуры.

## Сценарий (Given-When-Then)

**Given** приложение запущено в replay-режиме и открыта записанная
листинговая страница `listing_basic.mitm`

**When** страница полностью загружается и bridge выполняет начальный проход
инъекции

**Then** найдены ровно 5 элементов `li[id^="work_"].work.blurb`, с id
`work_900000001` .. `work_900000005` (соответствуют `ALL` из
`framework/data/works.py`), и извлечённые work id совпадают с `ao3_id`
эталонных работ

**Инвариант:** тот же контракт, что TC-068 (селектор+work id верны для
любого набора работ), проверенный здесь на детерминированном фиксированном
наборе — регрессионный якорь, не зависящий от live-контента.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Фикстура | `framework/data/recordings/listing_basic.mitm` |
| Работы | `LOVED`(900000001), `KUDOSED`(900000002), `READ`(900000003), `PENDING`(900000004), `DISLIKED`(900000005) |

## Заметки для автоматизации
- Фикстура уже существует и Verified-путём подтверждена рабочей (AT-BUG-004);
  инфраструктурных блокеров нет.
- Проверять ТОЧНОЕ множество id (не просто count>0) — детерминизм фикстуры
  это позволяет и делает пробу строже, чем TC-068 (live не может гарантировать
  состав).
- Маркер: `@pytest.mark.p0 @pytest.mark.replay`.
