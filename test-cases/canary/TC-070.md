# TC-070 — Rate-кнопка на неоценённом блёрбе (live)

## Предусловия
- Приложение запущено с чистыми данными (ни одна работа не оценена), режим
  **live**.
- Открыта реальная листинговая страница AO3 с ≥1 работой.

## Сценарий (Given-When-Then)

**Given** приложение запущено live с чистыми данными и открыта реальная
листинговая страница AO3

**When** страница полностью загружается и bridge выполняет начальный проход
инъекции

**Then** КАЖДЫЙ элемент `li[id^="work_"].work.blurb` содержит ровно один
элемент `[data-ao3-btn-wrap]`, а внутри него — ровно один `[data-ao3-rate-btn]`
(28px круглая кнопка), с `data-ao3-rate-btn` == work id родительского блёрба
**And** для неоценённой работы кнопка визуально в состоянии "неоценено" (фон
прозрачный, иконка "+", не заполненная BADGE-палитрой — см. `makeRateButton`/
`updateRateButton`, `s = currentRating ? BADGE[currentRating] : null`)

**Инвариант:** свойство «у каждого work-блёрба есть ровно одна собственная
Rate-кнопка, привязанная к ЕГО work id (не общая на страницу, не
перепутанная между блёрбами)» верно для любого количества работ на листинге
— DOM-контракт, не пример на одной работе.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Страница | реальная листинговая страница AO3 (произвольный search/tag browse) |
| Селектор | `[data-ao3-rate-btn]` (`framework/web/selectors.py::RATE_BUTTON`) |

## Заметки для автоматизации
- Проверка «кнопка привязана к СВОЕМУ work id» — сверить атрибут
  `data-ao3-rate-btn` каждой кнопки со значением work id родительского
  `li[id^="work_"]` (не полагаться на порядок DOM).
- Состояние "неоценено" — `background-color` кнопки прозрачный
  (`rgba(0, 0, 0, 0)`), см. `ListingPage.badge_for` из AT-BUG-004
  (class-fix: бейдж = цвет самой Rate-кнопки, не отдельный элемент
  `[data-ao3-badge]`, который никогда не создаётся).
- Маркер: `@pytest.mark.p0 @pytest.mark.live`.
- Сиблинг-кейс TC-071 — тот же контракт в replay; TC-072/073 — состояние
  "оценено" (badge) — другой Then, отдельный кейс.
