# TC-072 — Бейдж рейтинга = цвет Rate-кнопки (live)

## Предусловия
- Приложение запущено с чистыми данными, режим **live**.
- Открыта реальная листинговая страница AO3 с ≥1 работой.

## Сценарий (Given-When-Then)

**Given** приложение запущено live, открыта реальная листинговая страница AO3
с работой W (Rate-кнопка видна, не оценена)

**When** пользователь нажимает Rate-кнопку работы W и выбирает рейтинг R в
открывшемся `RatingOverlay`

**Then** после закрытия overlay та же `[data-ao3-rate-btn]` работы W (без
перезагрузки страницы, через `applyRatings`→`updateRateButton`) получает
непрозрачный `background-color`, равный `BADGE[R].bg`, и бордер, равный
`BADGE[R].border` — это и есть наблюдаемый "бейдж" контракта bridge (НЕ
отдельный DOM-элемент — см. class-fix AT-BUG-004, `[data-ao3-badge]` никогда
не создаётся)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | первая видимая работа реального листинга (id не важен — контракт не завязан на конкретную работу) |
| Рейтинг R | любой из 5 (Loved/Liked/Read/Pending/Disliked) — фиксировать один для детерминизма прогона |

## Заметки для автоматизации
- Это НЕ дублирует TC-009/010/011 (функциональность "рейтинг сохраняется и
  обновляет бейдж" уже покрыта там) — здесь предмет проверки СЕЛЕКТОР/CSS-
  контракт бейджа против живого DOM, а не факт сохранения рейтинга.
- Сверка "непрозрачный" — `background-color` до != `rgba(0, 0, 0, 0)` (проще и
  устойчивее, чем сверять точный hex — живой AO3/тема light/dark не влияют на
  сам факт закраски).
- Не требует известного `ao3_id` — работает на ЛЮБОЙ видимой работе живого
  листинга, поэтому не упирается в блокер TC-009 (синтетический id недостижим
  на live).
- Маркер: `@pytest.mark.p0 @pytest.mark.live`.
- Сиблинг-кейс TC-073 — тот же контракт в replay (детерминированный, через
  сидинг рейтинга вместо live-тапа).
