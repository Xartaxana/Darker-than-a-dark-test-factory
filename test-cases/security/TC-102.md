# TC-102 — JS-bridge exposure: доверенный origin vs не-AO3-контент

## Предусловия
- Приложение запущено с чистыми данными, открыта вкладка Browse на AO3 root
  (`HOME_URL`) — `ao3_bridge.js` уже инжектирован (`browse-bridge-injection`).
- Для не-AO3-состояния переиспользуется механизм TC-046 (кастомная error page
  приложения): переход на заведомо недоступный `.test`-домен даёт
  детерминированный `NXDOMAIN` без зависимости от живой сети/mitm-записи,
  открывается `onReceivedError` → локальная error-страница на `about:blank`
  base URL (НЕ AO3-контент).

## Сценарий (Given-When-Then)

**Given** активная вкладка отображает доверенный AO3-origin контент (`HOME_URL`)

**When** в контексте WebView этой вкладки выполняется JS-проба `typeof
window.Android` — снимается baseline на доверенном origin
**And** та же вкладка переводится на заведомо недоступный URL (`.test`
TLD, тот же приём, что TC-046), в результате чего показывается кастомная
error-страница приложения (не AO3-контент, не сетевой AO3-домен) — в этом
контексте та же JS-проба `typeof window.Android` выполняется повторно

**Then** на доверенном AO3-origin `window.Android` доступен (`!==
'undefined'`) — подтверждает штатную функциональную интеграцию (это baseline,
не сам предмет риска)
**And** фактическая доступность `window.Android` на локальной error-странице
(не-AO3-контент) зафиксирована как наблюдаемый факт: нативный `@JavascriptInterface`
(добавленный `addJavascriptInterface(bridge, "Android")` при создании WebView)
по семантике Android НЕ скоупится по origin сам по себе — если проба
показывает объект доступным и здесь, это ожидаемый по документации Android
результат и одновременно точный предмет minimum-smoke этой области (мотивация
P1: «@JavascriptInterface на удалённом AO3-origin — активная поверхность
приватности», docs/01-test-strategy.md §9); кейс документирует наблюдаемый
факт, не выносит сам вердикт «уязвимость»/«не уязвимость» (E4-min — smoke, не
полный аудит, §8)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| JS-проба | `typeof window.Android` (не `'undefined'` = доступен) |
| Доверенный контекст | вкладка на `HOME_URL` (archiveofourown.org) |
| Не-AO3 контекст | локальная error-страница приложения (`about:blank` base URL, недоступный `.test`-URL) |

## Заметки для автоматизации
- Блокера нет: `driver.execute_script(...)` в WebView-контексте — стандартный
  Appium-примитив уже задействован во фреймворке для JS-проб (см. паттерны в
  `core/contexts.py`); механизм error-страницы (недоступный `.test`-URL →
  `NXDOMAIN` → кастомная страница) уже реализован и автоматизирован TC-046
  (`browser_steps.open_unreachable_url` и связанные шаги) — переиспользовать,
  не изобретать новый способ попасть на не-AO3-контент.
- **Testability gap (важное уточнение, чтобы не спутать два разных механизма).**
  `window.Android` (нативный `@JavascriptInterface`-объект из Kotlin
  `addJavascriptInterface`) — это НЕ то же самое, что `ao3_bridge.js` (JS-файл,
  инжектируемый скриптом только `onPageFinished` для AO3-страниц,
  `browse-bridge-injection`). Эта проба целится именно в нативный интерфейс.
  Полная инспекция WebSettings/кода вызова `addJavascriptInterface`
  (использован ли `originAllowList`/подобная фильтрация) НЕ читается black-box
  через UiAutomator2 — штатно доступна только эта поведенческая проба, как
  явно названо в docs/01-test-strategy.md §9 area E4. Отрицательный результат
  пробы (bridge недоступен на error-странице) доказывает лишь «в ЭТОМ прогоне
  на ЭТОЙ error-странице недоступен» — не исчерпывающее доказательство
  origin-scoping для ЛЮБОГО не-AO3-контента (напр. iframe стороннего домена
  внутри AO3-страницы этим кейсом не проверяется — вне минимального смоук-скоупа).
