# TC-064 — Сортировка Author (A–Z): алфавит + пустой author в конце

## Предусловия
- Приложение запущено, в Room засеяны 3 работы с одним рейтингом через
  `seed_db.seed()`/`seed_with_comment` (произвольные `Work`-инстансы, НЕ
  обязательно из `works.py::ALL`):
  - W1: `author="Zoe Martinez"`
  - W2: `author="Amy Chen"`
  - W3: `author=""` (пустой автор — допустим схемой, `WorkRating.author: String`
    NOT NULL, но пустая строка проходит)
- Достаточно работ/филлеров для видимого скролла на вкладке (переиспользовать
  `SCROLL_FILLERS` с непустым `author`, как в TC-030/063).
- Открыт экран Library, вкладка проскроллена вниз, активна дефолтная сортировка
  Last read.

## Сценарий (Given-When-Then)

**Given** вкладка Library содержит W1 (author "Zoe Martinez"), W2 (author "Amy
Chen"), W3 (author пустой) и филлеры с непустым author; список проскроллен вниз

**When** пользователь открывает dropdown сортировки в top bar и выбирает "Author
(A–Z)"

**Then** среди работ с непустым author порядок лексикографический по возрастанию,
без учёта регистра: Amy Chen (W2) раньше Zoe Martinez (W1)
**And** W3 (пустой author) расположена ПОСЛЕДНЕЙ в списке, несмотря на то что пустая
строка лексикографически меньше любого непустого значения (наивное сравнение
поставило бы её первой) — специальный случай "есть значение" как первичный ключ
сортировки, аналогично TC-031 для отсутствующего word_count
**And** список визуально возвращён к началу (скролл сброшен) — обобщение TC-030 на
сортировку по автору

**Инвариант:** сортировка не меняет множество работ вкладки, только порядок (банк);
работы с пустым `author` всегда уходят в конец списка независимо от направления
сравнения (аналог TC-031, где в конец уходят работы без `word_count`); смена
сортировки сбрасывает скролл к началу (обобщение TC-030).

## Проверяемые данные
| Параметр | Значение |
|---|---|
| W1.author | `"Zoe Martinez"` |
| W2.author | `"Amy Chen"` |
| W3.author | `""` (пустая строка) |
| Сортировка | Author (A–Z) |

## Заметки для автоматизации
- Работы с произвольным `author=""` не входят в `framework/data/works.py::ALL` —
  создать отдельные `Work(...)`-инстансы прямо в тесте (dataclass публичный,
  конструктор принимает любые значения полей) и засеять через `seed()`/
  `seed_with_comment` с одним общим рейтингом. Блокера нет — то же самое уже делает
  TC-031 для `word_count=None` (`Work` с явным полем в обход `ALL`).
- Сравнение регистронезависимое (`author.lowercase()`) — можно намеренно взять
  авторов с разным регистром первой буквы для дополнительной проверки, но это
  усложнило бы один сценарий двумя фактами; не делать, если не требуется отдельным
  кейсом.
- Не путать с TC-029 (фильтр по фандому — другое поле, не сортировка).
