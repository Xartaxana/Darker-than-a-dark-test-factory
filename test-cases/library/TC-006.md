# TC-006 — Подписи вкладок Library

## Предусловия
- Приложение запущено с чистыми данными (сидинг работ не требуется — набор
  вкладок статический: `val tabs = LibTab.entries`, `LibraryScreen.kt:123`,
  список не зависит от содержимого библиотеки).
- Открыт экран Library.

## Сценарий (Given-When-Then)

**Given** приложение запущено (чистые данные, сидинг не требуется) и открыт
экран Library

**When** пользователь смотрит на панель вкладок

**Then** подписи вкладок слева направо, верхний регистр:
`FAVORITE · KUDOSED · READ · PENDING · DISLIKED · FILES`

## Проверяемые данные
| Вкладка (LibTab) | `label` в коде | Рендер (`.uppercase()`) |
|---|---|---|
| SUPER_LIKE | `"Favorite"` | FAVORITE |
| LIKE | `"Kudosed"` | KUDOSED |
| READ | `"Read"` | READ |
| PENDING | `"Pending"` | PENDING |
| DISLIKE | `"Disliked"` | DISLIKED |
| DOWNLOADED | `"Files"` | FILES |

Источник: `app/src/main/java/com/example/ao3_wrapper/ui/library/LibraryScreen.kt:71-78`
(`private enum class LibTab(val label: String, ...)`) и `:207`
(`Text(text = tab.label.uppercase(), ...)` внутри `Tab { }` — комментарий кода на
`:196-199` прямо называет "FAVORITE", "DISLIKED", "PENDING" как известные
длинные подписи, что подтверждает верхний регистр как продуктовое, не
случайное поведение).

## Заметки для автоматизации
- Кейс — регрессионный якорь точных строк подписей, зафиксированных решением
  владельца 2026-07-17 (эталон = код, не устаревший `PROJECT.md`). Блокеров
  нет: сидинг не нужен, replay-фикстура листинга не нужна (экран Library —
  нативный Compose UI, не WebView).
- Порядок вкладок — порядок объявления `LibTab.entries` (SUPER_LIKE, LIKE,
  READ, PENDING, DISLIKE, DOWNLOADED); сверять и порядок, не только состав.
