# TC-062 — Сортировка Last read по умолчанию: самая недавно обновлённая работа первая

## Предусловия
- Приложение запущено, в Room засеяны 3 работы с ОДНИМ рейтингом, добавленные ТРЕМЯ
  ПОСЛЕДОВАТЕЛЬНЫМИ вызовами `seed_db.seed_with_comment` (по одной работе на вызов,
  НЕ одним батчем — см. «Заметки для автоматизации», почему это существенно),
  в хронологическом порядке W1 → W2 → W3 (W3 засеяна последней, у неё самый свежий
  `timestamp`).
- Заголовки подобраны так, чтобы НИ порядок вставки, НИ алфавитный порядок не
  совпадали с ожидаемым порядком по `timestamp`: W1="Mango Work" (вставлена первой,
  самый старый `timestamp`), W2="Apple Work" (вставлена второй), W3="Zebra Work"
  (вставлена последней, самый свежий `timestamp`). Порядок вставки —
  Mango→Apple→Zebra; алфавитный порядок — Apple→Mango→Zebra; ожидаемый порядок по
  `timestamp` (см. Then) — Zebra→Apple→Mango: не совпадает ни с одним из первых
  двух, поэтому совпадение результата с любым из них исключает случайное
  прохождение теста по стечению данных.
- Открыт экран Library, соответствующая вкладка. Сортировка НЕ выбиралась вручную —
  проверяется дефолтное состояние `LibrarySort.LAST_READ`.

## Сценарий (Given-When-Then)

**Given** на вкладке Library 3 работы: Mango Work (засеяна первой, самый старый
`timestamp`), Apple Work (второй), Zebra Work (последней, самый свежий
`timestamp`); сортировка в top bar не менялась пользователем

**When** пользователь открывает вкладку Library (без действий с dropdown сортировки)

**Then** список показан в порядке Zebra Work, Apple Work, Mango Work — от самой
недавно обновлённой к самой старой
**And** этот порядок НЕ совпадает ни с порядком вставки (Mango, Apple, Zebra), ни с
алфавитным (Apple, Mango, Zebra) — совпадение результата с любым из них
опровергло бы гипотезу «сортировка идёт по `timestamp`»

**Инвариант:** сортировка не меняет множество работ вкладки, только порядок (банк:
«сортировка не меняет множество работ, только порядок»); Last read упорядочивает по
убыванию `timestamp` последнего обновления строки (сидинг/смена рейтинга/заметки —
не факт прочтения в буквальном смысле), а не по порядку вставки или алфавиту
заголовков — дефолтная сортировка приложения без явного выбора пользователя.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работы | 3, один рейтинг, засеяны тремя отдельными вызовами `seed_with_comment` |
| Порядок вставки (хронология) | Mango Work → Apple Work → Zebra Work |
| Ожидаемый порядок в списке | Zebra Work, Apple Work, Mango Work |

## Заметки для автоматизации
- **Важно (не блокер, но неочевидный нюанс сидинга):** `framework/data/seed_db.py`
  — `_insert_rows`/`_insert_rows_full` вычисляют `now = int(time.time() * 1000)`
  ОДИН РАЗ на весь список аргумента (строки 74/98), поэтому все строки ОДНОГО
  вызова `seed()`/`seed_with_comment()` получают ОДИНАКОВЫЙ `timestamp` — сортировка
  между ними станет неопределённой (равенство ключа сортировки). Чтобы получить
  РАЗЛИЧАЮЩИЕСЯ `timestamp`, каждую работу нужно засевать ОТДЕЛЬНЫМ вызовом
  `seed_with_comment([(work, rating, None, None)])` в нужном хронологическом
  порядке — существующие функции это уже поддерживают (повторный вызов пуллит
  текущую БД устройства, `INSERT OR REPLACE` не трогает ранее вставленные строки
  других `ao3Id`, только добавляет/обновляет переданную), поэтому НЕ заводится
  test_debt-баг — это особенность композиции существующих шагов, а не отсутствующая
  инфраструктура. Три последовательных вызова (force_stop+pull+push каждый) дают
  реальный интервал в десятки-сотни мс между `timestamp` — коллизия на одной
  миллисекунде маловероятна, но если test-automator столкнётся с флаки на этой
  почве, стоит добавить явную микро-паузу между вызовами в шаге, а не менять
  сборку сидинга.
- Данные (Mango/Apple/Zebra) намеренно подобраны так, чтобы ни порядок вставки, ни
  алфавит не совпадали с ожидаемым результатом (см. «Предусловия») — сама эта
  несовпадаемость и есть проверка, отдельного assert на неё заводить не нужно:
  достаточно одного assert на итоговый порядок списка.
- Не путать с TC-016 (смена рейтинга перемещает work между вкладками — другой
  фокус) — здесь рейтинг у всех трёх одинаков и не меняется, наблюдается только
  порядок внутри одной вкладки.
