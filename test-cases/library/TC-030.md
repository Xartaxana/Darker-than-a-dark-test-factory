# TC-030 — Сортировка Word count high-to-low и сброс скролла

## Предусловия
- Приложение запущено, в Room засеяны 5 работ с одним рейтингом и разным word_count
  (300/800/1500/4200/12000), достаточно для видимого скролла на вкладке (можно
  досеять дополнительные работы для гарантированной прокрутки списка).
- Открыт экран Library, вкладка проскроллена вниз, активна дефолтная сортировка
  Last read.

## Сценарий (Given-When-Then)

**Given** вкладка Library содержит работы с разным word_count, список проскроллен
вниз (не на самом верху)

**When** пользователь открывает dropdown сортировки в top bar и выбирает "Word count"
(иконка trend-down, high to low)

**Then** список переупорядочен по убыванию word_count (12000, 4200, 1500, 800, 300)
**And** список визуально возвращён к началу (скролл сброшен на верх), а не остаётся на
прежней позиции
**And** иконка-триггер в top bar сменилась на иконку выбранной сортировки

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работы | `ALL` из `framework/data/works.py`, один рейтинг |
| word_count values | 300, 800, 1500, 4200, 12000 |
| Сортировка | Word count (high to low) |

## Заметки для автоматизации
- Сброс скролла — ключевая проверяемая деталь риска R-08/R-06 пересечения по UX;
  проверять через позицию первого видимого элемента списка (`LazyListState` не
  доступен из UI-теста напрямую — сравнивать видимый текст/id первой карточки до и
  после).
- Сортировка hoisted в MainActivity и не персистится между сессиями — не проверяется
  здесь (см. отдельно, если понадобится P2-кейс на персистентность выбора сортировки
  после рестарта — сейчас не описан в §9, не заводить самовольно).
- Триггер сортировки icon-only, без текстовой подписи выбранного варианта —
  локатор по content-description/иконке, свериться с живым деревом перед
  автоматизацией.
