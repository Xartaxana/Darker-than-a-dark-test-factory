# TC-065 — Сортировка Rating на вкладке Files группирует скачанные работы по рангу

## Предусловия
- Приложение запущено, в Room засеяны 5 работ `framework/data/works.py::ALL`, у
  КАЖДОЙ заполнен `downloadPath` на реально существующий (для теста достаточно
  переиспользуемый) HTML-файл через `seed_db.seed_with_download` — тот же механизм,
  что уже используют TC-034/035/036/021.
- Рейтинги распределены по одному на работу — LOVED=SAVE, KUDOSED=LIKE, READ=READ,
  PENDING=PENDING, DISLIKED=DISLIKE — но ЗАСЕЯНЫ (порядок вызова
  `seed_with_download`) в ОБРАТНОМ порядке относительно ожидаемого результата:
  DISLIKED, PENDING, READ, KUDOSED, LOVED — чтобы порядок отображения нельзя было
  объяснить порядком вставки/timestamp, только самим рангом рейтинга.
- Открыт экран Library, активна вкладка Files (Downloaded).

## Сценарий (Given-When-Then)

**Given** на вкладке Files видно 5 скачанных работ с разными рейтингами, засеянных
в порядке DISLIKED→PENDING→READ→KUDOSED→LOVED (обратном ожидаемому)

**When** пользователь открывает dropdown сортировки в top bar (доступен на вкладке
Files) и выбирает "Rating"

**Then** список переупорядочен по рангу рейтинга: LOVED (Favorite/SAVE), KUDOSED
(Kudosed/LIKE), READ (Read/READ), PENDING (Pending/PENDING), DISLIKED (Disliked/
DISLIKE) — именно в этом порядке, не в порядке вставки
**And** переключение на любую другую вкладку (например Favorite) и открытие
dropdown сортировки НЕ показывает "Rating" среди опций — опция специфична для
вкладки Files

**Инвариант:** сортировка не меняет множество скачанных работ вкладки Files, только
порядок (банк: «сортировка не меняет множество работ, только порядок»); Rating
группирует строго по рангу SAVE→LIKE→READ→PENDING→DISLIKE независимо от порядка
вставки/`timestamp`; опция Rating существует и осмысленна ТОЛЬКО на вкладке Files/
Downloads (`librarySortOptionsFor(isFilesTab)`) — на остальных вкладках недоступна.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работы | `ALL` из `framework/data/works.py`, каждая с `downloadPath` заполненным |
| Рейтинги | LOVED=SAVE, KUDOSED=LIKE, READ=READ, PENDING=PENDING, DISLIKED=DISLIKE |
| Порядок сидинга | DISLIKED, PENDING, READ, KUDOSED, LOVED (обратный ожидаемому) |
| Ожидаемый порядок после сортировки | LOVED, KUDOSED, READ, PENDING, DISLIKED |

## Заметки для автоматизации
- Сидинг — `seed_db.seed_with_download(rows)`, `rows: list[tuple[Work, rating,
  local_html_path]]` — рейтинг ЗАДАЁТСЯ НЕЗАВИСИМО на строку (`_insert_rows_with_
  download` требует валидный `rating` на каждую, `None` не поддерживает — здесь и
  не нужен). Один и тот же локальный HTML-файл можно переиспользовать для всех 5
  строк (содержимое файла не проверяется этим кейсом, важен только сам факт
  `downloadPath != null`) — по образцу `downloaded_work_seeded` в `conftest.py`.
  Блокера нет.
- Второй Then-пункт (опция Rating недоступна вне Files) — дешёвая, но отдельная
  проверка UI dropdown; если при автоматизации оценка стоимости велика
  (потребуется переключение вкладки внутри уже собранного теста), можно вынести в
  отдельный лёгкий assert того же теста, не отдельный TC — это одна связная
  проверка того же инварианта «Rating осмысленна только на Files», не новый
  сценарий.
- Порядок сидинга специально обратный ожидаемому результату (см. «Предусловия») —
  это и есть защита от случайного совпадения с порядком вставки; отдельного assert
  не требует.
- Не путать с TC-016/017 (смена/сброс рейтинга и его влияние на вкладки — другой
  фокус, там нет сортировки).
