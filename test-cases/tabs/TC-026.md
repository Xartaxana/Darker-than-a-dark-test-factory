# TC-026 — Long-press ссылки открывает фоновую вкладку

## Предусловия
- Приложение запущено с чистыми данными.
- Открыта одна вкладка на листинговой странице AO3 с как минимум одной ссылкой на
  работу (напр. записанная replay-страница листинга).

## Сценарий (Given-When-Then)

**Given** открыта одна вкладка на листинговой странице с видимой ссылкой на работу
(активная вкладка = текущая)

**When** пользователь делает long-press по ссылке на работу — фоновая вкладка
открывается СРАЗУ, без промежуточного context-меню (факт кода:
`BrowserScreen.kt:640-648`, `setOnLongClickListener` по `HitTestResult` зовёт
`viewModel.openTab(url, background = true)` напрямую; уточнено 2026-07-18 при
дроблении реестра, прежняя развилка «либо-либо» снята)

**Then** появляется новая вкладка в strip с URL этой работы
**And** активной остаётся исходная вкладка (листинг) — контент экрана не переключился
на новую вкладку
**And** новая (фоновая) вкладка доступна по тапу на её чип в strip и показывает
корректно загруженную страницу работы

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Исходная вкладка | листинговая страница с минимум 1 работой |
| Ссылка | первая работа в листинге |

## Заметки для автоматизации
- UI-механизм long-press установлен по коду (2026-07-18): нативный
  `setOnLongClickListener` на WebView с разбором `HitTestResult` —
  `BrowserScreen.kt:640-648`; context-меню НЕТ, открытие фоновое напрямую.
  Для автоматизации всё ещё может потребоваться `W3C Actions` long-press или
  JS-эмуляция touch-hold — подобрать по живому поведению.
- Второй по значимости риск здесь — не пропустить фактическое переключение
  активности: assert на "исходная вкладка осталась активной" через видимый контент
  экрана (не просто через наличие новой вкладки в strip).

**Заблокировано (test-automator, разведка 2026-07-18) — см. `bugs/AT-BUG-018.md`.**
Координатное отображение CSS px ссылки → экранные px найдено и ДВАЖДЫ подтверждено
эмпирически (короткий тап по вычисленной точке надёжно навигирует по нужной ссылке).
Сам синтетический LONG-PRESS поверх WebView-контента через доступные механизмы Appium/
UiAutomator2 (`mobile: longClickGesture` по x/y, сырые W3C Actions, `adb shell input
swipe` тем же началом/концом) — ненадёжен: из 12 попыток (3 механизма × повторы,
включая 6 подряд без перезагрузки страницы) фоновая вкладка появилась 1 раз (<10%).
