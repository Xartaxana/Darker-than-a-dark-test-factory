# TC-012 — Синхронизация бейджа при нескольких вхождениях одной работы на странице

## Предусловия
- Приложение запущено с чистыми данными.
- Открыта листинговая страница (replay-фикстура), сконструированная так, чтобы
  одна и та же работа W встречалась дважды на странице (например, в основном
  списке и в блоке "related works"/повторный блёрб — либо синтетическая
  replay-страница, специально подготовленная framework/data для этого случая).

## Сценарий (Given-When-Then)

**Given** приложение запущено и открыта листинговая страница, где работа W
отображена в двух разных блёрбах на одной странице, оба без рейтинга

**When** пользователь ставит рейтинг «Read» через Rate-кнопку одного из двух
вхождений работы W

**Then** бейдж/стиль Rate-кнопки обновляется на «Read» у ОБОИХ вхождений работы W
на странице без перезагрузки

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | синтетическая работа, дублированная в replay-фикстуре листинга (два блёрба с одинаковым `ao3_id`) |

## Заметки для автоматизации
- Требует подготовки специальной replay-страницы с дублированным блёрбом — если
  такой фикстуры нет в `framework/data/recordings/`, завести её как часть
  автоматизации этого кейса (задача test-automator/data, не test-designer).
- Если техническая подготовка дублированной фикстуры окажется непропорционально
  дорогой, альтернативный наблюдаемый прокси: открыть одну и ту же работу в двух
  разных табах (мультитаб) и проверить синхронизацию бейджа панели/листинга между
  табами — но это уже пересекается с area=tabs (P1, вне скоупа этого прохода);
  в этом кейсе фиксируется исходная формулировка из §9 (два вхождения на одной
  странице).

- [x] Given описывает полное состояние, воспроизводимое фикстурами
