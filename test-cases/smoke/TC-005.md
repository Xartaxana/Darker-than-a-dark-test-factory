# TC-005 — Стабильность переключения тем

## Сценарий (Given-When-Then)
**Given** открыт экран Settings
**When** последовательно выбраны Dark, Light, System
**Then** экран Settings по-прежнему отрисован (приложение не упало)
**And** финально выбранный режим (System) сохранён как выбранный (SharedPreferences
`theme_mode`, тот же источник истины, что читает и Settings, и side panel при
следующем открытии — экран НЕ выставляет выбор темы через accessibility
selected/checked, см. заметку автоматизации). Глубокая проверка ПРИМЕНЕНИЯ темы
(визуальный эффект System/Dark/Light на UI и WebView) сюда не входит — она уже
покрыта TC-047/048/049/059, дублировать здесь не нужно.

## Заметки для автоматизации

**misc-batch-0722 (builder, 2026-07-22).** Усиление assert'а — решение Lead
по замечанию красной пробы 2026-07-22T01:19:44Z («assert кейса слабый —
"Settings отрисован" вместо явной проверки применённой темы»). Расхождение
со спекой батча, зафиксировано явно: спека предполагала проверку через
«selected/checked-состояние элемента темы» в UI Settings — эмпирически (код
+ живое дерево, `framework/screens/settings_screen.py` и комментарий
`framework/steps/settings_steps.py:116-124`) такого состояния не существует:
`ThemeModeRow`/`FontSizeRow` рисуют выбор ЦВЕТОМ ФОНА `TextButton`, не через
accessibility `selected`/`checked` (на живом дереве `selected="false"` у
ВСЕХ вариантов независимо от факта выбора). Единственный наблюдаемый
источник истины финального выбора — `SharedPreferences` (`theme_mode`),
общий для входа через Settings и через side panel — читается существующим
шагом `settings_steps.assert_theme_mode_pref`. Использован он, а не
UI-локатор selected/checked. Это НЕ дублирует глубокую проверку применения
темы (визуальный эффект — TC-047/048/049/059): здесь проверяется только
факт финального сохранённого выбора, не его визуальное воплощение.
