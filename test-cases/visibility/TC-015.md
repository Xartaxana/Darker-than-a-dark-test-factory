## Предусловия
- Приложение запущено, в Room засеяна работа W с `rating=DISLIKE`
  (`DISLIKED` из `framework/data/works.py`).
- В Settings → Content Visibility тумблер «Hide Disliked works» (per-rating
  `Switch`, строка ряда с `row.rating == Rating.DISLIKE`) включён — дефолтное
  состояние (`hiddenRatings = setOf(Rating.DISLIKE)` по умолчанию,
  `SettingsScreen.kt:66`), т.е. работа W изначально скрыта на листинге — то
  же исходное состояние, что в TC-013.
- Открыта листинговая страница (replay-фикстура), содержащая блёрб работы W.

## Сценарий (Given-When-Then)

**Given** приложение запущено, работа W засеяна с рейтингом Disliked
**And** в Settings тумблер «Hide Disliked works» включён (дефолт), работа W
скрыта на листинге

**When** пользователь в Settings выключает тумблер «Hide Disliked works»
(`viewModel.toggleHideRating(Rating.DISLIKE)`, `SettingsScreen.kt:749`), не
покидая уже открытую листинговую страницу с блёрбом работы W (приложение
проталкивает смену фильтра на открытую вкладку без ре-навигации —
`MainActivity.kt:169-171` → `BrowserViewModel.kt:664-668` →
`ao3_bridge.js:443-446` `window.setHiddenRatings`, который сразу вызывает
`applyAllFilters()`)

**Then** блёрб работы W отображается (видим) — `hiddenRatings` больше не
содержит `DISLIKE`, JS-бридж (`applyAllFilters`, вызванный live-push'ем сразу
при смене `window.__ao3HiddenRatings`) не исключает блёрб
**And** бейдж/цвет Rate-кнопки работы W по-прежнему отражает Disliked (визуальный
бейдж не зависит от тумблера скрытия, только видимость самого блёрба)

**Инвариант:** видимость блёрба на листинге определяется исключительно
членством `rating` работы в множестве `hiddenRatings`; выключение per-rating
тумблера для рейтинга R удаляет R из `hiddenRatings` — ни одна работа с этим
рейтингом впредь не исключается из рендера, независимо от того, что это тот
же самый переключатель, что в дефолтном состоянии её скрывал (TC-013).
Видимость блёрба и хранимый `rating` — независимые свойства: тумблер влияет
только на первое.

**Отличие от TC-013 (не дубль):** TC-013 фиксирует ДЕФОЛТНОЕ состояние
(тумблер включён "из коробки" → Disliked скрыт) — Given там не содержит
пользовательского действия над тумблером. TC-015 проверяет ОБРАТНОЕ
направление: явное пользовательское действие — выключение того же тумблера
— и что это действие меняет видимость (не только начальное состояние,
но и реакцию на toggle). Разные Given (дефолт vs дефолт+явный toggle-off),
разные When (нет действия vs явное действие), взаимодополняющие Then
(скрыт vs виден) — не перечисление одного и того же сценария.

## Заметки для автоматизации
- Тумблер под тестом — per-rating `Switch` в Settings → Content Visibility,
  строка ряда рейтинга Disliked (`SettingsScreen.kt:747-757`,
  `onCheckedChange = { viewModel.toggleHideRating(row.rating) }`); глобального
  "Enable filtering" тумблера в коде НЕТ (решение владельца 2026-07-17,
  `bugs/BUG-001.md`) — прежняя формулировка кейса ссылалась на
  несуществующий элемент, исправлено.
- Требует ту же replay-фикстуру листинга, что и TC-013, для сравнимости (тот
  же блёрб работы W, тот же `ao3_id`).
- Реальный путь — LIVE-PUSH, не page-load-only: приложение проталкивает смену
  фильтра на уже открытую вкладку без ре-навигации/reload. Сверено по коду:
  `MainActivity.kt:169-171` — `LaunchedEffect(hiddenRatings)` реагирует на
  смену `settingsUiState.hiddenRatings`; `BrowserViewModel.kt:664-668` —
  `setHiddenRatings` зовёт `evalJsAllTabs("window.setHiddenRatings(...)")`;
  `ao3_bridge.js:443-446` — `window.setHiddenRatings` пишет
  `__ao3HiddenRatings` и СРАЗУ вызывает `applyAllFilters()` на текущей
  странице. Автотест выключает тумблер в Settings, затем только возвращает
  фокус на уже открытую вкладку Browse (без повторной навигации листинга) и
  проверяет видимость блёрба — это и есть проверяемый продакшн-сценарий
  (пользователь выключил тумблер, не покидая уже открытый Browse).
