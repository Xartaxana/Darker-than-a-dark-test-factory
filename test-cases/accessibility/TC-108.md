# TC-108 — Contrast sanity: различимость текст/фон в dark и light

## Предусловия
- Приложение запущено с чистыми данными.
- Работа W засеяна (переиспользовать seed TC-009), страница работы открыта.
- Скриншот-примитив фреймворка доступен (тот же, что используют
  luma-проверки TC-009/TC-010/BUG-012).

## Сценарий (Given-When-Then)

**Given** приложение открыто на странице работы W

**When** тема приложения переключается в Dark (side panel toggle), делается
скриншот экрана; затем переключается в Light, делается второй скриншот

**Then** в ОБОИХ скриншотах текстовые зоны (bottom bar, заголовок работы)
различимы от фона по luma-прокси (тот же метод, что TC-009/TC-010: разница
luma текст/фон выше sanity-порога — не точный WCAG-ratio, см. testability gap
ниже)

**Инвариант:** свойство «текст различим от фона по luma-прокси» верно ДЛЯ
КАЖДОГО из двух системных вариантов темы (dark и light) независимо —
не только для варианта по умолчанию; кейс использует ту же тема-семью
(«темы»), что и стартовый банк инвариантов C4 («смена темы не меняет URL,
активную вкладку и сохранённые данные») — здесь проверяется соседнее
свойство той же семьи (различимость), не идентичное, но того же класса
«верно для всех членов семейства, не для одного примера».

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа | ao3_id из `data/works.py` (та же, что TC-009) |
| Зоны сверки | bottom bar, заголовок работы |
| Метод | luma-прокси скриншота (как TC-009/TC-010/BUG-012) |

## Заметки для автоматизации
- Блокера нет: переключение темы и скриншот — существующие примитивы
  (side panel toggle уже используется TC-047/TC-048/TC-049, luma-метод — уже
  используется TC-009/TC-010 и BUG-012 witness).
- **Testability gap (явно названо в docs/01-test-strategy.md §9 area E1):**
  точный WCAG-ratio (относительная яркость по формуле W3C, порог 4.5:1 для
  обычного текста) требует пиксельной выборки со скриншота + кастомного
  расчёта — вне штатного Appium-assert. Доступна ТОЛЬКО sanity-проверка
  (различимость bg/fg по грубому luma-порогу, тот же метод, что уже принят
  для rating-бейджей) — не формальное WCAG AA/AAA соответствие. Привязка к
  R-11 — та же хрупкая тема-зона (CLAUDE.md «Dark mode fragile» в
  app-under-test).
