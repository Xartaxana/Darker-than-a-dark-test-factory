# TC-039 — Restore + auto-scan показывают ровно один диалог

## Предусловия
- Приложение запущено, в Room пусто (после Clear all ratings / чистая установка).
- Подготовлен backup JSON-файл (`{"version":2,"works":[…],"filterProfiles":[…]}`) с
  одной работой, чей `ao3_id` совпадает с именем orphan-файла, заранее положенного в
  папку загрузок (аналогично фикстуре TC-038), но backup не содержит `downloadPath`
  (т.к. на момент backup файла на этом устройстве не было).
- SAF file picker доступен для выбора файла backup.

## Сценарий (Given-When-Then)

**Given** Library пуста, backup-файл содержит 1 работу, папка загрузок содержит
соответствующий orphan-файл

**When** пользователь выполняет "Restore from backup" и выбирает подготовленный файл

**Then** появляется ровно один диалог результата (`AlertDialog`), объединяющий и
результат restore (количество восстановленных работ), и результат скана (количество
перелинкованных файлов) — не два последовательных всплывающих диалога
**And** после закрытия диалога восстановленная работа в Library имеет заполненный
`downloadPath`, указывающий на orphan-файл

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Backup | 1 работа, без downloadPath |
| Orphan-файл | `..._<ao3_id>.html`, тот же ao3_id, что в backup |

**2026-07-15 — test-automator (кейс автоматизирован, статус остаётся Approved до
ревью):** SAF-блокер снят (`bugs/AT-BUG-005.md`, инкремент 1 — переиспользован без
доработки). Реализовано `framework/tests/test_downloads.py::
test_restore_folds_orphan_scan_into_single_dialog`. Точный текст объединённого
диалога сверен с кодом (`SettingsScreen.kt:1181-1200`, `ImportState.Done`):
«Restored 1 works, 0 filters. Also relinked 1 and added 0 downloaded file(s).» —
подтверждает, что restore- И scan-счётчики объединены в ОДИН диалог, а не просто
что диалог появился. После закрытия явно проверяется ОТСУТСТВИЕ второго диалога
(`settings_steps.assert_no_scan_complete_dialog`, короткое ожидание + проверка
отсутствия, не только присутствие первого — как того требует заметка ниже) — само
это структурно гарантировано кодом (`importFromUri` вызывает
`scanForOrphanedDownloads()` напрямую, не через `ScanDownloadsState`), assert это
доказывает поведенчески, не полагаясь на чтение исходников. После закрытия
восстановленная работа в Library показывает open-иконку (downloadPath заполнен).
3 зелёных прогона подряд (58.9s/51.3s/56.4s) + полный регресс
`test_downloads.py` (5/5) и `test_saf_infra_probe.py` (3/3), `arch_check.py` 0/0.

Побочная находка (задокументирована в `bugs/AT-BUG-005.md`, эта же дата, НЕ
блокирует этот кейс): `SettingsViewModel.setDownloadFolderUri` запускает
`scanForDownloads(silent=true)` АСИНХРОННО (`viewModelScope.launch`) — тап ALLOW в
SAF-пикере возвращает управление ДО того, как эта корутина гарантированно
проходит по каталогу. Если положить целевой orphan-файл в папку ДО её выбора (как
делает TC-038 для СВОЕГО сценария), для TC-039 это создаёт гонку: скан может
добавить stub-строку с тем же `ao3Id`, что и работа из backup, ДО того как Restore
успевает её импортировать — тогда `importFromUri` видит `ao3Id` уже существующим и
ПРОПУСКАЕТ работу как дубликат вместо восстановления (воспроизведено на первом
прогоне этого теста: «Restored 0 works, 0 filters (1 works already existed).»
вместо ожидаемого объединённого диалога). Обойдено на уровне теста decoy-файлом
(другой `ao3Id`, кладётся в папку ДО выбора) — его собственный диалог «Scan
complete» детерминированно доказывает, что асинхронная корутина уже отработала,
и только ПОСЛЕ его закрытия в ту же папку кладётся настоящий orphan-файл (см.
докстринг `restore_scan_workspace` в `test_downloads.py`).

- Ключевой наблюдаемый факт — именно ОДИН диалог, а не два подряд; assert должен
  явно проверять отсутствие второго диалога после закрытия первого (напр. коротким
  ожиданием + проверкой отсутствия элемента), а не просто присутствие первого.
- Пересекается с TC-021 (P0, backup→clear→restore) по механике SAF, но не дублирует
  её: TC-021 фокусируется на целостности данных, этот кейс — на UX-контракте единого
  диалога restore+scan, специфичном для downloads.
- ~~SAF-ограничение автоматизации то же, что в TC-021/TC-038~~ — снято, см. выше
  (`bugs/AT-BUG-005.md`).
