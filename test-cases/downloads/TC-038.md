# TC-038 — Auto-scan (silent) при смене папки загрузок перелинковывает файл

## Предусловия
- Приложение запущено; в новой (целевой) SAF-папке загрузок заранее лежит `.html`
  файл с именем, соответствующим паттерну `buildFilename` (оканчивается на
  `_<workId>.html`, где workId совпадает с `ao3_id` засеянной работы).
- В Room засеяна работа с этим `ao3_id`, рейтингом, но `downloadPath=null` (файл
  считается потерянным/не привязанным — напр. после переустановки).
- Открыт экран Settings, текущая папка загрузок отличается от целевой.

## Сценарий (Given-When-Then)

**Given** в целевой папке лежит orphan-файл `..._<workId>.html`, соответствующая
работа в Room существует, но без `downloadPath`

**When** пользователь меняет папку загрузок в Settings на целевую через SAF-пикер

**Then** запускается silent-скан автоматически (без явного нажатия "Scan for
downloads")
**And** появляется диалог результата, так как что-то было перелинковано (найден и
привязан 1 файл)
**And** работа в Library получает связанный файл — карточка показывает open-иконку

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа | засеяна с известным `ao3_id`, `downloadPath=null` |
| Файл | `<любой_префикс>_<ao3_id>.html` в целевой SAF-папке |

## Заметки для автоматизации
- SAF file/folder picker — известная сложность автоматизации (см. заметку в TC-021);
  без готового обходного пути (intent-подмена) кейс не автоматизируем как есть,
  оставить дизайн готовым, эскалировать техническую задачу framework отдельно, если
  к моменту автоматизации обходного пути не появится.
- Диалог должен появиться именно потому, что "что-то реально перелинковано" — отличие
  от TC-037 (диалог всегда, но с "0"), и от гипотетического случая смены папки без
  orphan-файлов (диалог НЕ появляется — не покрыт отдельным кейсом, т.к. не описан
  явно как отдельный риск в §9; при необходимости можно добавить как P2).
