# TC-034 — Открытие file:// применяет стилизацию (viewport + reader.css)

## Предусловия
- Приложение запущено, в Room засеяна работа со скачанным файлом (`downloadPath`
  указывает на существующий `.html` в `ao3_downloads`, помещённый туда заранее через
  adb push фикстурного HTML-файла — без реального сетевого скачивания).
- Открыт экран Library, вкладка FILES содержит эту работу.

## Сценарий (Given-When-Then)

**Given** на вкладке FILES есть работа со связанным локальным `.html` файлом

**When** пользователь тапает по карточке работы

**Then** файл открывается в браузере через `file://`/`content://` URL на активной
вкладке
**And** страница отображается в мобильной ширине (не desktop-width-then-shrunk) —
текст читаемого размера, а не мелкий убористый
**And** применены стили `reader.css` (наблюдаемо визуально/через проверку наличия
инжектированного `<style>`/`<meta name=viewport>` в загруженном DOM)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа | фикстурный `.html` файл (сырой экспорт AO3 без viewport meta), помещён в
`ao3_downloads` через adb, `downloadPath` в Room указывает на него |

## Заметки для автоматизации
- Ключевая деталь риска: PROJECT.md явно фиксирует, что путь реализован НЕ через
  `WebViewClient.shouldInterceptRequest` (ненадёжен на части версий WebView), а через
  прямое чтение файла на IO dispatcher + `loadDataWithBaseURL`. Assert должен ловить
  именно потребительский симптом (читаемый размер текста / наличие meta viewport в
  DOM), а не деталь реализации.
- Подготовка фикстуры: нужен сырой HTML-экспорт AO3 без `<meta name="viewport">` —
  зафиксировать такой файл в `framework/data/recordings/` или отдельной директории
  fixtures для скачанных работ.
- Приложение не делает сетевых вызовов при этом сценарии — годится и live, и replay
  режим одинаково (нет обращения к AO3).
