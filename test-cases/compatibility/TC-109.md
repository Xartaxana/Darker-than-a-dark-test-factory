# TC-109 — Базовый smoke-путь на втором AVD (API 29, ближайший практичный уровень >= minSdk 26)

**AT-BUG-028 (2026-07-22): второй AVD переведён API 26 -> API 29.**
Второй AVD физически существовал на API 26 (AT-BUG-024), но embedded
System WebView этого образа — Chrome 69.0.3497 (EOL ~2018) — структурно
несовместим с readiness-проверкой ЛЮБОГО chromedriver-бинарника,
поддерживающего эту версию Chrome (проверено эмпирически: chromedriver
2.41-2.44 не несут поле `status.ready`, требуемое текущим
appium-chromedriver; первая версия с этим полем, 2.45, уже не
поддерживает Chrome 69 — см. `docs/environment-setup.md`). Второй AVD
переведён на `system-images;android-29;google_apis;x86_64` (тот же
класс образа — rootable, без Google Play), embedded WebView — Chrome
74.0.3729.185, autodownload chromedriver работает штатно. Кейс теперь
проверяет НЕ буквальную нижнюю границу `minSdk` (26), а ближайший
практично-автоматизируемый API level ≥26 — путь, явно разрешённый
критерием готовности AT-BUG-028/AT-BUG-024. **Отмечено для прохода
test-strategist**: смещение риска R-14 (докс/01 §9) с буквального
minSdk=26 на «ближайший практичный уровень» — не переоценено этим
проходом (test-maintainer, B4-скоуп).

## Предусловия
- Второй AVD с нижним API level (нижняя граница — minSdk 26 по манифесту),
  rootable-образ без Google Play (для mitm-CA, см. docs/01 §4) —
  **инфраструктурная предпосылка, на момент дизайна физически ОТСУТСТВУЕТ**
  в `tools/avd` (только `ao3_test_api34.ini`) — см. «Заметки для
  автоматизации» ниже, блокер заведён test_debt-багом.
- Работа W засеяна тем же способом, что на основном AVD (переиспользовать
  seed TC-009/TC-006/TC-007).

## Сценарий (Given-When-Then)

**Given** приложение установлено и запущено на втором AVD (API 29 —
ближайший практичный уровень >= minSdk 26, AT-BUG-028) с чистыми
данными, работа W засеяна

**When** пользователь проходит базовый smoke-путь: запуск → Browse (страница
работы W) → простановка рейтинга → Library (вкладка соответствующего
рейтинга) → открытие/закрытие вкладки (переиспользовать шаги TC-001/006/007,
тот же приём, что TC-098/TC-105 — не изобретать новый маршрут)

**Then** каждый шаг маршрута завершается тем же наблюдаемым результатом, что
на основном AVD (API 34): рейтинг сохраняется и виден без reload, работа
появляется в соответствующей вкладке Library, таб открывается/закрывается без
краша — WebView и Compose UI рендерятся корректно на ближайшем практичном
API level >= нижней границы minSdk
