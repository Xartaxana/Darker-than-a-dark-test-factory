# TC-041 — Применение сохранённого фильтр-профиля

## Предусловия
- Приложение запущено, в Room засеян один `FilterProfile` (имя "My saved search",
  `queryString` с известными параметрами — напр. фильтр по конкретному тегу/фандому),
  через прямую вставку в БД по аналогии с `seed_db.py` (или расширение сидинг-скрипта
  на таблицу `filter_profiles`, если отсутствует).
- Открыта листинговая страница AO3 (search/tag browse, filterable), фильтр-панель
  видна и свёрнута.

## Сценарий (Given-When-Then)

**Given** сохранён фильтр-профиль "My saved search" с известным `queryString`
**And** открыта листинговая страница без применённого фильтра

**When** пользователь раскрывает фильтр-панель и выбирает "My saved search" из списка

**Then** страница перезагружается/обновляется с параметрами из `queryString` этого
профиля (наблюдаемо через URL страницы, содержащий соответствующие `work_search[…]`
параметры, и/или изменившийся список результатов)
**And** фильтр-панель показывает "My saved search" как активно применённый

## Проверяемые данные
| Параметр | Значение |
|---|---|
| FilterProfile | queryString с известным набором параметров (напр. `work_search[query]=...`) |

## Заметки для автоматизации
- `seed_db.py` в текущем виде работает только с таблицей `work_ratings` (см. файл) —
  для этого кейса потребуется отдельная функция сидинга `filter_profiles` (новая
  таблица, схема из `FilterProfileDao.kt`/`AppDatabase.kt`); эскалировать как техдолг
  framework, если функции ещё нет к моменту автоматизации.
- Проверка "применения" — через URL query-параметры активной вкладки, не через
  прямое чтение внутреннего состояния Compose.
- Не дублирует TC-040 (сохранение) — этот кейс стартует с уже готового профиля в БД,
  не создаёт его заново.
