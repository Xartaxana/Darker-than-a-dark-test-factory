# CH-004 — Остаток путей CH-003 + смена настроек посреди сценария

## Mission
Две сросшиеся миссии. (1) Остаток CH-003 (таймбокс не дал): seed 1
«Первый вечер» (rating из ПАНЕЛИ работы на ОТСКРОЛЛЕННОЙ длинной
странице — AT-BUG-015-класс), seed 4 «Уборка» (backup→изменения→restore
ПОСЕРЕДИНЕ сессии), seed 5 «Фильтр-мастер» (профиль + deep-link),
deep-link на лимите 10 вкладок (`browse-deep-link-*`, 0 кейсов).
(2) НОВЫЙ слой (запрос оператора 2026-07-21): пользователь меняет
НАСТРОЙКИ приложения ПОСРЕДИ пути — не как отдельную фичу с чистого
экрана (это покрыто settings-кейсами), а как возмущение накопленного
состояния. Разведка ch-004-settings-recon (журнал 13:52) дала карту:
17 настроек, ПЯТЬ РАЗНЫХ механизмов применения — JS live-push в
открытый WebView (hide_*, filter_display_mode, tap_to_scroll,
infinite_scroll, auto_apply_filter, debug_copy_url), WebView RELOAD
(theme_mode — BrowserScreen.kt:157: FORCE_DARK берёт эффект только
при загрузке), textZoom без reload (font_size_step,
BrowserScreen.kt:203-209), window-attributes вне WebView
(brightness_value), только-следующий-рендер (reset_brightness_on_start,
инъекции onPageFinished). Гипотеза: каждый механизм × «богатое»
состояние (отскролленная длинная страница, 10 вкладок, применённый
фильтр-профиль, pending restore) — непокрытый шов; особенно
theme-RELOAD, который посреди пути молча перезагружает страницу под
ногами у скролла/формы/профиля. Рисковый якорь пуст сознательно
(пути пересекают R-01/R-04/R-08/R-09/R-11 + settings) — риск находки
называется в самой находке, как в CH-003.

## Scope
- **In:** остаточные seed-пути CH-003 (1/4/5 + лимит вкладок), КАЖДЫЙ
  обогащённый минимум ДВУМЯ сменами настроек ПОСРЕДИ пути (правило
  генерации ниже); матрица механизмов применения как чек-лист охвата;
  инвариант-чек после каждого шага (канон CH-003); прерывания между
  шагами (kill/home/rotate).
- **Out:** прямые пути одиночных настроек с чистого старта (покрыты
  settings-кейсами — ассертить только шов); зона темы В ГЛУБИНУ
  (CH-002 — прерывания ВНУТРИ смены темы); швы, уже пройденные CH-003
  под replay (rating↔бейджи↔library-tabs↔note↔visibility,
  switchTab-базовый, тема/шрифт×мульти-таб В НАЧАЛЕ пути — не
  повторять; здесь смена настроек стоит ПОСЛЕ накопления состояния);
  BUG-012 «Clear all ratings» (known_issue — фон, не находка).

## Риски и гипотезы
- **theme_mode посреди пути = скрытый RELOAD активного WebView**
  (BrowserScreen.kt:157): на отскролленной длинной странице —
  теряется scrollY/якорь; на листинге с применённым фильтр-профилем —
  профиль пере-применяется только если auto_apply_filter=true (шов
  theme↔R-09); при 10 открытых вкладках — все или только активная?
  CH-003 проверил тему ДО скролла глубоко/профиля — здесь ПОСЛЕ.
- **font_size_step (textZoom) на отскролленной странице**: zoom меняет
  высоту документа — scrollY численно «на месте», но СЕМАНТИЧЕСКИ
  позиция уехала (читал абзац N — вижу абзац M). Инвариант «в допуске»
  мерить по видимому маркеру, не по пикселям.
- **infinite_scroll OFF посреди догруженного листинга**: уже
  догруженные страницы остаются/исчезают? Обратный ON — дозагрузка
  возобновляется с правильной страницы или с первой?
- **filter_display_mode hide→dim поверх листинга со скрытыми
  работами**: работы появляются (dim) — скролл/счёт прыгает; обратно
  dim→hide на отскролленном — виден ли «провал» позиции.
- **tap_to_scroll ON на УЖЕ открытой работе** (JS live-push
  window.__ao3TapToScroll): конфликт тапа-для-скролла с
  рейтинг-панелью/ссылками; после theme-reload инъекция переживает?
- **seed 4 + настройки**: backup → смена настроек (theme+hide_read) →
  restore: восстанавливает ли restore prefs или только данные? Если
  только данные — листинг обязан показать восстановленные рейтинги
  ПОД текущими (не забэкапленными) настройками видимости — проверить,
  что не смесь. Что именно входит в backup-JSON — установить по коду
  ДО прогона (SettingsScreen.kt:943-995) и записать в протокол.
- **seed 1 (остаток CH-003)**: rating из панели на отскролленной
  живой странице работы + font A+ посреди — двойное возмущение
  скролла (AT-BUG-015-класс; scrollY-механизм AT-BUG-022 «Fixed» —
  прогон даёт данные и его верификации).
- **Лимит 10 вкладок**: deep-link 11-й → AlertDialog
  (BrowserViewModel.kt:263,340; MAX_TABS=10 :1258); смена
  live-push-настройки при 10 открытых — применяется ко ВСЕМ вкладкам
  или лениво при активации (наблюдение CH-003 p2v: НОВЫЙ таб
  наследует prefs; здесь — СТАРЫЕ табы под сменой при полном доме);
  undo закрытия вкладки на лимите (эвикции-undo нет — есть undo
  закрытой: BrowserViewModel, сверить точное поведение по коду).
- **panel_side LEFT→RIGHT посреди пути**: side panel и её
  quick-toggles переезжают (Compose recomposition,
  MainActivity.kt:580-601) — handle-иконка отзывчива только по
  видимой части (ловушка CH-001 x≤~37 — при RIGHT зеркалится?);
  быстрые тумблеры панели после переезда бьют в те же prefs.
- **brightness + reset_brightness_on_start=false → kill/релонч посреди
  пути**: brightness_value читается на старте (MainActivity.kt:120-131)
  — единственная настройка класса «только-следующий-рендер» в карте,
  прерывание — её родной шов.

## Эвристики / туры
**Правило генерации пути (обязательное, наследует CH-003 + новое):**
путь = персона × цель × 4-7 шагов из разных фич; на каждый seed —
минимум одна вариация (перестановка/прерывание/повтор). НОВОЕ: каждый
путь несёт >=2 смены настроек ПОСРЕДИ пути (не шагом 1 и не последним),
из РАЗНЫХ строк матрицы механизмов: {JS live-push | WebView-reload
(theme) | textZoom (font) | window-attr (brightness) |
следующий-рендер (reset_brightness_on_start)}. К финалу сессии каждый
из пяти механизмов должен быть тронут хотя бы одним путём — это
чек-лист охвата, недотронутый механизм — явной строкой в протокол.
После КАЖДОГО шага — инвариант-чек CH-003: активная вкладка/URL те же
(если шаг не навигационный и не theme-reload — там ассертить ВОЗВРАТ
на тот же URL), скролл в допуске ПО ВИДИМОМУ МАРКЕРУ, бейджи
соответствуют Room, prefs записаны.

Seed-пути (стартовый набор, НЕ исчерпывающий):
1. «Первый вечер придирчивого» (live, реальный work-id,
   `AO3_EMU_GPU=host`): browse → открыть работу → скролл вглубь →
   **font A+ ×2** → рейтинг из ПАНЕЛИ → **theme Dark** (reload!) →
   позиция/рейтинг живы? → kill → релонч → всё на месте?
2. «Уборка с ремонтом» (replay+seed): рейтинги на 3+ работах →
   backup → **hide_read ON** → ещё смена рейтинга → **theme Dark** →
   restore → prefs текущие или из бэкапа (установить по коду до
   прогона)? листинг/вкладки показывают восстановленное без смеси?
3. «Фильтр-мастер в сумерках» (replay): Sort&Filter → сохранить
   профиль → применить → листинг отскроллен → **theme Dark** (reload:
   профиль пере-применён? скролл?) → **auto_apply OFF** → deep-link в
   новую вкладку → профиль там применён/не применён — что канон? →
   назад → профиль на исходном листинге жив?
4. «Коллекционер на лимите» (replay, tab_markers): открыть 10 вкладок
   → deep-link 11-й → диалог, не потеря → **filter_display_mode
   hide→dim** + **infinite_scroll OFF** при полном доме → пройтись по
   3-4 СТАРЫМ вкладкам — настройки доехали? → закрыть вкладку → undo
   → снова 10, порядок цел?
5. «Левша на ночёвке» (replay): открыть работу → скролл →
   **panel_side RIGHT** → из ПЕРЕЕХАВШЕЙ панели: **tap_to_scroll ON**
   + **brightness вниз** + reset_brightness_on_start OFF → тап-скролл
   работает, не жрёт клики рейтинга? → kill → релонч → brightness
   восстановлена (reset OFF), panel справа, tap-to-scroll жив?
