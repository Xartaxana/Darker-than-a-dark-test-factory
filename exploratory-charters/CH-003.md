# CH-003 — Пользовательские пути: сквозные сценарии вместо прямых путей одной фичи

## Mission
Весь scripted-корпус (74+ кейса) проверяет ПРЯМОЙ путь одной конкретной
функции: Given сажает приложение в точку, When дёргает одну фичу, Then
ассертит её исход. Реальный пользователь так не ходит: он тянет ПОДРЯД
несколько фич в одной сессии (нашёл работу → читал → оценил → положил в
библиотеку → переключил вкладку → сменил тему → вернулся), и состояние
каждой фичи становится ВХОДОМ следующей. Швы между фичами — скролл после
рейтинга, бейджи после смены темы, вкладки после deep-link, Room↔WebView
после clear/restore — не покрыты ни одним Then. Миссия: ГЕНЕРИРОВАТЬ
правдоподобные пользовательские пути и проходить их целиком, охотясь на
состояние, протекающее между фичами. Рисковый якорь пуст сознательно:
пути пересекают R-01 (данные), R-04 (Room↔бейджи), R-08 (табы), R-09
(фильтр-профили), R-10 (заметки), R-11 (тема) — конкретный риск находки
называется в самой находке.

## Scope
- **In:** сквозные пути из 4+ шагов через >=3 разные фичи; состояние-как-вход
  (проверка инвариантов ПОСЛЕ КАЖДОГО шага, не только в конце); вариации
  порядка шагов; прерывания МЕЖДУ шагами (home/recents/rotate/kill —
  пользователь отвлекается). Стартовый набор путей — §Эвристики, но
  исполнитель ОБЯЗАН сгенерировать и пройти свои вариации (правило генерации
  там же).
- **Out:** прямые пути одиночных фич (покрыты кейсами — не повторять
  регрессию руками); внутренности одного шага, уже имеющие кейс (ассертить
  только шов — что предыдущий шаг не сломал следующий); зона темы В ГЛУБИНУ
  (CH-002 — там прерывания ВНУТРИ смены темы; здесь тема — лишь шаг пути).

## Риски и гипотезы
- Рейтинг из панели на отскролленной странице → скролл теряется/прыгает
  (шов rating↔scroll-restore; ср. AT-BUG-015-класс).
- Смена рейтинга работы, открытой в ДРУГОЙ вкладке library, → перенос
  между вкладками рассинхронизирован с открытым WebView (зона TC-016 —
  watch-item watch-tc016-smoke-fail в журнале: побочный smoke-фейл уже
  наблюдался разово; воспроизведение в пути = данные для watch).
- Deep-link при 10 открытых вкладках (лимит) / после эвикции undo —
  вкладка теряется или reuse ломает историю (browse-deep-link-* ветки
  реестра — дыры с нулём кейсов).
- Применение фильтр-профиля поверх открытого листинга с бейджами →
  бейджи/скролл после перезагрузки листинга неконсистентны (R-09↔R-04).
- Backup → изменения → restore ПОСЕРЕДИНЕ сессии (не с чистого старта,
  как кейсы R-01) → открытые вкладки/листинги показывают дорестор-данные.
- Смена шрифта/темы посреди пути НЕ трогает URL/вкладку/скролл (инвариант
  C4), но фактически перерисовка теряет позицию на длинной работе.

## Эвристики / туры
**Правило генерации пути (обязательное):** путь = персона × цель × 4-7
шагов из РАЗНЫХ фич; на каждый пройденный seed-путь — сгенерировать и
пройти минимум одну вариацию (перестановка двух шагов, вставка прерывания
между шагами, повтор шага дважды). После КАЖДОГО шага — инвариант-чек:
активная вкладка/URL те же (если шаг не навигационный), скролл в допуске,
бейджи листинга соответствуют Room (`webview_avg_luma`/бейдж-прокси и
`ui_snapshot.py`), prefs записаны. Путь журналится в протокол как
нумерованная последовательность «шаг → инвариант-чек» — воспроизводимо.

Seed-пути (стартовый набор, НЕ исчерпывающий):
1. «Первый вечер»: холодный старт → browse → Sort&Filter поиск → открыть
   работу → скролл вглубь → рейтинг из панели → work в library → kill →
   релонч → работа, рейтинг, скролл на месте?
2. «Запойный ридер»: открыть 3-4 работы фоновыми вкладками из листинга →
   переключаться тапом по чипам (switchTab — запись реестра
   browse-tab-switch-active, кейсов 0) → в каждой отскроллить → тема Dark →
   шрифт A+ → рестарт → вкладки/скроллы/тема живы?
3. «Куратор»: library → фильтр по рейтингу → сменить рейтинг работы
   (перенос между вкладками) → comment-only заметка на другой работе →
   скрыть один рейтинг (visibility) → листинг-бейджи и вкладки library
   консистентны после каждого шага?
4. «Уборка»: несколько рейтингов → backup → ещё изменения → restore →
   открытые листинг/вкладки показывают восстановленное (не смесь)? Про
   «Clear all ratings» см. ловушки — известный BUG-012, не дубль.
5. «Фильтр-мастер»: Sort&Filter форма → сохранить профиль → применить с
   панели на другом листинге → deep-link работы в новую вкладку → назад →
   профиль всё ещё применён, вкладок столько, сколько ждём?
