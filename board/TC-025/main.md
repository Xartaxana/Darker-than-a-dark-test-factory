# TC-025 — Вкладки переживают adb kill/relaunch с URL и скроллом

## Предусловия
- Приложение запущено с чистыми данными.
- Открыто 2 вкладки с разными URL, обе проскроллены на разную ненулевую позицию.

## Сценарий (Given-When-Then)

**Given** открыто 2 вкладки с разными URL, каждая проскроллена на свою ненулевую
позицию, позиция скролла подтверждена (`onScrollChanged` сработал — не просто снапшот
"на лету")

**When** приложение принудительно останавливается через `adb shell am force-stop` и
запускается заново

**Then** после перезапуска обе вкладки присутствуют в strip в исходном порядке
**And** URL каждой вкладки совпадает с тем, что было до рестарта
**And** позиция скролла каждой вкладки восстановлена приблизительно к тому же месту
(peek-restore применяется на загрузке)

**Инвариант:** множество вкладок переживает смерть процесса — персистентность через
SharedPreferences переживает process death; после force-stop+relaunch (url, scroll,
порядок) каждой сохранённой на диск вкладки == состоянию до рестарта, независимо от
того, сколько времени/чем занят пользователь между сохранением и рестартом.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Вкладки до рестарта | 2, разные URL, разный ненулевой скролл |
| Способ рестарта | `adb shell am force-stop <package>` + `am start` |

## Заметки для автоматизации
- Использовать `core/adb` (`force_stop`), не kill процесса из Android Studio —
  единообразие с остальным фреймворком.
- Скролл — peek, не consume: значение переживает "shields up" интерстишл AO3, но в
  replay-режиме такого интерстишла нет, поэтому проверка стабильнее в replay.
- Различать этот кейс от TC-023 (undo восстанавливает закрытую вкладку в рамках
  живого процесса) — здесь проверяется персистентность через полный рестарт
  процесса и чтение SharedPreferences, а не in-memory undo-стек.

**Фактическое поведение (test-automator, разведка 2026-07-18).** Как и TC-023 —
идентификация вкладок по нативному заголовку чипа, содержимое через deep-link,
скролл через нативный свайп (см. TC-023.md за деталями обхода прилипания
chromedriver к вкладке-0). Дополнительно для ЭТОГО кейса: `saveTabsToPrefs`
(`BrowserViewModel.kt scheduleSave`) дебаунсит запись на 500мс после каждого
скролл-события — `am force-stop` ДО истечения этого окна теряет несохранённое
состояние безвозвратно; перед рестартом тест опрашивает САМ файл SharedPreferences
устройства (`run-as cat .../ao3_settings.xml`, не UI) на предмет ожидаемого URL —
подтверждение реального сохранения на диск, не просто in-memory снапшот. Использован
`core/adb.force_stop` + `am start -W`, как и предписано выше.
`framework/tests/test_tabs.py::test_tabs_persist_url_and_scroll_after_restart`.
