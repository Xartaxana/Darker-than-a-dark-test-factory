# BUG-013 — Гонка записи theme_mode: kill между toggle и apply() теряет выбор темы

## Окружение
- **Приложение:** ao3-wrapper, версия 1.10 (versionCode 11), сборка от 2026-07-02T02:39:46Z
- **Эмулятор:** emulator-5554 (Android API 33, можно воспроизвести)
- **Режим:** live (интерактивный, Appium с верификацией дерева и файла prefs)
- **Тема при старте:** LIGHT (по умолчанию, `pm clear` обнулил настройки в начале тура)

## Шаги воспроизведения (Given-When-Then)

**Given**
- Приложение установлено и запущено с чистыми предпочтениями (prefs `theme_mode` не персистирована).
- Текущая тема визуально LIGHT; `theme_mode` из файла `ao3_settings.xml` прочитана как отсутствующая (дефолт).

**When**
1. Открыть Settings → Appearance.
2. Выбрать режим **Dark** (тап по text-узлу «Dark» через Appium, прочитано из дерева до toggle).
3. Проверить, что toggle зарегистрирован деревом (Appium-снимок дерева после toggle подтверждает contrast-desc изменился на «Switch to light mode», UI отрендерился тёмным) — файл `ao3_settings.xml` на этом шаге НЕ читался (окно 0мс не даёт времени на флаш).
4. **НЕМЕДЛЕННО, через 0 мс**, выполнить `am force-stop com.ao3.wrapper` (убить процесс приложения).
5. Перезапустить приложение (холодный старт).

**Then (ожидалось)**
- Приложение должно восстановить тему DARK (т.е. `theme_mode=DARK` из `ao3_settings.xml` прочитан после перезапуска, UI отрендерится тёмным).

**Actual (фактически)**
- **Приложение запускается с темой LIGHT.** Проверка `ao3_settings.xml` показывает `theme_mode=LIGHT` (или отсутствует, дефолт).
- Выбранная тема DARK потеряна.

## Частота и градиент задержки

| Задержка между toggle и kill | Трайл 1 | Трайл 2 | Результат | Вывод |
|---|---|---|---|---|
| **0 мс** (немедленный kill) | DARK потерян | DARK потерян | 0/2 (fail) | **Гонка воспроизведена** |
| 100 мс | DARK персистирован | DARK персистирован | 2/2 (pass) | Окно записи закрыто |
| 300 мс | DARK персистирован | DARK персистирован | 2/2 (pass) | Окно записи закрыто |

**Заключение:** Монотонный градиент потеря→персист при 0→100 мс указывает на окно гонки (race condition) **продолжительностью < 100 мс** между моментом toggle и успешной записью в `ao3_settings.xml`. Это окно соответствует асинхронной записи через `SharedPreferences.apply()` без механизма дебаунса/синхронизации.

## Верификация (исключает промах тапа)

- **Pre-state прочитан из файла ДО toggle:** Перед каждым трайлом файл `ao3_settings.xml` явно прочитан через `adb run-as cat`, подтверждена начальная `theme_mode=LIGHT`.
- **Toggle зарегистрирован визуально:** Appium-снимок дерева сразу после toggle показал контраст-текст, переключённый с «Switch to dark mode» на «Switch to light mode» — UI успешно применил тему. **РЕГИСТРИРУЕТСЯ КЛИЕНТОМ.**
- **То же действие (Dark-тап) персистирует на 100/300 мс:** Один и тот же тап (same locator, same UI state) УСПЕШНО сохраняется на бОльших задержках → тап не промах, потеря = недостаток окна ФЛАША, НЕ распознавания ввода.

## Анализ (дефект приложения, не теста/окружения)

### Класс дефекта: асинхронная запись SharedPreferences с гонкой к смерти процесса

Приложение использует `SharedPreferences` для персистирования `theme_mode`. Метод `apply()` в Android — это **асинхронная запись:** контрольный поток возвращается немедленно, фактическая запись в файл произойдёт позже (обычно десятки мс). Если процесс убить в промежутке между `apply()` и окончанием записи, данные теряются.

**Окно гонки обнаружено эмпирически:**
- Окно **< 100 мс** (потеря при 0 мс, персист при ≥100 мс).
- Вероятно, стандартное поведение `apply()` в SharedPreferences без явной синхронизации.

### Почему это баг приложения, а не теста/окружения:

1. **Не промах тапа или UI-состояния:** Тот же Dark-выбор в том же состоянии UI успешно персистирует на 100/300 мс → интерфейс работает; потеря на 0 мс неизбежна при любой реализации `apply()` с таким окном.
2. **Не tooling-класс:** Дерево Appium и прочтение файла через `adb run-as` — стандартные, проверенные методы; pre-state явно верифицирован перед toggle.
3. **Реальный риск для пользователя:** Если процесс приложения будет завершён системным OOM-killer'ом или другой причиной в течение стадии записи предпочтений, выбор темы будет потерян (хотя и маловероятное стечение, это реальный путь дефекта).

## Критерий готовности (рекомендация для разработчика)

**Возможный путь фикса:** Защитить запись `theme_mode` по образцу механизма `BrowserViewModel.scheduleSave()`:

- `BrowserViewModel.scheduleSave()` дебаунсит запись состояния ВКЛАДОК на **500 мс**, буферизуя частые изменения.
- Тест **TC-025** (`app_steps.wait_tabs_persisted`) явно проверяет, что `am force-stop` до истечения дебаунса теряет несохранённые вкладки — **дефект признан и обработан**.
- `theme_mode` использует `apply()` напрямую без дебаунса и **БЕЗ теста на kill-гонку**.

**Предложение:** Рассмотреть введение дебаунса (500 мс, по образцу вкладок) для записи `theme_mode` и добавить регрессионный тест, аналогичный TC-025, проверяющий гарантию персистирования темы при kill-гонке.

**Замечание:** Вероятно, `apply()` без дебаунса был выбран как компромисс (немедленное применение темы UI без задержки). Решение о приоритете (срочный фикс, откладывание в очередь, или WontFix с принятием риска) остаётся за разработчиком и product-владельцем.

## Связанные наблюдения (D-0043: класс дефекта)

**Sibling-класс — персистирование состояния при гонке kill:**

- **BrowserViewModel.scheduleSave()** — дебаунс 500 мс для записи вкладок + явный тест TC-025.
- **theme_mode (этот баг)** — apply() без дебаунса + нет теста kill-гонки.
- **Общая ось:** `ao3_settings.xml` (носитель обоих параметров) + паттерн «асинхронная запись SharedPreferences + гарантия персистирования при смерти процесса».

Правильность фикса должна быть проверена:
1. Регрессионным тестом, аналогичным TC-025 (вкладки).
2. Проверкой консистентности `theme_mode` между UI и файлом после kill в разных окнах задержки.
