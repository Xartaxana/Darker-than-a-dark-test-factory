# TC-084 — Тап по чипу неактивной вкладки активирует её и переключает контент

## Предусловия
- Приложение запущено с чистыми данными (`clean_app`).
- Replay-режим, фикстура `tab_markers.mitm` (те же маркерные страницы с уникальным
  `<title>` на страницу, что использует TC-023/024/025 — детерминизм без живой сети).
- Открыто 3 вкладки с различимыми URL: 0 — стартовая Home, 1 — маркерная страница 1,
  2 — маркерная страница 2 (каждая создаётся deep-link'ом, который добавляет вкладку
  В КОНЕЦ и делает её активной — см. `openOrNavigateDeepLink`,
  `BrowserViewModel.kt:637-644). Активная вкладка на момент Given — 2 (последняя
  созданная).

## Сценарий (Given-When-Then)

**Given** открыто 3 вкладки (0 — Home, 1 — маркер 1, 2 — маркер 2), активна вкладка 2

**When** пользователь тапает по чипу вкладки 0 (неактивной)

**Then** вкладка 0 становится активной: отображаемый контент WebView соответствует
её URL (Home) — наблюдается ЧЕРЕЗ scrollY-АСИММЕТРИЮ нативного скролла
(`assert_tab_became_active_via_scroll`, см. bugs/AT-BUG-022.md, Fixed), НЕ через
reduce-to-one (тот приём, унаследованный из TC-026, для ЭТОЙ цели — вкладки 0 —
структурно не различает «switchTab сработал» от «switchTab — no-op», см. полный
разбор конфаунда в AT-BUG-022)
**And** набор вкладок не изменился этим действием — по-прежнему 3 вкладки, с теми же
URL/заголовками на позициях 1 и 2, что были до тапа (`switchTab` — чистая смена
индекса, не создаёт и не закрывает вкладки, `BrowserViewModel.kt:289-295`)
**And** повторный тап по уже активному чипу (вкладка 0) не меняет состояние —
контент и число вкладок остаются теми же (идемпотентность на уже активном индексе)

**Инвариант:** switchTab(i) для любого i из текущего множества вкладок делает
tabs[i] ЕДИНСТВЕННОЙ активной вкладкой — отображаемый WebView-контент соответствует
ИМЕННО ей, ни одна другая вкладка не становится активной как побочный эффект; само
множество вкладок (число, порядок, URL) переключением не меняется — свойство верно
для любого исходного/целевого индекса, не только для примера выше.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Вкладки до переключения | 3 (Home, маркер 1, маркер 2), активна вкладка 2 |
| Целевая вкладка тапа | позиция 0 (Home) |

## Заметки для автоматизации
- **Блокер AT-BUG-022 снят (Fixed, 2026-07-20) — reduce-to-one заменён на
  scrollY-асимметрию нативного скролла.** Reduce-to-one (established-приём
  TC-026: `switch_to_tab` + сведение числа вкладок к одной + `assert_active_
  tab_url`) для ЦЕЛИ = вкладка 0 структурно не различает «switchTab сработал»
  от «switchTab — no-op» (полный разбор конфаунда — `bugs/AT-BUG-022.md`,
  раздел «Суть долга»). Заменяющий приём — `browser_steps.
  assert_tab_became_active_via_scroll(driver, target_position=0,
  scroll_baseline=...)`: нативный свайп после тапа детерминированно бьёт по
  ФИЗИЧЕСКИ видимому View (не подвержен sticky-прилипанию chromedriver к
  вкладке-0), поэтому его эффект на `scrollY` вкладки 0 (читаемый через
  sticky WEBVIEW-контекст) прямо доказывает, стала ли вкладка 0 физически
  активной. Эмпирически подтверждено на эмуляторе (живой прогон 2026-07-20,
  ровно этот Given: 3 вкладки, активна 2, тап на чип 0): контрольный свайп
  ДО тапа (активна вкладка 2) не изменил scrollY вкладки 0 (0 -> 0); свайп
  ПОСЛЕ тапа изменил его (0 -> 720). См. `bugs/AT-BUG-022.md` за полным
  протоколом и `framework/tests/test_tabs.py::
  test_tap_inactive_tab_chip_activates_it` за автоматизированным кейсом (3
  зелёных прогона подряд).
- Локатор чипа и тап — уже существующие `BrowserScreen.tab_chip_locator`/
  `tap_tab_chip` (`framework/screens/browser_screen.py:278-288`), обёрнутые шагом
  `browser_steps.switch_to_tab(driver, position)` (`browser_steps.py:1025-1026`) —
  использовались TC-023/025/026, инфраструктура тапа/локаторов была готова
  и до фикса — блокером было только наблюдение активности (снято выше).
- Доказательство «набор вкладок содержит те же URL на позициях 1/2» —
  established-приём из TC-023/025/026: позиционная сверка заголовков чипов
  (`assert_tab_title_at_position`) ДО и ПОСЛЕ тапа — этот приём НЕ затронут
  AT-BUG-022 (он не про активность вкладки 0, а про неизменность соседей).
- Идемпотентность повторного тапа по уже активной вкладке — второй `switch_to_tab`
  на тот же индекс, затем повтор той же content-проверки.
- Сознательно ВНЕ скоупа этого кейса (не входит в Then, чтобы не гадать по
  непроверенному вопросу): визуальная индикация активного чипа (фон/рамка
  `TabChip`, `TabStrip.kt` — не имеет accessibility-семантики `selected`, чисто
  цветовая; относится к смежной фиче `browse-tabstrip-indicators`, не к этой,
  согласно записи реестра) и взаимодействие `switchTab` со scroll-restore
  вкладки-назначения (`_activeScrollY=0` при переключении, `BrowserViewModel.kt:292`
  сбрасывает только внутренний счётчик ViewModel, а не обязательно фактический
  визуальный скролл живого WebView — открытый вопрос дизайна, явно поднят
  docs/01-test-strategy.md §9 «tabs: переключение активной вкладки тапом по чипу»
  и НЕ разрешён по коду однозначно без живого прогона; не включаю в Then, чтобы не
  фиксировать непроверенное поведение как приёмочный критерий — см. отчёт).
