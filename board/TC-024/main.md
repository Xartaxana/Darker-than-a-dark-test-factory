# TC-024 — Эвикция самого старого снапшота при 6-м закрытии подряд

## Предусловия
- Приложение запущено с чистыми данными.
- Открыто 7 вкладок с различимыми URL (например разные страницы работ), достаточно
  для 6 последовательных закрытий и минимум одной оставшейся активной вкладки.

## Сценарий (Given-When-Then)

**Given** открыто 7 вкладок с различимыми URL

**When** пользователь последовательно закрывает 6 вкладок по одной (swipe-up по чипу
каждый раз, snackbar каждый раз просто исчезает/сменяется следующим без нажатия Undo)

**Then** после 6-го закрытия snackbar с Undo относится к 6-й (самой последней)
закрытой вкладке
**And** нажатие Undo восстанавливает именно 6-ю закрытую вкладку
**And** самая первая (1-я) из шести закрытых вкладок больше не восстановима — её
снапшот вытеснен из истории (проверяется невозможностью откатиться к ней дальше
одного доступного Undo-снапшота на момент проверки)

**Инвариант:** undo-история — очередь фиксированной ёмкости 5 c FIFO-эвикцией;
самое старое закрытие сверх лимита (Home, эвиктится синхронно в closeTab до первого
snackbar) невосстановимо — число успешных Undo-восстановлений == числу распознанных
по нативному заголовку маркеров из последних ≤5 закрытий.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Вкладки до закрытий | 7, с различимыми URL/заголовками |
| Число последовательных закрытий | 6 |
| Лимит истории закрытий | 5 (`up to 5 most-recent closes are undoable`) |

## Заметки для автоматизации
- Прямой способ проверить "невосстановимость" 1-й закрытой вкладки — ограничен UI:
  snackbar показывает Undo только для последнего закрытия. Assert строится на том,
  что после Undo текущего (6-го) снапшота новый snackbar для следующего отката не
  появляется автоматически (Undo не "цепочка"), либо (если реализовано иначе) на
  количестве вкладок после серии undo-действий не превышающем ожидаемое. Уточнить
  фактическое поведение по живому UI перед реализацией теста — возможна корректировка
  формулировки Then без потери сути риска (эвикция старых снапшотов).
- Общий риск R-08 ("персистентность табов, undo") — этот кейс покрывает граничное
  значение лимита истории, отдельно от TC-023 (happy path одного undo).

**Фактическое поведение (test-automator, разведка + живой прогон 2026-07-18).**
`SnackbarHostState.showSnackbar` (Material3, `MainActivity.kt onCloseTab`) при 6
закрытиях подряд ставит 6 корутин, но количество РЕАЛЬНО интерактивных snackbar'ов,
успеваемых поймать автотестом в разумном окне, эмпирически МЕНЬШЕ 5 (наблюдалось 2-3,
дважды воспроизведено, похоже на ограничение таймингов очереди snackbar'ов под
программной нагрузкой быстрее человеческого темпа, отдельное от риска этого кейса —
не опровергает и не подтверждает точное «5»). Автоматизированный тест проверяет
ИНВАРИАНТ, устойчивый к этой неопределённости: закрывает 6 вкладок, исчерпывающе
перебирает ВСЕ доступные snackbar Undo (до 8 попыток), затем сверяет число успешных
восстановлений с числом РАСПОЗНАННЫХ по нативному заголовку чипа маркерных вкладок
(1..5) — если бы среди восстановленных оказался Home (1-я/самая старая закрытая, чей
снапшот эвиктится синхронно ДО первого показа snackbar'а), числа разойдутся. Это
строго доказывает «Home никогда не восстановим», не заявляя точного количества успешных
попыток. `framework/tests/test_tabs.py::test_undo_history_evicts_oldest_after_six_closes`.

Попутно (не из ревью, обнаружено при собственных прогонах после доработки):
`exhaust_undo_snackbars` — правка кода теста ДЛЯ ЭТОГО кейса всё-таки
потребовалась, 2 отдельных гонки в самой функции, обе в `browser_steps.py`
(steps-слой, не app-under-test):
1. Между `undo_snackbar_visible` и `tap_undo_snackbar` был зазор, в который
   snackbar мог авто-скрыться (`SnackbarDuration.Short`) — непойманный
   `TimeoutException` валил тест целиком (2 из 3 прогонов). Тап обёрнут
   коротким таймаутом+try/except.
2. `restored_count` от `exhaust_undo_snackbars` сравнивался с ОТДЕЛЬНЫМ,
   ОТЛОЖЕННЫМ на конец функции снимком видимых заголовков — снимок мог
   отставать от факта восстановления на десятки секунд (лишние ожидания цикла
   после последнего успеха), что 1 раз дало ложное `restored_count(2) >
   len(found)(1)`, ложно похожее на утечку Home. Функция переработана:
   возвращает `(restored_count, confirmed_titles)`, заголовки читаются СРАЗУ
   после каждого успешного восстановления, не единым снимком в конце.
   Инвариант («Undo никогда не восстанавливает Home») не ослаблен — стал точнее.

4+ зелёных прогонов подряд после обеих правок + полная регрессия
`test_tabs.py`. `review: changes_requested` снято.
