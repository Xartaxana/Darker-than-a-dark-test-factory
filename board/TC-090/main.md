# TC-090 — Добавление личного тега сохраняет его среди выбранных

## Предусловия
- Приложение запущено с чистыми данными, режим replay (`listing_basic.mitm`).
- Работа W = `KUDOSED` (900000002) в дефолтном состоянии — ни рейтинга, ни
  личных тегов (сидинг не требуется).
- Открыта листинговая страница, содержащая блёрб работы W.

## Сценарий (Given-When-Then)

**Given** приложение запущено с чистыми данными, открыта replay-листинговая
страница с работой W (`KUDOSED`), у которой нет ни рейтинга, ни личных
тегов; Rate-кнопкой работы W открыт нативный bottom-sheet

**When** пользователь выбирает рейтинг «Kudosed» (разблокирует раздел
тегов), раскрывает раздел тегов (тоггл «Add tags»), вводит в текстовое поле
свободный тег «spoiler-test» (заведомо отсутствующий среди suggested-тегов
карточки W — не входит ни в `suggestedRelTags`, ни в `suggestedFreeformTags`
блёрба `listing_basic.mitm`) и нажимает «Add»

**Then** тег «spoiler-test» появляется как выбранный чип в разделе тегов
(текста «spoiler-test» не было на экране ДО добавления — тег не входит в
suggested-набор карточки, поэтому его появление однозначно означает
добавление в `tags`, а не показ уже существовавшего suggested-чипа)
**And** после сворачивания раздела (тоггл «Hide tags» → тап снова) лейбл
показывает счётчик «Saved tags (1)» вместо исходного «Add tags»
**And** после закрытия bottom-sheet тапом по затемнённой области, повторного
открытия Rate-кнопкой работы W и раскрытия раздела тегов чип «spoiler-test»
по-прежнему присутствует — доказательство персистентности в Room, не только
локального Compose-состояния overlay

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | `framework/data/works.py::KUDOSED` (900000002), рейтинг/теги не заданы |
| Добавляемый рейтинг | Kudosed (LIKE) |
| Добавляемый тег | «spoiler-test» (вне suggested-набора карточки W) |

## Заметки для автоматизации
- Шаги для ввода тега уже существуют и используются в
  `rating_steps.add_tag_via_listing_overlay` (TC-076/077, живая ветка):
  `RatingOverlay.toggle_tags()` / `.enter_tag_input(text)` /
  `.confirm_tag_input()` (`framework/screens/rating_overlay.py:107-123`) —
  переиспользовать напрямую в replay-контексте; перед этим — выбор рейтинга
  через `RatingOverlay.choose("LIKE")` (уже существует, строка 28-30).
- Дробление реестра НЕ потребовалось (проверено при проектировании, title
  реестра `rating-tags-chip-add` содержит «suggested-чип или свободный
  ввод»): оба пути вызывают одну и ту же функцию `addTag()`
  (`RatingOverlay.kt:109-116`, вызовы из suggested-чипа ~294/299 и из
  текстового поля ~339/346) — Then идентичен («тег появляется среди
  выбранных»), не два разных исхода; этот кейс покрывает путь свободного
  ввода (уже поддержан фреймворком), suggested-чип — тот же код, отдельного
  кейса не требует.
- Новая ассерция: счётчик «Saved tags (N)» — тривиальный `by_text` (не
  реализован сейчас), предлагаемый метод
  `RatingOverlay.tags_count_label_visible(n)` → `is_present(by_text(f"Saved
  tags ({n})"))`, по образцу существующего `add_note_toggle_visible()` —
  рутинная работа, не блокер.
- Присутствие/отсутствие текста чипа читается уже существующим generic
  `is_present(by_text_contains(...))` (используется, например, в
  `comment_text_visible`) — новый локатор для самого чипа не нужен.
- Реплей-инфраструктура и сидинг уже верифицированы (`bugs/AT-BUG-004.md`,
  Verified) — блокеров нет.
