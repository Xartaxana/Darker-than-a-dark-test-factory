# TC-111 — Поворот экрана сохраняет вкладку/URL/скролл без пересоздания WebView

## Предусловия
- Приложение запущено с чистыми данными, устройство в portrait.
- Работа W засеяна (переиспользовать seed TC-009), открыта в браузере,
  пользователь проскроллил страницу вниз на заметную величину.
- Поддержка поворота подтверждена по коду (не по памяти):
  `AndroidManifest.xml` (`<activity .MainActivity>` несёт
  `configChanges="uiMode|orientation|screenSize|screenLayout|
  smallestScreenSize"`, без `screenOrientation`-лока) и `MainActivity.kt`
  ~стр. 563 (явная landscape-логика side panel) — поворот обрабатывается
  приложением конфигурационно, без пересоздания Activity по умолчанию.

## Сценарий (Given-When-Then)

**Given** приложение открыто на странице работы W в portrait, страница
проскроллена вниз, активна конкретная вкладка (URL известен)

**When** устройство поворачивается в landscape (через Appium/`adb`), затем
обратно в portrait

**Then** после поворота В landscape: активная вкладка и URL не изменились,
позиция скролла страницы сохранена (в пределах терпимого допуска — не сброс
к началу страницы), WebView не пересоздан (снимок JS-состояния/тот же
`wv`-инстанс — см. заметки), side panel отображает landscape-компоновку
**And** после поворота ОБРАТНО в portrait: та же вкладка/URL/скролл
сохранены — свойство симметрично, не только в одну сторону

**Инвариант:** поворот экрана (portrait↔landscape, в ОБЕ стороны) сохраняет
активную вкладку, URL и scroll-позицию и не пересоздаёт WebView — свойство
верно для каждого направления поворота, не только для одного наблюдения;
новая формулировка по образцу существующего банка C4 (аналог «undo закрытия
таба восстанавливает URL, scroll и позицию» — тот же класс «состояние вкладки
переживает системное событие», здесь событие — поворот, а не закрытие/undo).

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа | ao3_id из `data/works.py` (та же, что TC-009) |
| Направления поворота | portrait→landscape, landscape→portrait |
| Проверяемые инварианты | активная вкладка, URL, scroll-позиция, отсутствие пересоздания WebView |

## Заметки для автоматизации
- Блокера нет: поворот — стандартная возможность Appium-сессии (`driver.
  orientation = "LANDSCAPE"/"PORTRAIT"`, штатный WebDriver-capability, не
  требует нового framework-примитива) ЛИБО `adb shell settings put system
  user_rotation` при выключенном auto-rotate — выбор конкретного механизма
  (Appium-capability vs adb-settings) — решение test-automator при
  кодировании, задокументировать в тесте.
- Детекция «WebView не пересоздан» — операционализация на усмотрение
  test-automator: например, JS-маркер, инжектированный после первой загрузки
  (`window.__ao3TestMarker = true`), переживший поворот, если он всё ещё
  `true` после поворота — WebView не пересоздавался (проверяется через
  `execute_script`); альтернатива — сравнение `wv`-инстанса на уровне
  Compose, если доступно для black-box наблюдения. Задокументировать выбор в
  тесте при кодировании.
- Допуск на «скролл сохранён» — не пиксель-в-пиксель (лейаут меняется при
  повороте: другая ширина/side panel), а «не сброшен к верху страницы» —
  относительный критерий, аналог щедрого бюджета в TC-096/TC-099.
