# TC-063 — Сортировка Word count (low to high) и сброс скролла

## Предусловия
- Приложение запущено, в Room засеяны 5 работ с одним рейтингом и разным word_count
  (`framework/data/works.py::ALL`: 300/800/1500/4200/12000) плюс `SCROLL_FILLERS`
  (10 работ с word_count 11–20, тот же рейтинг) — гарантированная прокрутка списка,
  тот же набор данных, что уже использует TC-030 для Word count (high to low).
- Открыт экран Library, вкладка проскроллена вниз, активна дефолтная сортировка
  Last read.

## Сценарий (Given-When-Then)

**Given** вкладка Library содержит работы с разным word_count, список проскроллен
вниз (не на самом верху)

**When** пользователь открывает dropdown сортировки в top bar и выбирает "Word
count (low to high)" (иконка trend-up)

**Then** пять эталонных работ ALL расположены между собой по возрастанию word_count
(300, 800, 1500, 4200, 12000) относительно друг друга — независимо от того, где
среди них оказались филлеры (у филлеров word_count 11–20, меньше самого маленького
из ALL, поэтому они окажутся ПЕРЕД всеми пятью — это не мешает проверке
относительного порядка именно эталонных пяти)
**And** список визуально возвращён к началу (скролл сброшен на верх), а не остаётся
на прежней позиции — то же свойство, что TC-030 доказывает для направления high to
low, здесь подтверждается для low to high (design fact §9: смена ЛЮБОЙ сортировки
сбрасывает скролл)
**And** иконка-триггер в top bar сменилась на иконку выбранной сортировки

**Инвариант:** сортировка не меняет множество работ вкладки, только порядок (банк);
смена сортировки (в данном случае — на Word count low to high) сбрасывает скролл
вкладки к началу независимо от направления сортировки — обобщение TC-030
(проверявшего только high to low) на второе направление того же поля.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работы | `ALL` из `framework/data/works.py` + `SCROLL_FILLERS`, один рейтинг |
| word_count values (ALL) | 300, 800, 1500, 4200, 12000 |
| Сортировка | Word count (low to high) |

## Заметки для автоматизации
- Полностью симметричен TC-030 по механике (тот же набор данных, тот же приём со
  скроллом) — переиспользовать существующие шаги/фикстуры без изменений, только
  выбрать другой пункт dropdown. Блокера нет.
- Не дублирует TC-031 (работы БЕЗ word_count всегда в конце независимо от
  направления — граничный случай null, не проверяется здесь заново) и не дублирует
  TC-027 (фильтр min/max, не сортировка) — этот кейс только про порядок при
  ASCENDING направлении на полном наборе с известными значениями.
- Сброс скролла — проверять через позицию первого видимого элемента списка (см.
  заметку TC-030: `LazyListState` не доступен из UI-теста напрямую, сравнивать
  видимый текст/id первой карточки до и после).
