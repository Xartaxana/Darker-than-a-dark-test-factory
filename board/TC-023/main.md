# TC-023 — Undo закрытия вкладки восстанавливает позицию и историю

## Предусловия
- Приложение запущено с чистыми данными.
- Открыто 3 вкладки с разными URL (напр. вкладка 0 — главная AO3, вкладка 1 —
  страница работы, вкладка 2 — листинг), вкладка 1 проскроллена вниз.
- Активна вкладка 1 (средняя по позиции).

## Сценарий (Given-When-Then)

**Given** открыто 3 вкладки, вкладка 1 (средняя) имеет ненулевую позицию скролла

**When** пользователь делает swipe-up по чипу вкладки 1, закрывая её
**And** в появившемся snackbar "Tab closed" нажимает действие Undo

**Then** вкладка восстанавливается на исходной позиции (индекс 1, между исходными
соседями)
**And** URL вкладки соответствует тому, что было до закрытия
**And** позиция скролла восстановлена (страница открыта на том же месте, а не с нуля)

**Инвариант:** restoreClosedTab — точная инверсия closeTab: восстановленная вкладка
тождественна закрытой по (позиция, url, scroll), соседние вкладки не смещены (close
затем undo = тождество наблюдаемого состояния).

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Вкладки до закрытия | 3, разные URL |
| Закрываемая вкладка | индекс 1 (средний), с ненулевым скроллом |

## Заметки для автоматизации
- Скролл-позицию проверять через сравнение видимого контента/JS `scrollY`, а не
  точным пикселем — WebView может слегка отличаться при рендере.
- Swipe-up на чипе вкладки — жест по конкретному элементу strip, не по всему экрану;
  локатор чипа взять из `TabChip` (MainActivity.kt), затем свериться с живым деревом.
- Этот кейс — happy path одного undo сразу после закрытия. Работа с 5 последними
  закрытиями (эвикция старых снапшотов) — отдельный сценарий, см. TC-024.
- `peekPendingScrollRestore` (не `consume`) — скролл не должен "съедаться" короткой
  промежуточной загрузкой; для детерминизма использовать replay-режим, не live.

**Фактическое поведение (test-automator, разведка 2026-07-18).** Appium/chromedriver
экспонирует ОДИН WEBVIEW-контекст на пакет и при >1 живой WebView-вкладке
прилипает к вкладке-0 (первый когда-либо созданный WebView) — `driver.get()`/
`execute_script`/`current_url` НЕ следуют за переключением активной вкладки в UI
(эмпирически подтверждено, см. докстринг `framework/screens/browser_screen.py` у
`tab_chip_locator`). Реализация обходит это: (1) содержимое вкладок задаётся через
реальный Android deep-link intent (`app_steps.open_deep_link`, минует chromedriver
целиком), (2) идентификация вкладок — по НАТИВНО видимому заголовку чипа
(`TabInfo.title`), не по URL из WebView-контекста, (3) скролл активной вкладки —
НАТИВНЫЙ свайп-жест по экрану (реальный touch, не JS `scrollTo`), (4) финальная
проверка `scrollY` восстановленной вкладки делается ПОСЛЕ сведения числа вкладок к
одной (закрытие соседей форсирует chromedriver переподключиться к единственной
оставшейся — тоже подтверждено эмпирически). Отдельная синтетическая replay-фикстура
`framework/data/recordings/tab_markers.mitm` (высокие статичные страницы с уникальным
`<title>` на страницу) вместо `/tos`-подхода из заметки выше — не требует живой сети,
детерминирована. `framework/tests/test_tabs.py::test_swipe_close_undo_restores_position`.
