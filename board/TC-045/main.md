# TC-045 — Личные теги не скрывают работу

## Предусловия
- Приложение запущено, в Room засеяна ПАРА работ с ОДИНАКОВЫМИ личными тегами
  (`tags=["spoiler", "reread-candidate"]`), различающихся только рейтингом:
  W_disliked (rating=Disliked) и W_visible (rating=Loved/SAVE, не в hidden-set) —
  требует расширения сидинг-скрипта под непустой `tags` (JSON-конвертер Room),
  см. заметки.
- В Settings включена фильтрация с Disliked в hidden-set (дефолт).
- Открыта листинговая страница, содержащая обе работы.

## Сценарий (Given-When-Then)

**Given** W_disliked имеет rating=Disliked (скрывается фильтрацией по правилу
видимости) и W_visible — rating=Loved (не скрывается), у ОБЕИХ один и тот же
набор личных тегов

**When** пользователь просматривает листинговую страницу с включённой фильтрацией

**Then** W_disliked скрыта на листинге — как и любая другая Disliked-работа, а
W_visible — с ТЕМИ ЖЕ тегами — видна: одинаковые теги дают ПРОТИВОПОЛОЖНЫЙ исход
видимости, что доказывает, что решение excluded/visible переключает исключительно
`rating`, а личные теги на него не влияют ни в какую сторону
**And** переход на экран Library показывает: вкладка DISLIKED — W_disliked с
сохранёнными личными тегами, вкладка FAVORITE — W_visible с ТЕМИ ЖЕ личными тегами
(видны через индикатор/детали карточки в обоих случаях)

**Инвариант:** видимость блёрба на листинге — функция ТОЛЬКО членства `rating`
работы в `hiddenRatings` (`applyAllFilters`, `ao3_bridge.js`, читает только
`ratings[workId]`/`hidden`); `tags` в это решение НЕ входит вообще — ни как
условие скрытия, ни как условие показа. Одинаковые `tags` у W_disliked и
W_visible при противоположном исходе видимости — прямое наблюдаемое
доказательство этого свойства (не единичный пример «Disliked+теги всё ещё
скрыта», который доказывал бы только «теги не спасают от скрытия», но не
исключал бы «теги могли бы прятать иначе-видимую работу»).

## Проверяемые данные
| Параметр | Значение |
|---|---|
| W_disliked | rating=DISLIKE, tags=["spoiler", "reread-candidate"] |
| W_visible | rating=SAVE (Loved), tags=["spoiler", "reread-candidate"] (те же) |
| Фильтрация | Enable filtering = ON, Disliked в hidden-set |

## Заметки для автоматизации
- Цель кейса — доказать отсутствие эффекта: результат видимости идентичен тому, что
  было бы без тегов вообще (ссылка на существующее покрытие Disliked-скрытия в
  TC-013/TC-014/TC-015, area=visibility, P0). Не дублирует их — там нет тегов в
  фикстуре, здесь специально добавлены, чтобы исключить регресс "теги случайно стали
  влиять на видимость".
- `seed_db.py::_insert_rows` пишет `tags=None` жёстко — расширение сидинга должно
  поддержать JSON-сериализованный список (см. `Converters.kt` в app-under-test для
  формата хранения — не изменять сам код приложения, только формат INSERT в тестовом
  сидинге должен ему соответствовать).
- Подсветка совпадающих AO3-тегов индикатором (вторая часть требования про личные
  теги) — не проверяется в этом кейсе (отдельный аспект UI, не про видимость); при
  необходимости можно завести отдельный P2/P3-кейс на индикатор подсветки, сейчас не
  запрошено явно в §9 для видимости.

## Заблокировано (test-automator, 2026-07-08)

**Снято (Lead, 2026-07-10):** AT-BUG-004 Verified 2026-07-09 — инфраструктура готова, блокировка неактуальна; ниже — исторический текст. Кейс в очереди Review→Approved (решение человека).

Возвращён в `Review` — невозможно закодировать имеющимися средствами, тот же класс
блокера, что и TC-013/014/015 (см. `bugs/AT-BUG-004.md`), дословно совпадающий по
механизму: скрытие работы W на листинге — это тот же `applyAllFilters`
(`ao3_bridge.js`), что и в TC-013, и требует ту же replay-фикстуру листинга с
блёрбом `li#work_{ao3Id}` засеянной синтетической работы W. Личные теги
(`window.__ao3Tags`) не меняют этот путь — `applyAllFilters` вообще не читает
`tags` при принятии решения excluded/visible (см. код: только `ratings[workId]` и
`hidden`), поэтому сам предмет кейса (отсутствие влияния тегов) наблюдаем
исключительно через ту же листинговую страницу, что уже заблокирована для
TC-013/014/015.

Вторая часть Then (вкладка DISLIKED показывает работу W с сохранёнными тегами) не
требует листинга и технически автоматизируема через Library — но проверять только
её означало бы не проверить именно риск R-10 («личные теги не влияют на
видимость»), а лишь то, что теги сохраняются в БД. Разделять сценарий и
маскировать половину слабым ассертом не делаем.

Для этого кейса также потребовалось бы расширение `seed_db.py` под непустой JSON
`tags` (см. заметку кейса, `Converters.kt`) — не сделано, так как без разблокировки
листинга сама фикстура не даёт наблюдаемого результата.

**Что нужно, чтобы разблокировать:** см. критерий готовности в `bugs/AT-BUG-004.md`
(та же replay-фикстура листинга, что нужна TC-013/014/015 — работа W должна быть
Disliked и присутствовать в её блёрбе) + расширение сидинга под `tags` (отдельная,
небольшая доработка `seed_db.py`, не блокирующая сама по себе).

Область помечена нуждающейся в доводке replay-инфраструктуры — не APP_BUG,
не TEST_BUG; инфраструктурный блокер, тот же класс, что и в visibility/downloads.

Логика автотеста

верна и стабильна: зелёный прогон пройден. Замечание — к ДИЗАЙНУ кейса.

1. **Комбинаторная область «видимость» без строки `Инвариант:` + единичный пример
   вместо свойства (пункт 3 / C4).** Кейс формулирует ИНВАРИАНТ-НЕЗАВИСИМОСТЬ:
   «личные теги никак не меняют факт скрытия — ни ослабляя, ни усиливая» (feature
   `bridge-hide-filter`, R-10, тот же `applyAllFilters`, что у сиблингов
   `visibility/TC-013…015`, которые все несут строку `**Инвариант:**`). Но:
   - формальной строки `**Инвариант:**` в теле кейса нет (сиблинги её имеют —
     TC-015:43);
   - тест
     (`framework/tests/test_rating_listing.py:203`,
     `browser_steps.assert_blurb_hidden`) проверяет ОДИН пример — что Disliked-работа
     *с тегами* скрыта. Это доказывает «теги не ОТМЕНЯЮТ скрытие Disliked», но НЕ
     доказывает независимость видимости от тегов как свойство (нет сравнения
     видимость-с-тегами vs видимость-без-тегов; ветка «тег случайно СКРЫЛ бы
     иначе-видимую работу» не наблюдается вовсе). Это ровно «единичный пример из
     шагов» вместо проверки СВОЙСТВА, о котором предупреждает пункт 3.
   **Что сделать:** (а) добавить строку `**Инвариант:** …` (видимость блёрба =
   функция ТОЛЬКО членства `rating` в hidden-set, `tags` в решение excluded/visible
   не входят — см. код `applyAllFilters`, уже процитированный в секции «Причина»);
   (б) усилить наблюдаемость свойства независимости — например, парой
   Disliked-с-тегами скрыта / не-Disliked-с-теми-же-тегами видна на одной странице
   (доказывает, что переключает видимость именно rating, а tags инвариантны к
   исходу в обе стороны), а не только одиночный «Disliked+теги скрыт». Designer-
   чек-лист дополнить пунктом комбинаторной области (как `library/TC-060`).
