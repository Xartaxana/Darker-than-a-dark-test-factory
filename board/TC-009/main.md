# TC-009 — Пять рейтингов из листинга

## Предусловия
- Приложение запущено с чистыми данными.
- Открыта листинговая страница AO3 (search/tag browse/user works — replay-фикстура
  с записанным листингом из `framework/data/recordings/`), содержащая как минимум
  одну работу (`li.work.blurb`) с инжектированной Rate-кнопкой (28px circle,
  top-right блёрба, после `p.datetime`).

## Сценарий (Given-When-Then)

**Given** приложение запущено с чистыми данными и открыта листинговая страница с
работой W (Rate-кнопка видна и не активирована)

**When** пользователь нажимает Rate-кнопку у работы W, в открывшемся нативном
bottom-sheet (`RatingOverlay`) выбирает рейтинг «R» (Loved / Liked / Read / Pending /
Disliked)

**Then** bottom-sheet закрывается (или подтверждает сохранение согласно UI)
**And** бейдж/стиль Rate-кнопки на карточке блёрба работы W обновляется без
перезагрузки листинга
**And** работа W появляется в соответствующей вкладке Library после перехода на экран
Library

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | синтетическая работа из replay-фикстуры листинга (см. framework/data/recordings) |
| Рейтинг R | один из: Loved(SAVE) / Liked(LIKE) / Read(READ) / Pending(PENDING) / Disliked(DISLIKE) |

## Заметки для автоматизации
- Требует replay-режим (детерминированный листинг) или live с известной тестовой
  работой — предпочтителен replay согласно docs/01 §4.
- Локатор Rate-кнопки: искать в живом DOM/дереве, не по предположению из кода —
  `ao3_bridge.js` инжектирует круг 28px после `p.datetime`; сверить фактическое
  content-description/accessibility label на живом дереве перед фиксацией локатора
  в screens/web-слое (test-automator).
- Параметризовать по 5 значениям Rating, аналогично TC-007, но точка входа — листинг,
  не панель work page (разные компоненты: bottom-sheet `RatingOverlay` vs встроенная
  `RatingMenu`).
- LIKE/SAVE может авто-кликать kudos на открытой work-page вкладке того же work, если
  такая открыта — не проверяется здесь.

## Заблокировано (test-automator, 2026-07-03)

**Снято (Lead, 2026-07-10):** AT-BUG-004 Verified 2026-07-09 — инфраструктура готова, блокировка неактуальна; ниже — исторический текст. Кейс в очереди Review→Approved (решение человека).

Возвращён в `Review` — невозможно закодировать имеющимися средствами, тот же класс
блокера, что и TC-013/014/015.

**Причина:** кейс по дизайну требует «листинговую страницу … содержащую как минимум
одну работу (`li.work.blurb`) с инжектированной Rate-кнопкой» — детерминированный
листинг с известным набором работ. На реальном archiveofourown.org с синтетическими
`ao3_id` (`framework/data/works.py`) такого листинга не существует (это не
work-page, где достаточно URL с `/works/(\d+)` для срабатывания `isWorkPage` —
здесь нужен реальный HTML листинга с блёрбами конкретных работ, инжектированный
`ao3_bridge.js`). Живой AO3-поиск/tag-browse вернёт реальные чужие работы без
контроля над `ao3_id`/`title`, что не позволяет детерминированно проверить
"работа W появляется в нужной вкладке Library" (нет способа адресовать конкретную
синтетическую работу W в реальной выдаче).

Тот же replay-блокер, что и в TC-013/TC-014/TC-015:
- `framework/data/recordings/` — пустая директория.
- Спайк B (`docs/environment-setup.md`) не завершён (захват трафика эмулятора на
  этом Windows-хосте не работает, подозревается Windows Defender Firewall).
- `framework/core/mitm.py` — только интерфейс, не подключён к `conftest`/`contexts`.

Разница с TC-007 (тоже live/панель, но там Automated): TC-007 требует лишь URL
`/works/{id}` (панель `RatingMenu` реагирует на URL-паттерн, а title/author можно
предзаполнить сидингом Room до захода на страницу — см. заметку "Автоматизация" в
TC-007.md). TC-009 требует реальный HTML листинга с блёрбом конкретной работы — это
принципиально невозможно получить без replay или без похода на реальный чужой
контент AO3 (не наш риск/скоуп тестировать).

**Что нужно, чтобы разблокировать:** см. TC-013.md — тот же спайк B + запись/
конструирование листинговой HTML-фикстуры с блёрбом синтетической работы в
`framework/data/recordings/`.

Область помечена нуждающейся в доводке replay-инфраструктуры — не APP_BUG,
не TEST_BUG; инфраструктурный блокер.
