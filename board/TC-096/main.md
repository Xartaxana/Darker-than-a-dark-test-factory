# TC-096 — Холодный старт: бюджет TotalTime/WaitTime

## Предусловия
- Приложение установлено, данные очищены (`pm clear`), процесс приложения
  принудительно остановлен (`am force-stop`) непосредственно перед замером —
  гарантирует ХОЛОДНЫЙ старт (нет уже тёплого процесса/Activity в памяти).
- Режим не важен (live/replay) — замер идёт до появления WebView-контента,
  сетевой трафик AO3 не участвует в измеряемом интервале.

## Сценарий (Given-When-Then)

**Given** приложение установлено, данные очищены, процесс приложения не запущен
(после `pm clear` + `am force-stop`)

**When** приложение запускается командой `adb shell am start -W
-n <package>/<activity>`

**Then** `TotalTime` (и `WaitTime`) из вывода `am start -W` укладывается в
щедрый ОТНОСИТЕЛЬНЫЙ бюджет, откалиброванный по нескольким baseline-прогонам
того же эмулятора (не жёсткий абсолютный SLA в мс — разброс таймингов AVD
ожидаем и не является дефектом сам по себе)
**And** приложение фактически запущено (нативная оболочка отрисована —
переиспользовать существующую проверку `app_steps.wait_ui_ready`)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Команда замера | `adb shell am start -W -n com.example.ao3_wrapper/com.example.ao3_wrapper.MainActivity` |
| Метрики из вывода | `TotalTime`, `WaitTime` (парсятся из stdout `am start -W`) |
| Бюджет | относительный — baseline (медиана N прогонов на этом же эмуляторе, напр. 5) + щедрый запас (напр. ×2-3); конкретный множитель и N — решение test-automator при кодировании, задокументировать в тесте, не выдумывать здесь точное число |

## Заметки для автоматизации
- `am start -W` уже используется во фреймворке в этом же качестве:
  `framework/core/adb.py::_run`/`shell` (докстринг явно упоминает `am start -W`
  как «блокирующий вызов, ждёт полной прорисовки окна»), `framework/data/seed_db.py
  ::ensure_db_initialized` и `framework/steps/app_steps.py::restart_app_via_adb`
  уже зовут именно эту команду — переиспользовать `adb.shell(...,
  timeout=settings.ADB_LAUNCH_TIMEOUT)`, не изобретать новый вызов. Блокера нет:
  инфраструктура запуска готова, не хватает только парсинга `TotalTime`/`WaitTime`
  из stdout и порогового assert'а — рутинная автоматизация.
- Порядок: `adb.force_stop()` + `adb.clear_app_data()` ДО замера — иначе
  измеряется тёплый/повторный старт, а не холодный (искажает метрику).
- Относительный бюджет — эмулятор существенно медленнее/вариативнее реального
  устройства; конкретный порог не в спеке кейса намеренно (task: «щедрый
  относительный бюджет, не абсолютный SLA»).
- **`features` — точная nf-привязка `nf-perf-cold-start-budget`.** Прежняя
  приблизительная функциональная привязка (`browse-initial-load`) снята
  решением 2026-07-21 (анти-двойной-зачёт, `docs/01-test-strategy.md` §9):
  нефункциональная область получает собственную запись реестра вместо
  приближения к ближайшей функциональной. Поведенческое покрытие
  `browse-initial-load` — за его собственными кейсами, не за этим.

**2026-07-22 — test-automator, решение по порогу (N/множитель):** N=5
baseline-прогонов (медиана), бюджет = медиана x2.5 (отдельно для TotalTime и
WaitTime), с полом 500ms на случай вырожденно малой медианы —
`framework/tests/test_performance.py::BUDGET_MULTIPLIER`/`BASELINE_RUNS`/
`MIN_COLD_START_BUDGET_MS`. Обоснование: живая калибровка на emulator-5554
дала тёплый повторный `am start -W` TotalTime=0ms (см.
`adb.parse_am_start_metrics` докстринг — `force_stop`+`clear_app_data` ДО
каждого замера обязательны, иначе метрика тривиальна), холодный —
TotalTime≈3728ms; x2.5 над медианой холодных прогонов — щедрый запас,
ловящий кратную деградацию, не единичное дрожание AVD. 3 зелёных прогона
подряд (`test_cold_start_within_relative_budget`, ~34-35s каждый).

## Ревью автотеста

**test-reviewer, 2026-07-22 — вердикт: PASS (`Approved → Automated`,
`automation_status: active`).** Весь чек-лист F1 пройден.

- **Архитектура (C1):** `python scripts/arch_check.py` — 0 ошибок / 0
  предупреждений; тест не в ALLOWLIST. Локаторов/driver в tests/ нет — замер
  инкапсулирован в `perf_steps.measure_cold_start`/`cold_start_baseline`
  (композиция adb-примитивов), `sleep` отсутствует (`am start -W` блокирует до
  прорисовки штатно).
- **Traceability:** `@allure.id("TC-096")` == id; `@pytest.mark.p1` ↔
  `priority: P1`; `automated_by` указывает на существующую
  `test_cold_start_within_relative_budget`.
- **Соответствие кейсу:** assert проверяет суть — `TotalTime` И `WaitTime`
  укладываются в относительный бюджет (медиана 5 baseline ×2.5, пол 500мс),
  плюс And «приложение фактически запущено» (`app_steps.wait_ui_ready`).
  `force_stop`+`clear_app_data` перед КАЖДЫМ замером гарантируют холодный старт
  (не тёплый no-op с TotalTime=0). Не ослаблено до «элемент существует».
- **Фикстуры/flake:** только `driver`; данные не требуются; observed —
  независимый прогон, не входит в baseline (нет самосравнения).
- **Независимое воспроизведение (п.6):** собственный зелёный прогон модуля
  `test_performance.py` — PYTEST_EXIT=0, 4 passed in 144s (emulator-5554).
- **Красная проба (п.7), 2026-07-22:** порча на уровне проверяемого условия —
  временно `BUDGET_MULTIPLIER=0.001` + `MIN_COLD_START_BUDGET_MS=1` (сжат
  бюджет). `Invoke-Pytest -k cold_start` → FAILED: «холодный старт
  TotalTime=4505ms превышает относительный бюджет 4ms». Осмысленный assert, не
  таймаут-мусор. Порча откачена в том же ходе (файл untracked → значения
  возвращены Edit'ом; дифф чист). Тест умеет падать.
