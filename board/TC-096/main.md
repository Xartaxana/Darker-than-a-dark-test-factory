# TC-096 — Холодный старт: бюджет TotalTime/WaitTime

## Предусловия
- Приложение установлено, данные очищены (`pm clear`), процесс приложения
  принудительно остановлен (`am force-stop`) непосредственно перед замером —
  гарантирует ХОЛОДНЫЙ старт (нет уже тёплого процесса/Activity в памяти).
- Режим не важен (live/replay) — замер идёт до появления WebView-контента,
  сетевой трафик AO3 не участвует в измеряемом интервале.

## Сценарий (Given-When-Then)

**Given** приложение установлено, данные очищены, процесс приложения не запущен
(после `pm clear` + `am force-stop`)

**When** приложение запускается командой `adb shell am start -W
-n <package>/<activity>`

**Then** `TotalTime` (и `WaitTime`) из вывода `am start -W` укладывается в
щедрый ОТНОСИТЕЛЬНЫЙ бюджет, откалиброванный по нескольким baseline-прогонам
того же эмулятора (не жёсткий абсолютный SLA в мс — разброс таймингов AVD
ожидаем и не является дефектом сам по себе)
**And** приложение фактически запущено (нативная оболочка отрисована —
переиспользовать существующую проверку `app_steps.wait_ui_ready`)

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Команда замера | `adb shell am start -W -n com.example.ao3_wrapper/com.example.ao3_wrapper.MainActivity` |
| Метрики из вывода | `TotalTime`, `WaitTime` (парсятся из stdout `am start -W`) |
| Бюджет | относительный — baseline (медиана N прогонов на этом же эмуляторе, напр. 5) + щедрый запас (напр. ×2-3); конкретный множитель и N — решение test-automator при кодировании, задокументировать в тесте, не выдумывать здесь точное число |

## Заметки для автоматизации
- `am start -W` уже используется во фреймворке в этом же качестве:
  `framework/core/adb.py::_run`/`shell` (докстринг явно упоминает `am start -W`
  как «блокирующий вызов, ждёт полной прорисовки окна»), `framework/data/seed_db.py
  ::ensure_db_initialized` и `framework/steps/app_steps.py::restart_app_via_adb`
  уже зовут именно эту команду — переиспользовать `adb.shell(...,
  timeout=settings.ADB_LAUNCH_TIMEOUT)`, не изобретать новый вызов. Блокера нет:
  инфраструктура запуска готова, не хватает только парсинга `TotalTime`/`WaitTime`
  из stdout и порогового assert'а — рутинная автоматизация.
- Порядок: `adb.force_stop()` + `adb.clear_app_data()` ДО замера — иначе
  измеряется тёплый/повторный старт, а не холодный (искажает метрику).
- Относительный бюджет — эмулятор существенно медленнее/вариативнее реального
  устройства; конкретный порог не в спеке кейса намеренно (task: «щедрый
  относительный бюджет, не абсолютный SLA»).
- **`features` — приблизительная связь (attempt 2, task perf-stability-smoke-e2).**
  `browse-initial-load` — ближайшая существующая запись реестра (тот же путь
  запуска, что она описывает: приложение → вкладка Browse открывает HOME_URL),
  но Then ЭТОГО кейса — пороговый тайминг `am start -W`, а не наблюдаемое
  поведение фичи (какой URL открылся). Связь для схемного требования непустого
  `features` (trace-matrix v2 §1b), не заявление функционального покрытия
  `browse-initial-load` этим кейсом. `test-cases/README.md`/`docs/01-test-strategy.md`
  §9 (E2) документируют эту приблизительность явно (правка critic-входа приёмки,
  2026-07-21) — не заявляют `features: []`. Открытый вопрос владельцу схемы
  (не блокирует эту сдачу): должна ли схема допускать пустой `features` для
  нефункциональных cross-cutting областей вместо приблизительной привязки.
