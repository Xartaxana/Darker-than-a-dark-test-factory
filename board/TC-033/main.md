# TC-033 — Ручное скачивание из Library

## Предусловия
- Приложение запущено, в Room засеяна работа с рейтингом Loved/Liked/Pending
  (`downloadPath=null`), без включённого Auto-download.
- Открыт экран Library, вкладка с этой работой — карточка показывает download-иконку
  (Favorite/Liked/Pending tabs only, отсутствует файл).

## Сценарий (Given-When-Then)

**Given** карточка работы на вкладке Library показывает download-иконку (файл ещё не
скачан)

**When** пользователь нажимает download-иконку на карточке

**Then** запускается ручное скачивание (`downloadWork`: сначала regex-поиск
download-ссылки на странице работы, затем сохранение файла)
**And** по завершении download-иконка на карточке заменяется на open-иконку
**And** работа появляется на вкладке FILES (Downloads) экрана Library

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа | засеяна с рейтингом (не Disliked — download-иконка не на этой вкладке) |
| Auto-download | выключен (проверяем именно ручной путь) |

## Заметки для автоматизации
- Только вкладки Favorite/Liked/Pending показывают download-иконку — если работа
  засеяна с рейтингом Read или Disliked, кейс не воспроизводим как описан; выбрать
  рейтинг из допустимого набора при сидинге.
- Требует replay-записи страницы работы с download-ссылкой, как и TC-032.
- Отличается от TC-032 явным ручным триггером (тап по иконке) вместо побочного
  эффекта простановки рейтинга.

## Заблокировано (test-automator, 2026-07-08)

Возвращён в `Review` — тот же replay-блокер, что и TC-032 (см. заметку в
TC-032.md, там же — что нужно, чтобы разблокировать).

**Причина:** ручной путь тоже идёт через `DownloadRepository.downloadWork`
(`fetchDownloadUrl` через `OkHttpClient`, не WebView) — нужна та же replay-запись
страницы работы + `.html`-скачивания, которой пока нет в
`framework/data/recordings/` (там только `ao3_home_smoke.mitm`), и то же
отсутствующее подключение `framework/core/mitm.py` к pytest-фикстурам.
TC-034/TC-035/TC-036 (открытие/удаление уже скачанного файла) автоматизированы
отдельно — они обходят `DownloadRepository` через фикстуру с уже "скачанным"
файлом (без обращения к сети), поэтому не упираются в этот блокер.

Область помечена нуждающейся в доводке replay-инфраструктуры — не APP_BUG,
не TEST_BUG; инфраструктурный блокер, тот же класс, что и в rating/visibility.
