# TC-055 — Яркость жестом: реальная яркость окна + чёрный overlay ниже минимума

## Предусловия
- Приложение запущено с чистыми данными, вкладка Browse активна, открыта страница
  AO3, `brightnessValue` в дефолтном положении (0 = яркость окна не переопределена).
- Side panel свёрнут (жест по области контента).
- Режим прогона: replay либо live-smoke; яркость экрана управляется приложением.

## Сценарий (Given-When-Then)

**Given** на Browse открыта страница AO3, яркость на дефолтном уровне, чёрного
затемняющего overlay нет

**When** пользователь выполняет двухпальцевый параллельный вертикальный драг вниз
поверх страницы, продолжая тянуть за пределы минимальной яркости окна

**Then** сначала яркость экрана снижается (реальная яркость окна уменьшается,
появляется временный HUD-индикатор уровня яркости); при продолжении ниже системного
минимума поверх всего экрана появляется полупрозрачный чёрный overlay, дополнительно
затемняющий изображение; обратный драг вверх убирает overlay и повышает яркость

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Жест понижения | двухпальцевый параллельный драг вниз |
| Эффект в диапазоне v≥0 | снижение реальной яркости окна |
| Эффект при v<0 | чёрный overlay поверх экрана (растёт до заметного затемнения) |
| Побочный признак | HUD-индикатор уровня яркости на время жеста |

## Заметки для автоматизации
- **Техническая оговорка по наблюдаемости (важно).** Реальную яркость окна
  (`Window.attributes.screenBrightness`) UiAutomator2/Appium напрямую как UI-элемент
  не видит — это свойство окна, не view. Практичные прокси-сигналы наблюдаемого
  поведения: (1) появление/исчезновение HUD-индикатора уровня яркости во время
  жеста; (2) для нижнего диапазона (v<0) — чёрный overlay является реальной view в
  иерархии (полноэкранный Box с полупрозрачным чёрным фоном) и наблюдаем через
  скриншот-сравнение (общее затемнение кадра) — именно overlay даёт проверяемый
  визуальный результат, тогда как изменение аппаратной яркости окна может
  потребовать чтения системного состояния вне обычного UI-дерева. Кейс описан от
  пользовательского поведения; автоматизатору решать, какой прокси использовать, и
  зафиксировать это как известную сложность автоматизации.
- Жест яркости распознаётся при синхронном вертикальном смещении двух пальцев
  (totalDy) ≥ 20dp, полный диапазон ≈ 480dp; порог яркости (20dp) срабатывает
  раньше порога шрифта (30dp по span), поэтому для попадания в яркость нужно вести
  оба пальца вниз/вверх параллельно, не разводя/сводя их (иначе — шрифт, см. TC-053).
- Overlay включается только для значений ниже 0 (ниже системного минимума окна);
  максимум затемнения (~0.72 alpha) при нижней границе -0.3. Верхний диапазон
  (0..1) меняет только реальную яркость окна без overlay.
- Та же хрупкая зона R-11 (CLAUDE.md чек-лист п.5). Не зависит от других кейсов:
  состояние яркости приводится жестом внутри самого кейса.
- **Реализовано:** выбранный прокси — средняя яркость (luma) полноэкранного
  скриншота (`driver.get_screenshot_as_png()` + Pillow `.convert("L").histogram()`,
  framework/requirements.txt: добавлен `Pillow>=10`). Драг вниз/вверх — сырые W3C
  Actions с двумя touch-pointer'ами, двигающимися идентично на каждом шаге (гарантия
  одного знака dy0/dy1 → синхронный параллельный драг, не путается с pinch/spread
  шрифта, см. TC-053). Одним прогоном проверяется совокупный эффект «снижение
  реальной яркости + overlay» (кадр темнеет минимум на 30%) и обратный эффект (кадр
  светлеет минимум на 30% от тёмного состояния) — раздельно фазы v≥0/v<0 текущим
  прокси не различить (единый визуальный сигнал), см. ограничение выше.
