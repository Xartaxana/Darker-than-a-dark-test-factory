# TC-074 — Note-кнопка присутствует ⟺ есть comment (live)

## Предусловия
- Приложение запущено с чистыми данными, режим **live**.
- Открыта реальная листинговая страница AO3 с ≥1 работой.

## Сценарий (Given-When-Then)

**Given** приложение запущено live, открыта реальная листинговая страница AO3
с работой W (не оценена, без комментария)

**When** пользователь нажимает Rate-кнопку работы W, в `RatingOverlay`
выбирает рейтинг и вводит непустой комментарий, сохраняет ("Save note")

**Then** та же `[data-ao3-btn-wrap]` работы W (без перезагрузки листинга)
получает `[data-ao3-note-btn]`, вставленную ПЕРЕД Rate-кнопкой, с `title`,
равным введённому тексту
**And** для ЛЮБОЙ другой работы того же листинга, у которой нет комментария,
`[data-ao3-note-btn]` отсутствует

**Инвариант:** присутствие Note-кнопки — биусловное свойство: кнопка
существует у work-блёрба ровно тогда, когда для этой работы `comment` не
пустой (`applyRatings`: `if (comment) {...} else { existingNote?.remove() }`)
— не пример на одной работе, а правило для любой комбинации оценённых/
неоценённых/прокомментированных работ на странице.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | первая видимая работа реального листинга |
| Комментарий | произвольный непустой текст |

## Заметки для автоматизации
- Отличие от TC-044 (функциональность: Note-кнопка открывает overlay с
  развёрнутым комментарием) — здесь предмет проверки СЕЛЕКТОР/условие
  инъекции самой кнопки, не поведение клика по ней.
- Не требует известного `ao3_id` — работает на любой видимой работе, не
  упирается в блокер класса TC-009.
- Маркер: `@pytest.mark.p0 @pytest.mark.live`.
- Сиблинг-кейс TC-075 — тот же контракт в replay.
