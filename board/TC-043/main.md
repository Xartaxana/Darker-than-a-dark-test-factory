# TC-043 — Comment-only работа: видимость и отсутствие в рейтинговых вкладках

## Предусловия
- Приложение запущено, в Room засеяна работа W с `rating=null` и непустым `comment`
  (comment-only запись — требует создания через UI, т.к. `seed_db.py` сейчас не
  поддерживает прямой сидинг непустого `comment`/`rating=null` комбинированно, см.
  заметки) — предусловие достигается через реальный flow: открыть страницу листинга,
  тапнуть Note-кнопку/оставить комментарий без выбора рейтинга.
- В Settings включена фильтрация (Enable filtering = ON), Disliked в hidden-set.
- Открыта листинговая страница, содержащая работу W.

## Сценарий (Given-When-Then)

**Given** работа W имеет только комментарий, без рейтинга (`rating=null`,
`comment≠null`)
**And** фильтрация по рейтингу включена

**When** пользователь просматривает листинговую страницу с работой W и переходит на
экран Library

**Then** работа W видна на листинговой странице (не скрыта фильтрацией, несмотря на
включённый filtering — правило: работы с `rating=null` никогда не скрываются)
**And** работа W не отображается ни на одной рейтинговой вкладке Library (FAVORITE/
KUDOSED/READ/PENDING/DISLIKED) — только там, где рейтинг не требуется (если
приложение вообще не показывает comment-only записи ни на одной вкладке Library —
это ожидаемое поведение, не Files, т.к. downloadPath не задан)

**Инвариант:** видимость блёрба на листинге определяется исключительно членством
`rating` работы в `hiddenRatings` (`applyAllFilters`, `ao3_bridge.js`); работа с
`rating=null` НЕ является членом никакого рейтинга и, следовательно, никогда не
может быть членом `hiddenRatings` — она видна на листинге ∀ состав hidden-set (в
том числе при пустом hidden-set), не только при конкретной конфигурации из
Given (Disliked в hidden-set). Симметрично на стороне Library: рейтинговые
вкладки (FAVORITE/KUDOSED/READ/PENDING/DISLIKED) фильтруют по конкретному
значению `rating`, а `rating=null` не совпадает ни с одним из пяти значений —
работа W не может появиться ни на одной из них независимо от содержимого
`comment`. Оба свойства — следствие одного факта: `rating=null` вне области
определения и фильтрации по рейтингу, и группировки по рейтинговым вкладкам.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работа W | comment-only: rating=null, comment="Test note" |
| Фильтрация | Enable filtering = ON, hidden-set включает как минимум Disliked |

## Заметки для автоматизации
- `seed_db.py::_insert_rows` жёстко пишет `comment=None` для всех строк — для
  comment-only фикстуры либо расширить сидинг (принять `comment` и `rating=None`
  явно в INSERT), либо создавать запись через реальный UI-flow (note-кнопка на
  листинге, см. TC-044) перед проверкой видимости. Расширение сидинга предпочтительнее
  для скорости и независимости от flow создания заметки.
- Отдельно от TC-044 (сам UI note-overlay flow) — этот кейс проверяет только
  наблюдаемые правила видимости для уже существующей comment-only записи.

**Причина:** первая часть Then («работа W видна на листинговой странице, несмотря на
включённую фильтрацию») наблюдаема только через реальный DOM листинга —
`applyAllFilters` (`ao3_bridge.js`) скрывает/показывает конкретные
`li[id^="work_"].work.blurb` по `ao3_id`. Нужна replay-фикстура листинга,
содержащая блёрб именно засеянной синтетической работы W (`ao3_id` из
`framework/data/works.py`) — на реальном archiveofourown.org такой работы не
существует, а `framework/data/recordings/` пока содержит только
`ao3_home_smoke.mitm`. `framework/core/mitm.py` не подключён ни к одной
pytest-фикстуре.

Вторая часть Then (W не отображается ни на одной рейтинговой вкладке Library) уже
покрыта не требующим листинга способом — см. `test_library.py::test_comment_only_not_in_any_rating_tab`
(TC-017, Automated), использующий фикстуру `comment_only_work`/`seed_with_comment`.
Проверять в TC-043 отдельно ТОЛЬКО эту половину сценария означало бы маскировать
блокер слабым ассертом (пропуская наблюдаемую часть про фильтрацию на листинге,
которая и есть суть риска R-10) — не делаем этого.

**Что нужно, чтобы разблокировать:** см. критерий готовности в `bugs/AT-BUG-004.md`
(replay-фикстура листинга с блёрбом синтетической работы + подключение
`framework/core/mitm.py` к `conftest.py`); сидинг comment-only записи уже готов
(`seed_with_comment`, добавлен ранее для TC-014/TC-017).

Область помечена нуждающейся в доводке replay-инфраструктуры — не APP_BUG,
не TEST_BUG; инфраструктурный блокер, тот же класс, что и в visibility/downloads.
