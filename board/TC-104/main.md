# TC-104 — Приватность бэкапа: манифест-скоуп + права SAF-файла

## Предусловия
- APK тестируемой сборки установлен (для статической части — манифест
  доступен для инспекции, Appium-сессия не требуется).
- Для поведенческой части: приложение запущено, в Room засеяны рейтинги (тот
  же сидинг, что TC-021), SAF file picker доступен на эмуляторе.

## Сценарий (Given-When-Then)

**Given** APK установлен, в Room засеяны работы с рейтингами

**When** (1) манифест инспектируется статически (тот же приём, что
TC-100/101): атрибут `android:allowBackup` и наличие
`android:fullBackupContent` (ссылка на XML-правила) ИЛИ
`android:dataExtractionRules` (Android 12+ эквивалент)
**And** (2) пользователь в Settings выполняет «Back up data» (существующий
механизм `settings-backup-export`, тот же путь SAF-экспорта, что TC-021),
выбирает файл через SAF-пикер, дожидается результата

**Then** (1) `allowBackup=true` СОПРОВОЖДАЕТСЯ явным ограничивающим
механизмом — присутствует `fullBackupContent` ИЛИ `dataExtractionRules`, а не
голый `allowBackup=true` без какой-либо декларации области бэкапа (факт
манифеста, зафиксирован как есть)
**And** (2) созданный SAF-экспортный JSON-файл на диске (`adb shell ls -l
<путь>` / `stat`) НЕ имеет мир-читаемых прав доступа сверх обычного файла в
выбранной пользователем директории — приложение не расширяет права файла
вручную сверх того, что даёт сам SAF/файловая система по умолчанию для
выбранного места

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Пакет | `com.example.ao3_wrapper` |
| `android:allowBackup` | ожидается `true` (по коду манифеста, стр.9 — уже сверено при заведении записи реестра) |
| Ограничивающий механизм | `fullBackupContent` ИЛИ `dataExtractionRules` — присутствие проверяется, не оба обязательны |
| SAF-экспортный файл | путь, выбранный пикером в тесте (напр. `/sdcard/Download/<файл>.json`, как в TC-021) |
| Ожидаемые права | не мир-читаемые сверх дефолта выбранного места (никакого явного `chmod`/расширения приложением) |

## Заметки для автоматизации
- Блокера нет: статическая часть переиспользует приём TC-100/101
  (`adb.shell`/`aapt`); поведенческая часть переиспользует существующий
  SAF-экспорт (`framework/steps/backup_steps.py`, `saf_steps.py`,
  `backup_file_workspace`-фикстура из `test_backup_restore.py`, TC-021) —
  сам SAF-блокер автоматизации уже снят (`bugs/AT-BUG-005.md`). Проверка
  прав файла — новый, но тривиальный `adb shell ls -l`/`stat` вызов поверх
  уже известного пути файла.
- **Решение об объединении static+behavioral в ОДИН кейс** (в отличие от
  TC-100/101/102/103, где каждая запись реестра — отдельный кейс): сама
  запись реестра `nf-sec-backup-privacy` уже текстуально сочетает оба аспекта
  («allowBackup=true + fullBackupContent/dataExtractionRules — статический
  манифест-факт + поведенческая проверка», docs/feature-registry.yaml) как
  ОДНО распознаваемое свойство «приватность бэкапа» — манифест-декларация
  области бэкапа и фактическое поведение SAF-экспорта логически неразделимы
  (одно без другого не даёт ответа на вопрос «приватен ли бэкап»), поэтому
  один сценарий с двумя частями Then, не два отдельных кейса.
- **НЕ дублирует TC-021** (round-trip backup→clear→restore, комбинаторный
  инвариант сохранности данных) — этот кейс о ДРУГОМ свойстве: приватности/
  правах доступа экспортированного артефакта и манифест-декларации области
  бэкапа, не о сохранности полей при restore. Область НЕ комбинаторная в
  смысле банка C4 (не проверяет свойство «для всех входов» — единичная
  проверка факта манифеста + единичная проверка прав файла).
- **Независимость static/behavioral при кодировании (misc-batch-0722):**
  несмотря на объединение в один кейс (см. решение выше), при кодировании
  тест обязан оформить static-проверку (п.1, манифест) и behavioral-проверку
  (п.2, права SAF-файла) НЕЗАВИСИМЫМИ друг от друга — оба результата
  собираются и диагностируются по отдельности; падение манифест-инспекции
  не должно маскировать поведенческую часть и наоборот. Допустимы два теста
  с одним TC-id через allure-связку (`@allure.id("TC-104")` на обоих) либо
  soft-assert паттерн с совокупным падением (оба assert выполняются, отчёт
  называет оба факта, даже если один уже упал).
- **Инструмент статической инспекции манифеста (misc-batch-0722, замечание
  critic прохода (5), `docs/09-history.md:771-773`):** вести через `aapt dump
  xmltree <apk>` (`aapt` из build-tools SDK,
  `tools/android-sdk/build-tools/36.0.0`), НЕ `dumpsys package` — атрибуты
  `exported`/`cleartextTraffic`/`fullBackupContent` в `dumpsys` неполны/
  косвенны для надёжной автоматизации.
- Testability gap: полная модель того, ЧТО именно включает
  `fullBackupContent`/`dataExtractionRules` (какие пути реально включены/
  исключены при полном auto-backup через adb backup) — вне минимального
  смоука; здесь проверяется только ФАКТ присутствия ограничивающего
  механизма, не его содержимое построчно.
