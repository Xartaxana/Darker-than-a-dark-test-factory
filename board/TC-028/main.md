# TC-028 — Чекбокс downloaded-only фильтрует по наличию файла

## Предусловия
- Приложение запущено, в Room засеяны 3 работы с одним рейтингом (напр. LOVED), из
  которых у 1 заполнен `downloadPath` (файл присутствует), у 2 — `downloadPath=null`.
- Открыт экран Library, соответствующая вкладка (FAVORITE), фильтр не применён —
  видно все 3 работы.

## Сценарий (Given-When-Then)

**Given** на вкладке Library видно 3 работы, из которых только 1 имеет связанный
локальный файл (downloadPath не пуст)

**When** пользователь включает чекбокс "downloaded-only" в фильтр-панели

**Then** список показывает только 1 работу — ту, у которой есть скачанный файл
**And** остальные 2 работы (без файла) скрыты
**And** выключение чекбокса возвращает список ко всем 3 работам

**Инвариант:** список после `downloaded-only=true` — точное подмножество
исходных 3 работ, равное {work | downloadPath не пуст}; переключение
чекбокса обратно в false — round-trip, возвращающий множество к исходному
без потерь и дублей (фильтр обратим, не разрушает состояние списка).

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Работы | 3 из `framework/data/works.py::ALL`, один рейтинг (LOVED) |
| downloadPath | у 1 работы заполнен (напр. фиктивный путь на устройстве), у 2 — null |

## Заметки для автоматизации
- `seed_db.py::_insert_rows` сейчас жёстко пишет `downloadPath=None` для всех строк
  (см. заметку в TC-021) — для этого кейса потребуется расширение сидинг-скрипта,
  принимающее опциональный `download_path` на работу, либо создание файла руками через
  adb и апдейт колонки отдельным SQL после базового сидинга. Без этого кейс не
  автоматизируем — эскалировать как техдолг framework, если расширение не появится к
  моменту автоматизации.
- Наличие самого файла по пути не обязательно для проверки видимости в Library (это
  чисто клиентский фильтр по полю БД) — но для реалистичности сценария стоит
  использовать существующий на устройстве файл (иначе смешивается с disk-level
  edge case, не относящимся к этому кейсу).
