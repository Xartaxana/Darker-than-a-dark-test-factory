# TC-039 — Restore + auto-scan показывают ровно один диалог

## Предусловия
- Приложение запущено, в Room пусто (после Clear all ratings / чистая установка).
- Подготовлен backup JSON-файл (`{"version":2,"works":[…],"filterProfiles":[…]}`) с
  одной работой, чей `ao3_id` совпадает с именем orphan-файла, заранее положенного в
  папку загрузок (аналогично фикстуре TC-038), но backup не содержит `downloadPath`
  (т.к. на момент backup файла на этом устройстве не было).
- SAF file picker доступен для выбора файла backup.

## Сценарий (Given-When-Then)

**Given** Library пуста, backup-файл содержит 1 работу, папка загрузок содержит
соответствующий orphan-файл

**When** пользователь выполняет "Restore from backup" и выбирает подготовленный файл

**Then** появляется ровно один диалог результата (`AlertDialog`), объединяющий и
результат restore (количество восстановленных работ), и результат скана (количество
перелинкованных файлов) — не два последовательных всплывающих диалога
**And** после закрытия диалога восстановленная работа в Library имеет заполненный
`downloadPath`, указывающий на orphan-файл

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Backup | 1 работа, без downloadPath |
| Orphan-файл | `..._<ao3_id>.html`, тот же ao3_id, что в backup |

## Заметки для автоматизации
- Ключевой наблюдаемый факт — именно ОДИН диалог, а не два подряд; assert должен
  явно проверять отсутствие второго диалога после закрытия первого (напр. коротким
  ожиданием + проверкой отсутствия элемента), а не просто присутствие первого.
- Пересекается с TC-021 (P0, backup→clear→restore) по механике SAF, но не дублирует
  её: TC-021 фокусируется на целостности данных, этот кейс — на UX-контракте единого
  диалога restore+scan, специфичном для downloads.
- SAF-ограничение автоматизации то же, что в TC-021/TC-038.
