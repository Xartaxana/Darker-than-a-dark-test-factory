# TC-099 — Memory sanity длинной WebView-сессии (до 10 табов)

## Предусловия
- Приложение запущено с чистыми данными, ровно 1 вкладка (Home).
- Живой или replay-режим — не критично для этого кейса (сама память не зависит
  от источника контента); replay предпочтителен для устранения сетевой
  вариативности длительности сессии.

## Сценарий (Given-When-Then)

**Given** приложение запущено, открыта 1 вкладка (Home); снят baseline-замер
`adb shell dumpsys meminfo <package>` (PSS Total)

**When** пользователь открывает вкладки до `MAX_TABS`=10 (тот же механизм
создания вкладки, что TC-022), в каждой выполняет переход на новую
листинг/work-страницу (создаёт реальную WebView-нагрузку в каждой вкладке); снят
второй замер `dumpsys meminfo` (пик, после 10 табов)
**And** пользователь закрывает 9 из 10 вкладок обратно до 1 (swipe-up, тот же
механизм, что TC-023), даёт WebView-процессу время осесть (settle-пауза через
`wait_for`, не фиксированный `sleep`); снят третий замер `dumpsys meminfo`
(после освобождения)

**Then** третий замер (после закрытия 9 вкладок) МЕНЬШЕ второго (пикового) —
память откатывается вниз после освобождения ресурсов, а не остаётся
монотонно прижатой к пику или продолжает расти
**Инвариант:** рост памяти в длинной WebView-сессии обратим — за закрытием
вкладок следует наблюдаемое снижение PSS относительно локального пика; критерий
— ТРЕНД (пик → откат), не абсолютный порог PSS в МБ (единичное большое число
само по себе не дефект, если тренд откатывается). **Caveat для автоматизации**
(critic-вход приёмки, 2026-07-21): голый строгий `<` без допуска рискует
флаком — ART/WebView освобождают память асинхронно, третий замер может
оказаться ≈ пику даже при исправном поведении. test-automator может
потребоваться маржин (например «после отката ближе к baseline, чем к пику»),
а не голое строгое неравенство — решение операционализации за test-automator.

## Проверяемые данные
| Параметр | Значение |
|---|---|
| Команда замера | `adb shell dumpsys meminfo com.example.ao3_wrapper` (поле `TOTAL PSS`) |
| Точки замера | baseline (1 таб) → пик (10 табов, каждый нагружен) → после отката (закрыты 9 из 10) |
| Критерий | `после_отката < пик` (относительный тренд, не абсолютное число) |

## Заметки для автоматизации
- Блокера нет: `adb.shell(...)` уже существующий примитив фреймворка
  (`framework/core/adb.py`), `dumpsys meminfo` не требует новой инфраструктуры —
  парсинг `TOTAL PSS` из вывода `dumpsys meminfo` — рутинная строка кода.
  Открытие/закрытие до 10 вкладок — тот же механизм, что уже спроектирован и
  автоматизирован в TC-022 (лимит)/TC-023 (swipe-up close) — переиспользовать
  шаги `browser_steps`/`TabStrip`, не изобретать новый способ.
- Settle-пауза перед третьим замером — опрашивать факт остановки закрывающихся
  WebView (`wait_for`/`wait_until`), не `sleep` (конвенция фреймворка, README
  «Никаких sleep — только core/waits»); GC на JVM/ART асинхронен, поэтому
  разумный опрос с таймаутом, не одноразовое чтение сразу после закрытия
  последней вкладки.
- Открытие 10 вкладок — самый дорогой шаг сценария (см. заметку TC-022) —
  можно переиспользовать ту же оптимизацию (не ждать полной загрузки AO3 в
  каждой вкладке, важно только реальное создание WebView-инстанса с каким-то
  контентом, дающим память).
- Критерий НАМЕРЕННО не абсолютный порог PSS (docs/01 §9: «критерий — отсутствие
  безоткатного роста (тренд), не абсолютный порог») — конкретное число МБ
  какой угодно большое не является дефектом само по себе, если после отката
  вкладок оно снижается; растущий БЕЗ отката тренд — дефект.
- **`features` — приблизительная связь (attempt 2, task perf-stability-smoke-e2).**
  `browse-tab-limit-max` и `browse-tab-close-undo` — механизмы открытия/закрытия
  вкладок (тот же путь, что TC-022/TC-023), реально используемые для создания
  и снятия WebView-нагрузки в этом сценарии. Связь — по механике шагов, не по
  Then: этот кейс проверяет ТОЛЬКО тренд PSS (пик → откат), не лимит MAX_TABS
  и не поведение Undo из snackbar. Та же оговорка про приблизительность
  `features` (см. TC-096, `test-cases/README.md`, `docs/01-test-strategy.md`
  §9 E2), что в TC-096/097/098.
