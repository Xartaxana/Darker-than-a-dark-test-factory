# 03 — Система агентов

Модель заимствована из двух референсов:

- **dmtools / Dark Factory** — оркестратор со статусной машиной: правила вида
  «артефакт в статусе X и нет лока → запустить агента Y», локи против двойной
  обработки, эскалация человеку через `Blocked`.
- **BMAD TEA** — каждый агент = персона + пошаговый воркфлоу + шаблон выходного
  артефакта + чек-лист валидации «что значит готово».

Вместо Jira статусная машина живёт на **файлах с YAML frontmatter** в этом
репозитории (тест-кейсы, баги, прогоны). Это даёт офлайн-работу, версионирование в
git и простые правила для оркестратора. Опционально баги зеркалируются в GitLab
Issues репозитория приложения.

## 1. Реестр агентов

Агенты реализуются как субагенты Claude Code (`.claude/agents/*.md`), запускаемые
верхним уровнем `/qa-loop` или вручную через скиллы (`/run-suite`, `/triage` и т.п.).

**Runtime-модель оркестрации (пересмотрена 2026-07-07, находка репетиции
2026-07-04):** вложенная оркестрация субагентов не работает — Task, вызванный из
субагента, уходит в асинхрон; оркестратор-субагент не может дождаться воркера,
за проход диспатчится один агент, лок воркера сиротеет. Поэтому **диспетчеризацию
выполняет верхний уровень** — сам скилл `/qa-loop` (основная сессия): читает
`rules.yaml`, запускает pre_steps-скрипты, ставит локи и вызывает воркеров глубины 1
синхронным Task. Ни у одного воркера нет инструмента Task — глубина 1 обеспечена
конструктивно. qa-orchestrator остался read-only планировщиком для `--dry-run`.

| # | Агент | Когда действует (триггер) | Что делает | Выход |
|---|---|---|---|---|
| 0 | **qa-orchestrator** (планировщик, read-only) | Только `/qa-loop --dry-run` | Сканирует статусы всех артефактов, применяет правила (§3) и возвращает machine-readable план срабатываний — БЕЗ диспетчеризации и правок | План (yaml) в ответе агента |
| 1 | **test-strategist** | Старт проекта; изменился PROJECT.md/версия приложения; появилась новая функциональность | Обновляет `docs/01-test-strategy.md`: риски, приоритеты, скоуп. Помечает области без покрытия | Стратегия + список областей `needs-design` |
| 2 | **test-designer** | Есть область `needs-design`; стратегия обновлена | Пишет тест-кейсы в формате **Given-When-Then** по шаблону, с приоритетом и ссылкой на риск. Проходит чек-лист качества кейса | `test-cases/<area>/TC-xxx.md` в статусе `Draft` |
| 3 | **test-automator** | Тест-кейс в статусе `Approved` и не автоматизирован; либо возврат с ревью (`review: changes_requested`) | Кодирует тест по слоям фреймворка (docs/02), локально запускает его, добивается стабильного прохождения, линкует `@allure.id(TC-xxx)`; статус кейса НЕ меняет (гейт F1) | Код теста; заполненный `automated_by` |
| 3½ | **test-reviewer** (F1) | Кейс `Approved`, `automated_by` заполнен, нет `review` | «Вторые глаза»: чек-лист (arch_check, traceability, соответствие GWT по смыслу, фикстуры, flake-риск) + независимый прогон; тестовый код НЕ правит | TC → `Automated` + `automation_status: active`, либо `review: changes_requested` с замечаниями в кейсе |
| 4 | **test-runner** | Новая сборка APK; расписание (ночная регрессия, дневной canary); запрос fix-verifier | Поднимает эмулятор/mitm, ставит APK, гоняет нужный suite, собирает Allure + results.json | `runs/RUN-<ts>/report.md` в статусе `NeedsTriage` (если есть падения) или `Closed` |
| 5 | **failure-analyst** | Прогон в статусе `NeedsTriage` | По каждому падению: читает лог, скриншот, page source, при необходимости перезапускает тест изолированно. Выносит вердикт: `APP_BUG` / `TEST_BUG` / `SITE_CHANGED` / `ENV_ISSUE` / `FLAKY` | Вердикты в отчёте прогона; прогон → `Triaged` |
| 6 | **bug-reporter** | В прогоне есть вердикт `APP_BUG` без привязанного бага | Как в dmtools bug_creation — решает: **создать** новый баг / **прилинковать** к существующему открытому / признать дубликатом. Заполняет шаблон: GWT-репро, severity, версия, логи, скриншоты. **Никогда не правит код приложения** | `bugs/BUG-xxx.md` в статусе `Open`; линк в TC и прогоне |
| 7 | **fix-verifier** | Баг переведён человеком-разработчиком в `Fixed` **и** есть сборка новее версии, где баг найден | Прогоняет связанные тест-кейсы на новой сборке. Прошли → баг `Verified` (закрыт); нет → `Reopened` с новыми логами | Обновлённый статус бага + verification-прогон |
| 8 | **test-maintainer** | Вердикт `TEST_BUG` или `SITE_CHANGED`; либо поведение изменилось намеренно (баг закрыт как `Intended`) | Чинит/обновляет тестовый код, локаторы, записи mitmproxy, сами тест-кейсы; при `SITE_CHANGED` первым делом перезаписывает recordings и сверяет селекторы с актуальным `ao3_bridge.js` | Обновлённые тесты/кейсы; падения переведены в `Resolved` |

Разделение test-automator (новые тесты) и test-maintainer (починка существующих)
сознательное — как в dmtools разделены `*_test_automation` и `*_test_automation_rework`:
разные входы, разные определения «готово».

## 2. Статусные машины артефактов

Каждый артефакт — markdown с YAML frontmatter: `id`, `status`, `priority`, `links`,
`lock` (агент+timestamp — аналог `addLabel`/`skipIfLabel` из dmtools).

**Машиночитаемый источник правды по переходам — `schemas/transitions.yaml`**
(C3, с 2026-07-07): для каждого перехода — акторы (`by`), доступность с борды
(`via_board`), обязательные эффекты (`status_since`/`updated` всегда;
`reopen_count+1`, `escalation` и т.п.). Из матрицы выводится whitelist
board_inbound; целостность и паритет закреплены `scripts/tests/test_transitions.py`.
Схемы ниже — иллюстрация; при расхождении верить матрице. Смена `status` вне
матрицы запрещена — нет подходящего перехода → это эскалация, а не «переход по
смыслу».

**Тест-кейс** (`test-cases/`):
```
Draft → Review → Approved → Automated
                    ↑ (человек утверждает; допустим авто-approve для P2/P3)
```

**Баг** (`bugs/`):
```
Open → Fixed → Verified (закрыт)
  ↑       ↓ (fix-verifier: репро всё ещё есть)
  └── Reopened
Open → Rejected (человек: не баг/не воспроизводится) → фабрика перепроверяет репро:
        воспроизвёлся → доказательства + спор (awaiting: dev); нет → принято
Open → Intended (человек: так задумано) → test-maintainer согласует тесты/кейсы/доки
Open → Blocked (нужно решение человека)
```
Переход `Open → Fixed` делает **только человек** (разработчик приложения) — агенты
не чинят код приложения и не закрывают баги без верификации.

Полная матрица реакций на действия разработчика (Rejected, комментарии, сборка без
фикса, регрессия от фикса, изменение поведения без тикета и т.д.) — **docs/06 §2**.
Для её поддержки frontmatter бага расширен полями `status_since`, `last_seen_in`,
`reopen_count`, `dispute_count`, `regression_of`, `awaiting` и разделом
`## Обсуждение` (канал комментариев человек ↔ фабрика, зеркалится на борду).
Реестр вердиктов failure-analyst дополнен `APP_CHANGED` — поведение изменено
коммитом приложения без тикета (docs/06 D9).

**Прогон** (`runs/`):
```
NeedsTriage → Triaged → Closed (все падения: заведён баг / починен тест / признан flaky-карантин)
```

## 3. Правила оркестратора (аналог sm.json)

`state/rules.yaml` — декларативный список; верхний уровень `/qa-loop` идёт сверху
вниз, на артефакт берётся лок перед диспатчем, снимается по завершении. `Blocked` и
ручной лок `wip` всегда пропускаются (пауза для вмешательства человека).

С версии 2 каждому проходу предшествует **шаг 0** (`pre_steps`): снятие протухших
локов, SLA-sweep, приём переходов человека с борды и детект новой сборки из git
приложения — событийный контур тёмной фабрики, см. docs/06 §1. С версии 3 каждый
pre_step — идемпотентный скрипт (`scripts/*.py`), запускаемый верхним уровнем. Пример ниже —
сокращённый; актуальный список правил всегда в `state/rules.yaml`.

```yaml
rules:
  - when: "новый APK в state/app-under-test.yaml (изменился versionCode/hash)"
    dispatch: test-runner (suite=smoke, затем regression)
  - when: "run.status == NeedsTriage"
    dispatch: failure-analyst
  - when: "run имеет вердикт APP_BUG без linked_bug"
    dispatch: bug-reporter
  - when: "bug.status == Fixed && есть сборка новее bug.found_in"
    dispatch: fix-verifier
  - when: "вердикт TEST_BUG|SITE_CHANGED без resolution"
    dispatch: test-maintainer
  - when: "test-case.status == Approved && не автоматизирован"
    dispatch: test-automator
  - when: "область в стратегии помечена needs-design"
    dispatch: test-designer
  - when: "PROJECT.md приложения изменился с последнего анализа"
    dispatch: test-strategist
  - when: "расписание: ежедневно"
    dispatch: test-runner (suite=canary, mode=live)
```

Приоритет правил = порядок в списке (сначала разбор текущих прогонов и багов, потом
новая автоматизация — как в dmtools, где merge/check-правила идут раньше generation).

## 4. Что делает человек (и только он)

- Утверждает тест-кейсы P0/P1 (`Draft/Review → Approved`) — можно прямо с борды.
- Чинит баги в коде приложения и переводит их `Open → Fixed`.
- Решает судьбу `Blocked` и спорных вердиктов (`Rejected/Intended`).
- Одобряет изменения стратегии и большие рефакторинги фреймворка.
- Раз в день читает дайджест эскалаций (`state/escalations.md`, docs/06 §4).

Всё операционное (пересборка APK, перезапуск окружения, обновление борды, ответы на
типовые вопросы по багам, синхронизация тестов и документации с intended-изменениями)
человеку НЕ принадлежит — см. docs/06 §6. Ожидание человека всегда ограничено SLA
(`state/sla.yaml`): просрочка не блокирует конвейер молча, а поднимает варнинг.

## 5. Строение каждого агента (стандарт BMAD-стиля)

Файл `.claude/agents/<name>.md` обязан содержать:

1. **Персона и границы** — роль, что категорически нельзя (для всех: «не изменять
   файлы репозитория приложения»; для bug-reporter/fix-verifier — явно «не фиксить»).
2. **Триггер** — при каких статусах/событиях агент имеет право работать (проверяет
   сам в начале: если условие не выполнено — выходит без действий).
3. **Пошаговый воркфлоу** — нумерованные шаги с явными входами/выходами.
4. **Шаблон выходного артефакта** — ссылка на `docs/templates/*`.
5. **Чек-лист готовности** — агент проходит его перед снятием лока; не прошёл —
   статус не меняется, пишется причина.
6. **Протокол эскалации** — когда ставить `Blocked` и что писать человеку.
7. **Machine-readable результат (F2, с 2026-07-07)** — каждый воркер завершает
   ответ fenced-блоком ```yaml с ключом `agent_output` по
   `schemas/agent-output.schema.yaml` (agent, artifact,
   `result: success|blocked|degraded|failed`, summary, changed_files, evidence,
   next_rules, escalations). Диспетчер (/qa-loop) парсит блок
   (`scripts/agent_output.py`) и не разбирает свободный текст; требование
   передаётся в промпте каждого диспатча. Нет блока → исход считается degraded.

## 6. Пример сквозного цикла (happy + bug path)

1. Разработчик собирает новую версию → `state/app-under-test.yaml` обновлён.
2. Оркестратор: правило 1 → **test-runner** гоняет smoke, затем regression → 2 падения → `NeedsTriage`.
3. **failure-analyst**: падение №1 = `APP_BUG` (restore теряет filterProfiles), №2 = `SITE_CHANGED` (AO3 переименовал класс блёрба в записи canary).
4. **bug-reporter**: ищет дубликаты в `bugs/` → создаёт `BUG-014.md` (Open, severity=major, GWT-репро, логи), линкует к TC-031.
5. **test-maintainer**: перезаписывает recording, обновляет `web/selectors.py`, прогоняет затронутые тесты → падение №2 `Resolved`. Прогон → `Closed`.
6. Человек чинит BUG-014 в приложении, собирает сборку, ставит `Fixed`.
7. Оркестратор: правило 4 → **fix-verifier** гоняет TC-031 (+ smoke вокруг backup/restore) на новой сборке → зелёно → `Verified`.
