## 2. Подкладки (каждая — именованная ветка матрицы)

Подкладки уровня ПРИЛОЖЕНИЯ (сборки) требуют коммитов владельца в
app-under-test — фабрика код приложения не трогает (жёсткая граница).
Два режима: **A (full-fidelity, предпочтительно)** — владелец делает 2–3
реальных коммита по списку ниже; **B (симуляция)** — правка
`state/app-under-test.yaml` (bump versionCode/hash) имитирует «новую
сборку» без пересборки: маршрутизация веток проверяется, содержимое фикса
— нет. Режим — решение владельца до старта, фиксируется в отчёте.

**Режим B — жёсткое правило:** править ТОЛЬКО `versionCode`/hash в
`state/app-under-test.yaml`; `source_commit` НЕ трогать — `build_watch`
ключуется на нём (fetch origin/FETCH_HEAD): фейковый source_commit вызовет
реальный checkout/сборку либо эскалацию, сломав чистоту репетиции.

**Не упражняется первым (сжатым) прогоном** — явная оговорка (иначе
критерий §4.1 сочтёт дырами): sla-ветки `bug_open_*` («баг долго не
чинится», D3-часть), `blocked_any` (24 ч), `question_unanswered` (48 ч),
`quarantine_expired` (B3) — пороги длиннее окна; покрываются
backdate-подкладками при полном суточном прогоне (второй заход).

## 3. Таймлайн (первый прогон — СЖАТЫЙ цикл, решение владельца 2026-07-17)

Окно ~6 ч, heartbeat каждые 30 мин; шкала полного суточного прогона
(вторым заходом) — та же структура ×4.

- **T0:** гейт §1 подтверждён; Lead кладёт П5-сев/П10/П11/П16/П17/П19-сев
  и backdate П2; владелец кладёт борда-подкладки (П2-статус, П5–П9,
  П13–П15, П18) и уходит.
- **T0..T0+3ч («ночь»):** heartbeat-проходы разбирают борда-события;
  регрессия; триаж падений (П10/П11 пойманы и починены без человека).
- **T0+3ч («утро»):** владелец применяет подготовленные Lead'ом патчи
  режима A (П1, П3, П4, П12) — коммиты **+ push в origin** (build_watch
  сравнивает FETCH_HEAD, локального коммита недостаточно); единственное
  «появление разработчика», артефакты руками не трогает.
- **T0+3ч..T0+6ч («день»):** build_watch подхватывает сборки;
  D1/D3/D9/D11-ветки; canary по расписанию (если baseline поспел — §6).
- **T0+6ч:** стоп. Разбор.

## 4. Критерии успеха (все — измеримые)

1. **Покрытие веток:** каждая строка §2 имеет свидетельство из своей правой
   колонки (артефакт/строка журнала/статус). Непройденная ветка = находка
   (по чьей стороне — фабрики или спеки — решает разбор).
2. **Permission-окна = 0** за всё окно (свидетельство оператора + пост-фактум
   `/permission-audit` окна репетиции).
3. **Эскалации и реплики точны, каналы разведены:** (а) строка в
   `state/escalations.md`: П2, П16, П18, П19 — и только они; ложных
   эскалаций нет (каждая строка соотносится с подкладкой); (б) всплытие
   в `## Обсуждение`/дайджесте БЕЗ эскалации: П5 (`awaiting: dev`,
   dispute_count=1 < порога 2), П8 (ответ bug-responder).
4. **Длительности воркеров:** ни одной device-сессии с серией однотипных
   Timeout-фейлов без раннего blocked (fail-fast, чек 28 OS-репо — данные
   cc_usage окна).
5. **Дайджест полон измеримо:** каждая подкладка П1–П19 находима в
   дайджесте/factory-status по своему id/строке (таблица соответствия
   «подкладка → строка дайджеста» — обязательная часть отчёта).
6. **Локи не сиротеют:** к концу окна 0 живых локов старше lock_stale.

## 5. Разбор и повторяемость

- Отчёт репетиции — `runs/REHEARSAL-<дата>.md` (свободная форма + таблица
  §2 со свидетельствами); находки — штатными артефактами (bugs/ для дефектов
  фабрики — класс test_debt/механизм — решением Lead).
- Спека повторяемая: подкладки П10/П11/П16/П17 скриптуются
  (`scripts/rehearsal_seed.py` — кандидат, НЕ обязателен для первого
  прогона: первый раз — руками Lead по таблице).
- Триггер повторного прогона: крупное изменение механизмов конвейера
  (rules.yaml/transitions/pre_steps класса «меняет маршрутизацию») либо
  слово владельца перед релизной вехой.
