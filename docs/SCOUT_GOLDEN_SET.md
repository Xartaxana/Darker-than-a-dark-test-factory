# Scout Golden Set — регрессионный набор разведки (порт D-0057 OS-репо, 2026-07-10)

Порт механизма PROCESS/SCOUT_GOLDEN_SET.md из OS-репо
(D:\Improving_AI\Operating-System-for-LLMs): фиксированные
разведвопросы с закреплённым ключом, включая негативный и «отказ от
суждения» — типы, которые слабая модель проваливает тише всего
(плохой поиск неотличим от отсутствия). Ловит деградацию ЯРУСА
scout (правка промпта, смена модели); приёмку КАЖДОГО дайджеста
по-прежнему делает след (D-0046 OS-репо).

## Триггеры прогона

1. Правка .claude/agents/scout.md — прогон ДО коммита правки; строка
   в Runs log — тем же коммитом (правило 8 самого scout.md).
2. Смена модели яруса scout (frontmatter `model:` или привязка
   харнесса).
3. По назначению еженедельной калибровки OS-репо (рост
   failure_class=recon по этому деплою без внепланового прогона).

builder: набора НЕ будет — его выход проверяется артефактами на
каждой задаче (witness, D-0052 OS-репо). critic: набор в очереди
OS-репо (кандидатный дизайн: дифф с подсаженными дефектами).
13 QA-агентов конвейера решаются отдельно на данных конвейера
(оси 1/6 карты OS-репо).

## Протокол прогона

1. Сверка ключа ДО прогона: у каждого вопроса есть verify-команда;
   координатор прогоняет их или сверяет по свежему контексту.
   Уехавший ключ обновляется до прогона, тем же коммитом.
2. Диспатч scout: вопросы Q1–Q7 verbatim, обычный DoD — ответы +
   file:line + След; БЕЗ пометки «это тест».
3. Оценка координатором по ключу; каждый провал сначала проверяется
   на баг ключа (verify-команда), лишь потом трактуется как провал
   scout.
4. Строка в Runs log; при триггере-правке — тем же коммитом.
5. Contamination-чек: набор с ключом лежит в репо. Если След
   содержит чтение ЭТОГО файла — прогон помечается contaminated:
   принимается, только если каждый ответ прослежен к первичному
   источнику (file:line не в этом файле); пометка обязательна в
   строке Runs log.

## Критерий PASS

- Q3 (негативное) и Q4 (отказ от суждения) обязательны: провал
  любого из них = FAIL прогона независимо от счёта.
- Из остальных пяти допустима одна ошибка (итог >=6/7).
- FAIL -> правка role-файла не коммитится до разбора; повторный
  FAIL после починки промпта — решение человека/калибровки OS-репо.
