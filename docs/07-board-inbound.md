# 07 — Обратный канал борды (board-inbound): дизайн ядра

Реализует docs/06 §3 (Фаза 4.5 п.3). Борда перестаёт быть только зеркалом: человек
двигает карточку/пишет комментарий на GitHub Pages, фабрика применяет это к артефактам
(`bugs/`, `test-cases/`), которые остаются единственным источником правды.

## 1. Центральное затруднение

`board_sync.py` детерминирован: `board = f(артефакты)` и при каждом проходе делает
`rmtree(board/) + регенерация`. Значит правка человека прямо в `board/<KEY>/main.md`
затирается следующим sync. Наивный «diff борды с артефактами» не годится, потому что
`борда ≠ артефакт` возникает по ДВУМ разным причинам:

- **(H) человек** подвинул карточку на борде (борда впереди, артефакт позади);
- **(A) агент** изменил артефакт, борда ещё не пересобрана (артефакт впереди, борда позади).

Одно наблюдаемое состояние — две причины. Их нельзя различить, глядя только на два
текущих состояния. Нужна **третья точка отсчёта — снимок последней синхронизации**.

## 2. Решение: трёхсторонняя сверка через курсор

`board_sync` в конце каждого прохода записывает **курсор** `state/board-cursor.json` —
что борда и артефакты содержали В МОМЕНТ синхронизации (по построению они тогда
идентичны). Для каждого тикета: `{itype, artifact_status, board_status_id}`.

board-inbound (шаг 0 прохода, ДО любого sync) для каждого тикета читает три состояния:

| Источник | Что | Откуда |
|---|---|---|
| `cursor` | статус на момент прошлой синхронизации | `state/board-cursor.json` |
| `board_now` | статус карточки сейчас (мог подвинуть человек) | `board/<KEY>/main.md` frontmatter `status` |
| `artifact_now` | статус артефакта сейчас (мог изменить агент) | `bugs/…` / `test-cases/…` frontmatter `status` |

Классификация:
```
human_moved  = board_now    != cursor.board_status_id
agent_moved  = artifact_now != cursor.artifact_status
```
| human_moved | agent_moved | Действие |
|:-:|:-:|---|
| нет | нет | нет изменений |
| **да** | нет | **применить** переход человека к артефакту (если в whitelist §3) |
| нет | да | ничего: sync догонит борду (нормальный рассинхрон) |
| **да** | **да** | **конфликт**: обе стороны тронули один тикет → `Blocked` + эскалация |

Курсор — единственная надёжная база отсчёта; без него причины H и A неразличимы.
Хранится в `state/` (наша сторона), правки человека — в `board/` (его сторона), поэтому
`git pull` человеческих коммитов и наш курсор не пересекаются.

### 2.1. Кто продвигает курсор (критично для автономности)

Курсор пишут ДВОЕ, и это осознанно:
- **`board_sync`** в конце прохода — авторитетный финальный снимок (борда пересобрана
  из артефактов, `board_status == artifact_status`).
- **`board_inbound`** в конце своей работы — продвигает курсор к ТЕКУЩЕМУ
  согласованному состоянию (фактические, возможно различные, `board_status` и
  `artifact_status`) сразу после применения переходов.

Почему board_inbound ОБЯЗАН сам продвигать курсор, а не полагаться на board_sync:
если sync не отработает в конце прохода (краш, пропуск, ошибка между pre_step и
финалом), следующий проход при СТАРОМ курсоре увидит `борда = цель человека`,
`артефакт = применённая цель`, `курсор = исходный` → `human_moved И agent_moved` →
**ложный конфликт** → баг ошибочно уходит в `Blocked` + эскалация. Для фабрики, которая
должна сутки работать без человека и переживать краши, это недопустимо. Правило общее:
**курсор продвигает тот, кто обработал изменения** (потребитель), а не только тот, кто
их произвёл (`board_sync`). Регрессия закрыта тестом
`test_cursor_advanced_prevents_false_conflict`.

Два писателя не конфликтуют: они пишут в разное время (inbound — pre_step, sync —
финал прохода); финальная запись sync авторитетна. Если sync не сработал — запись
inbound остаётся корректной базой, и следующий проход стабилен (noop).

## 2.2. Почему у самого TrackState такой проблемы нет

TrackState — git-native трекер: **единственный источник правды у него — сам git-репозиторий**,
и правит его только он же через приложение. Конкурентные записи (два пользователя,
приложение vs внешняя правка) он решает **оптимистичной блокировкой по ревизиям**:
каждая запись файла несёт `expectedRevision` (SHA git-блоба, который редактор видел
последним); если на диске файл с тех пор изменился (другой SHA) — запись отклоняется
(`ensureCleanWorktree` + сверка `expectedRevision`, см. `trackstate_repository*.dart`,
`github_trackstate_provider.dart`). Это ТЕКСТОВЫЙ конфликт в рамках ОДНОГО источника.

У нас конфликт другого рода — **семантический, между ДВУМЯ источниками**: артефакты
(правда) и борда (генерируемое зеркало). `board_sync` намеренно перегенерирует `board/`
целиком из артефактов, затирая правки борды, поэтому блокировка по SHA-блобам к нам
неприменима — мы и так перезаписываем. Вопрос не «изменился ли файл», а «чьё изменение
статуса тикета выигрывает — человека или агента». TrackState с этим не сталкивается,
потому что НИКОГДА не регенерирует свою борду из внешней правды: борда и есть правда.
Мы выбрали проекцию (docs/05: артефакты — источник, борда — зеркало + узкий обратный
канал), и цена этого выбора — двойной писатель, который решается курсор-сверкой §2.
Оптимистичная блокировка TrackState и наш курсор — ответы на разные задачи.

## 3. Whitelist переходов человека (docs/06 §3)

Всё, что не в whitelist, игнорируется с варнингом (следующий sync вернёт карточку):

- **bug:** `Open|Reopened → Fixed | Rejected | Intended | Blocked`; `любой → Open`
  (ручное переоткрытие);
- **test-case:** `Draft|Review → Approved`; `любой → Review` (вернуть на доработку);
- **run:** переходы не принимаются (только комментарии).

Переход применяется как правка `status:` во frontmatter артефакта + обновление
`status_since` (для SLA-sweep) и `updated`. `reopen_count` инкрементируется при
`→ Reopened`. Данные, которых нет в статусе (причина Rejected и т.п.), приходят
комментарием (§5), не блокируют применение статуса.

## 4. Конфликт (H и A на одном тикете)

Правда не «затирается последним писавшим». Артефакт → `Blocked`, строка в
`state/escalations.md` (id, тикет, оба изменения, время), реплика в `## Обсуждение`.
Дальше решает человек. Так дизайн честно отказывается угадывать намерение.

## 5. Комментарии (делегируется как обвязка)

Комментарии карточки ↔ раздел `## Обсуждение` артефакта (docs/06). Конкретный формат
хранения комментариев в TrackState на диске нужно реверс-инжинирить (в текущих тикетах
комментариев ещё нет) — это исследовательская рутина. Ядро §2–§4 задаёт **интерфейс**:
`collect_board_comments(key) -> list[Comment]` и `append_discussion(artifact, comment)`;
реализацию парсинга TrackState-комментариев заполняет делегированный воркер.

## 6. Границы ядра vs обвязки

- **Ядро (спроектировано и написано сильной моделью):** курсор-контракт (запись в
  `board_sync`, чтение в `board_inbound`), трёхсторонняя сверка, whitelist-валидация,
  примитивы применения статуса к артефактам, конфликт→Blocked+escalations, dry-run.
- **Обвязка (делегируется):** реверс формата комментариев TrackState и их синхронизация;
  вызов из `pre_steps` оркестратора (`git pull` борды перед сверкой); интеграция в
  `board.ps1`; юнит-фикстуры на каждый переход whitelist; форматтер `escalations.md`.

## 7. Порядок в автономном проходе

```
pre_step board_inbound:
  git pull                       # человеческие коммиты борды с origin  [обвязка]
  reconcile(cursor, board, arts) # §2 классификация
  apply(actions)                 # §3 переходы, §4 конфликты, §5 комментарии
... правила rules.yaml, агенты работают с артефактами ...
конец прохода:
  board_sync.build()             # регенерация board/ + запись курсора
  git commit board + cursor; git push
```
