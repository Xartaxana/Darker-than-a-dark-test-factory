# 01 — Стратегия тестирования AO3 Reader

Статус: Draft · Владелец: test-strategist (агент) · Пересматривается при каждом мажорном изменении приложения

## 1. Объект тестирования

Android-приложение ao3-wrapper (minSdk 26, targetSdk 36): WebView-обёртка над
archiveofourown.org + нативные Compose-экраны (Library, Settings) + локальная БД Room +
JS-bridge (`ao3_bridge.js`), инжектируемый в каждую страницу.

Полная карта функциональности — `PROJECT.md` в репозитории приложения. Это главный
источник требований: API у AO3 нет, спецификация = PROJECT.md + фактическое поведение.

## 2. Ограничения и следствия для стратегии

| Ограничение | Следствие |
|---|---|
| Нельзя менять код приложения | Только black-box UI-тестирование через Appium/UiAutomator2; никаких testTag, никаких unit-тестов внутри приложения |
| Нет API у AO3, весь контент через WebView | Гибридное тестирование: нативный слой (Compose) + web-слой (DOM внутри WebView) |
| Живой сайт AO3: Cloudflare bot-check, нестабильная сеть, изменяющийся DOM | Два режима прогона: **live** (минимальный smoke) и **replay** (детерминированная регрессия на записанных страницах через mitmproxy) |
| Compose-экраны без testTag | Локаторы: content-description, текст, иерархия accessibility tree; хрупкость закладываем в дизайн локаторов (один локатор — одно место в коде) |
| Данные только локально (Room, SharedPreferences) | Управление состоянием через adb (`pm clear`, seed через restore-flow или `run-as` на debug-сборке) |

## 3. Уровни тестирования

| Уровень | Что проверяет | Инструмент | Доля |
|---|---|---|---|
| E2E UI (нативный + WebView) | Основные пользовательские сценарии | Appium + UiAutomator2 | ~70% |
| Web-слой (bridge-канарейки) | Живой DOM AO3 всё ещё соответствует селекторам `ao3_bridge.js` | Отдельный canary-suite (live) | ~10% |
| Данные | Целостность Room после операций: backup/restore, clear, scan for downloads | E2E + проверка экспортированного JSON | ~15% |
| Нефункциональные | Холодный старт, восстановление после kill, поворот экрана, offline | E2E-сценарии | ~5% |

## 4. Режимы прогона

### live (smoke, canary)
Настоящий AO3, залогиненная тестовая учётка (cookie WebView). Только P0-смоук
(5–10 тестов) и canary-suite селекторов. Падения здесь чаще всего = изменение сайта
или bot-check, а не баг приложения — триаж это учитывает.

### replay (регрессия, по умолчанию)
Эмулятор + mitmproxy в режиме replay: записанные страницы AO3 (листинги, страница
работы, форма Sort&Filter) версионируются в `framework/data/recordings/`. Детерминизм,
скорость, независимость от Cloudflare. Требование: AVD-образ без Google Play (root)
для установки CA mitmproxy в системное хранилище — WebView не доверяет user-CA.

## 5. Риск-матрица (вероятность × влияние, 1–3)

| ID | Категория | Риск | P | I | Счёт | Митигация |
|---|---|---|---|---|---|---|
| R-01 | DATA | Потеря пользовательских данных при backup/restore/clear/delete | 2 | 3 | 6 | P0-покрытие всех разрушающих операций и restore старого формата |
| R-02 | TECH | AO3 меняет DOM → bridge перестаёт работать (бейджи, Rate-кнопки, фильтры) | 3 | 3 | 9 | Ежедневный canary-suite на live; триаж-категория SITE_CHANGED |
| R-03 | TECH | Cloudflare bot-check ломает загрузку/восстановление скролла | 3 | 2 | 6 | Спец-кейсы на interstitial (peek scroll restore); replay-режим для остального |
| R-04 | DATA | Рассинхронизация Room ↔ визуальные бейджи в WebView | 2 | 3 | 6 | P0: рейтинг из панели и из листинга, проверка бейджа без перезагрузки |
| R-05 | TECH | Скачивание: cookies, SAF-папки, orphan-скан, relink | 2 | 2 | 4 | P1-покрытие download-flow, включая смену папки |
| R-06 | BUS | Фильтрация скрывает не то (или не скрывает Disliked) | 2 | 2 | 4 | P1: приоритетные правила видимости из PROJECT.md §Priority rules |
| R-07 | OPS | Флаки самих тестов на гибридном стеке | 3 | 2 | 6 | Явные ожидания, retry-политика только для live, карантин flaky |
| R-08 | TECH | Состояние табов (10 шт., undo, персистентность, скролл) | 2 | 2 | 4 | P1-покрытие tab lifecycle, включая рестарт приложения |

## 6. Приоритеты покрытия

- **P0 (smoke, гоняется на каждой новой сборке):** запуск и загрузка AO3; выставление
  каждого из 5 рейтингов со страницы работы и из листинга; бейдж появляется без
  перезагрузки; работа попадает в нужную вкладку Library; скрытие Disliked при
  включённой фильтрации; Clear all ratings с подтверждением; backup → clear → restore
  возвращает данные.
- **P1 (регрессия):** мультитаб (создание, лимит 10, undo-закрытие, фоновые табы,
  персистентность после рестарта); фильтры Library (фандом, word count, теги,
  downloaded-only) и все 5 сортировок; download-flow (авто при Loved, ручной из
  Library, открытие file://, удаление, scan for downloads); фильтр-профили
  (сохранение из формы AO3, применение, удаление); заметки и личные теги; error page +
  Retry; темы Light/Dark/System.
- **P2:** длинный хвост: поворот экрана, kill/restore процесса, offline-переходы,
  граничные значения (пустая Library, работа без word count, blank author в
  сортировке), restore старого bare-array формата бэкапа.
- **P3:** косметика: цвета бейджей в dark/light, иконки сортировки, snackbar-тексты.

## 7. Критерии качества прогона (gate)

- P0 100% pass → сборка пригодна к использованию.
- P1 ≥ 95% pass, все падения оттриажены (см. docs/03, роль failure-analyst).
- Любое падение обязано получить вердикт: `APP_BUG | TEST_BUG | SITE_CHANGED | ENV_ISSUE | FLAKY` — «просто красный» прогон не закрывается.

## 8. Вне скоупа

| Что | Почему | Как закрыт риск |
|---|---|---|
| Функциональность самого сайта AO3 | Не наш продукт | Canary проверяет только контракт селекторов bridge |
| Unit/integration-тесты внутри приложения | Запрет на изменения кода | Логика покрывается через UI-наблюдаемое поведение |
| Публикация/совместимость с парком устройств | Личное sideload-приложение | Один эталонный AVD (API 34) + опционально реальное устройство |
| Нагрузочное/security | Локальное личное приложение без бэкенда | — |

## 9. Области, требующие тест-дизайна (`needs-design`)

Текущее покрытие Approved/Automated-кейсами: только запуск и загрузка AO3
(TC-001..005, area=smoke, P0, Automated). Всё остальное — ниже, по приоритету.
Один Draft-кейс уже существует в area=library (TC-006, о расхождении подписей
вкладок, связан с BUG-001) — он не закрывает область library целиком и не
меняет её статус `needs-design`.

Раскладка папок соответствует `test-cases/README.md`:
`rating`, `visibility`, `tabs`, `library`, `downloads`, `filter-profiles`,
`backup`, `settings`, `errors`, `canary`.

### P0 — needs-design

- **rating: пять рейтингов со страницы работы** (risk: R-04) — на `/works/{id}`
  открыть панель `RatingMenu`, поставить каждый из 5 рейтингов (Loved/Liked/
  Read/Pending/Disliked) по очереди, убедиться в записи в Room (косвенно —
  через бейдж/Library) и что повторный тап на выбранной кнопке снимает рейтинг
  (deselect → `rating=null`).
- **rating: пять рейтингов из листинга** (risk: R-04) — на листинговой странице
  тап по инжектированной Rate-кнопке (28px circle) открывает нативный
  bottom-sheet, каждый из 5 рейтингов сохраняется и обновляет бейдж на карточке
  блёрба.
- **rating: бейдж обновляется без перезагрузки страницы** (risk: R-04) —
  после простановки рейтинга (с панели или из листинга) бейдж/цвет кнопки Rate
  меняется мгновенно, без видимого reload WebView; проверить оба входа
  (панель и листинг) и что `applyRatings` синхронизирует все открытые вхождения
  того же work на странице.
- **visibility: скрытие Disliked при включённой фильтрации** (risk: R-06) —
  дефолтное правило видимости из PROJECT.md §Priority rules: work с
  `rating=DISLIKE` скрыт на листинге, когда filtering включён и Disliked в
  hidden-set; work без рейтинга или с `rating=null` (comment-only) никогда не
  скрывается; выключение "Enable filtering" в Settings показывает всё
  независимо от рейтинга.
- **library: работа попадает в нужную вкладку** (risk: R-04) — после простановки
  рейтинга work появляется в соответствующей вкладке Library (Loved/Liked/Read/
  Pending/Disliked); снятие/смена рейтинга перемещает work между вкладками;
  comment-only (rating=null) не появляется ни в одной рейтинговой вкладке.
- **settings/backup: Clear all ratings с подтверждением** (risk: R-01) — диалог
  подтверждения перед очисткой; после подтверждения все `WorkRating` строки
  удалены, Library пуста по всем вкладкам, бейджи на открытых страницах AO3
  сбрасываются; отмена диалога не удаляет данные.
- **backup: backup → clear → restore возвращает данные** (risk: R-01) —
  экспорт JSON (`{"version":2,"works":[…],"filterProfiles":[…]}`) через SAF,
  затем Clear all ratings, затем Restore из того же файла: количество и
  содержимое строк (rating, comment, tags, fandom, word_count) совпадает с
  исходным; результат показывается в `AlertDialog` (counts/error).

### P1 — needs-design

- **tabs: мультитаб (создание, лимит 10, undo, персистентность)** (risk: R-08)
  — создание вкладок кнопкой в strip, достижение лимита `MAX_TABS=10` (что
  происходит при попытке превысить — новая вкладка не создаётся/UI сообщает);
  swipe-up закрывает вкладку, snackbar "Tab closed" с Undo восстанавливает её
  на исходной позиции с историей навигации и скроллом; до 5 последних закрытий
  доступны для undo, более старые вытесняются; персистентность URL+scroll после
  рестарта приложения (adb kill/relaunch); long-press по ссылке открывает
  фоновую вкладку.
- **library: фильтры и сортировки** (risk: R-06 частично, иначе не покрыт
  риском напрямую — оценить как BUS/DATA при дизайне) — `LibraryFilters`:
  фандом, word count min/max, downloaded-only чекбокс, теги (AND), free-text
  search, применяются клиентски до сортировки; 5 вариантов `LibrarySort`
  (Last read по умолчанию, Word count ↓/↑, Author A-Z, Rating — только вкладка
  Downloads/Files), works без сортируемого поля (нет word_count, пустой
  author) уходят в конец списка; смена сортировки сбрасывает скролл вкладки к
  началу.
- **downloads: полный flow** (risk: R-05) — авто-скачивание при простановке
  Loved с включённым "Auto-download saved works" (JS находит download-ссылку
  на странице работы → `DownloadRepository.downloadFromUrl` с cookies WebView);
  ручное скачивание из Library (`downloadWork`, сначала регэксп по странице
  работы, затем сохранение); открытие скачанного файла через `file://` (в т.ч.
  проверка инжекции `reader.css` и viewport meta — иначе текст мелкий);
  удаление файла (`deleteDownload`, оставляет строку rating/note/tags) vs
  полное удаление работы (`deleteWork`, удаляет и файл, и строку); "Scan for
  downloads" — ручной триггер (всегда показывает диалог результата, включая
  "0 файлов"), авто-триггер при смене папки загрузок (`silent`, диалог только
  если что-то реально перелинковано/добавлено), авто-триггер после Restore
  (результат сканирования складывается в тот же диалог, что и restore, без
  повторного всплывающего окна).
- **filter-profiles: сохранение/применение/удаление** (risk: не в текущей
  риск-матрице явно — предложить R-09 BUS "потеря/некорректное применение
  сохранённого запроса" при дизайне) — кнопка "Save filter" в форме AO3
  Sort&Filter сохраняет `queryString` под именем через диалог; применение
  сохранённого фильтра из панели на листинговой странице; удаление профиля в
  Settings.
- **rating: заметки (comment) и личные теги (tags)** (risk: не в текущей
  риск-матрице явно — предложить R-10 DATA/BUS при дизайне) — comment-only
  запись (rating=null, comment≠null) не появляется в рейтинговых вкладках
  Library и не скрывается фильтрацией; note-кнопка (карандаш) на листинге
  открывает overlay с развёрнутым полем комментария; личные теги не влияют на
  видимость (PROJECT.md явно это гарантирует) и подсвечивают совпадающие
  AO3-теги индикатором.
- **errors: error page + Retry** (risk: R-03 частично, TECH) — `onReceivedError`
  на main-frame показывает кастомную themed error page (не Chrome-дефолт) с
  Retry-ссылкой на упавший URL; загружена через `about:blank` как base URL,
  поэтому не попадает в историю навигации и не должна ловить bridge-инъекцию
  на самой error-странице.
- **settings: темы Light/Dark/System** (risk: не в текущей риск-матрице явно —
  учитывая известную хрупкость (app-under-test/CLAUDE.md: "Dark mode has broken
  four times"), предложить отдельный TECH-риск R-11 при дизайне) — переключение
  темы без пересоздания Activity (Compose recomposes in place); WebView dark
  mode применяется на следующий холодный старт, а не мгновенно; System следует
  за настройками ОС; проверить согласованность бейджей/цветов между
  light/dark (детальные цветовые сверки — P3, здесь только функциональное
  переключение).

## 10. Открыто для утверждения человеком

Раздел §9 добавляет два риска, не входивших в матрицу §5 (filter-profiles,
notes/tags) и один потенциальный (темы, TECH) — они помечены как
"предложить R-09/R-10/R-11 при дизайне" вместо немедленного добавления в
таблицу §5, чтобы не переопределять риск-матрицу без утверждения. Прошу
подтвердить: (а) стоит ли формализовать эти три риска в §5 сейчас или
дождаться, пока test-designer нащупает реальные сценарии отказа; (б) приоритет
P1 для тем и filter-profiles — устраивает, или что-то из них поднять до P0/
опустить до P2.
