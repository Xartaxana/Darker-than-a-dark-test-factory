# 06 — Тёмная фабрика: события, реакции на разработчика, SLA, автономность

Целевой воркфлоу (утверждён владельцем, 2026-07-03):

> Разработчик пушит изменения в репозиторий приложения и переводит карточку бага на
> борде в «тестирование». Фабрика сама замечает это, собирает сборку, проверяет фикс,
> при необходимости обновляет тесты и документацию, переводит тикет дальше. Человек
> приходит раз в день; всё это время фабрика работает без единого permission-окна.
> Если проблема висит слишком долго — фабрика сама поднимает варнинг человеку.

Этот документ — недостающий контур поверх docs/03 (агенты) и docs/05 (борда):
**входные события**, **матрица реакций на действия разработчика**, **SLA/эскалации**
и **правила автономности**. Части, которые ещё не реализованы, помечены `[план]` и
внесены в роадмап (docs/04, Фаза 4.5).

## 1. Событийная модель (что будит фабрику)

Сейчас конвейер просыпается только вручную (`/qa-loop`). Для тёмной фабрики нужно
четыре входных сигнала; все они сводятся к **шагу 0** каждого прохода `/qa-loop`
(`state/rules.yaml → pre_steps` — идемпотентные скрипты, запускаемые верхним
уровнем; runtime-модель — docs/03 §1), а сам проход запускается по расписанию (§5).

| # | Событие | Как детектится | Что запускает |
|---|---|---|---|
| E1 | Пуш в репозиторий приложения | `[план]` build-watch: `git fetch` в `app-under-test/`, новые коммиты на default-ветке → `gradlew assembleDebug` → обновить `state/app-under-test.yaml` (versionCode/hash/commit) | Правило «новая сборка → smoke, regression» |
| E2 | Человек перевёл карточку на борде | `[план]` board-inbound: сравнить статусы карточек `board/` (HEAD) с артефактами; принять только переходы из whitelist §3 | Соответствующее правило (fix-verifier, test-maintainer, test-automator…) |
| E3 | Комментарий человека на карточке/в артефакте | board-inbound переносит комментарий в раздел `## Обсуждение` артефакта с меткой `awaiting: qa` | Правило «ответить на вопрос разработчика» |
| E4 | Расписание | Ночная регрессия, дневной canary, heartbeat-проход qa-loop каждые N часов | test-runner / полный проход |
| E5 | Таймер SLA | sla-sweep на шаге 0: сравнение `status_since` артефактов с порогами `state/sla.yaml` | Эскалация (§4), не диспатч агента |

Принцип: **артефакты (`test-cases/`, `bugs/`, `runs/`) остаются единственным
источником правды**. Борда — двунаправленный интерфейс для человека: наружу —
проекция (`board_sync.py`, как сейчас), внутрь — только узкий whitelist переходов §3.

## 2. Матрица реакций на действия разработчика

Главный пробел текущих правил: описан только happy-path (`Fixed → fix-verifier`).
Полная матрица — обязательная спецификация для оркестратора и агентов:

| # | Действие разработчика | Реакция фабрики | Кто |
|---|---|---|---|
| D1 | Перевёл баг в `Fixed`, есть сборка новее `found_in` | Прогнать связанные TC на новой сборке → `Verified` или `Reopened` (как сейчас) | fix-verifier |
| D2 | Перевёл в `Fixed`, но новой сборки нет | Ничего не гонять; ждать E1. Если сборка не появилась за `bug_fixed_waiting_build` — варнинг «Fixed без сборки» | sla-sweep |
| D3 | Принёс новую сборку, но открытые баги не пометил `Fixed` | Обычный smoke/regression; отдельно — перепроверить репро открытых `blocker/critical` багов, обновить поле `last_seen_in`. Баг, переживший ≥ `bug_open_*`-порог, — варнинг «баг долго не чинится» | test-runner, fix-verifier (mode=still-repro), sla-sweep |
| D4 | Отклонил баг: `Rejected` («не баг» / «не воспроизводится») | Перепроверить репро изолированно на актуальной сборке. Воспроизвёлся → приложить усиленные доказательства (пошаговые скриншоты, logcat, при возможности запись экрана), комментарий в `## Обсуждение`, `awaiting: dev`, инкремент `dispute_count`; при `dispute_count ≥ reopened_pingpong` — эскалация человеку. НЕ воспроизвёлся → согласиться: финализировать `Rejected`, при необходимости test-maintainer правит ложноположительный тест | fix-verifier (mode=recheck-rejected), bug-reporter, test-maintainer |
| D5 | Пометил `Intended` («так задумано») | Согласовать ожидания с реальностью: обновить тест-кейсы, автотесты, локаторы; test-strategist фиксирует изменение поведения в стратегии; расхождение с PROJECT.md — отметить в баге документации | test-maintainer, test-strategist |
| D6 | Оставил вопрос/комментарий на карточке | Ответить из артефактов (репро, логи, версии) в `## Обсуждение`, переключить `awaiting: dev`. Не хватает данных для ответа → перегнать репро и приложить. Вопрос вне компетенции (продуктовое решение) → `Blocked` + эскалация | bug-reporter (role=responder) |
| D7 | Фикс сломал что-то другое (регрессия на fix-сборке) | Новый баг со ссылкой `regression_of: BUG-xxx`; исходный баг закрывается `Verified` только если его собственные TC зелёные — «сломал соседнее» не держит исходный баг открытым | failure-analyst, bug-reporter |
| D8 | Пинг-понг: баг переоткрыт ≥ `reopened_pingpong` раз | Варнинг человеку: «фикс не сходится, нужно живое обсуждение»; баг → `Blocked` | sla-sweep |
| D9 | Изменил поведение приложения без тикета (тесты упали, бага не было) | failure-analyst при триаже обязан свериться с `git log` app-under-test между сборками: коммит явно описывает изменение → вердикт `APP_CHANGED` (аналог SITE_CHANGED для приложения) → test-maintainer обновляет тесты + strategist фиксирует; иначе — обычный `APP_BUG` | failure-analyst |
| D10 | Пометил баг дубликатом другого | Слить ссылки: `duplicates` у копии, TC/прогоны перелинковать на оригинал | bug-reporter |
| D11 | Несколько пушей подряд за короткое время | Coalescing: тестируется только последняя сборка; промежуточные фиксируются в `app-under-test.yaml`-истории прогона как пропущенные | build-watch |
| D12 | Перевёл TC `Review → Approved` на борде | board-inbound применяет к артефакту → штатное правило «автоматизировать Approved-кейс» | test-automator |
| D13 | Явно принял риск / решил не чинить в этом релизе (`resolution: accepted_risk\|wontfix`) | Держим linked TC активными, не блокируем release-gate. Обязателен `resolution_comment` (обоснование человека — validate_frontmatter ловит отсутствие как ошибку). sla-sweep больше не шлёт периодический SLA-варнинг по severity для этого бага — решение уже принято, нагрузка была бы шумом | sla-sweep (гасит bug_open_*), test-strategist (учитывает в стратегии) |
| D14 | Пометил `known_issue: true` («баг подтверждён, релиз идёт с ним») | Баг остаётся `Open`/`Reopened`, попадает в отдельную секцию дайджеста `factory-status.md`; bug-reporter при новом `APP_BUG` сверяется с known-issue багами В ПЕРВУЮ очередь (не плодит дубли); still-repro (D3) проверяет его на каждой новой сборке независимо от severity («не ухудшился»); sla-sweep не шлёт периодический SLA-варнинг по severity (как D13) | bug-reporter (дедуп), fix-verifier (mode=still-repro), sla-sweep |

Новые поля frontmatter бага для поддержки матрицы (внесены в шаблон
`docs/templates/bug-report.md`): `status_since`, `last_seen_in`, `reopen_count`,
`dispute_count`, `regression_of`, `awaiting` (`dev|qa|none`) + раздел `## Обсуждение`
(журнал реплик `[автор @ ISO-время]`, единый канал комментариев борда↔артефакта).

**B1/B2/B5 (docs/09 Этап 2, добавлено 2026-07-07):** `resolution`
(`accepted_risk|wontfix` + обязательный `resolution_comment`, D13), `known_issue`
(`true|false`, D14), `blocked_reason` (`environment|missing_fixture|product_decision|
dev_answer|permissions`, все три схемы — bug/test-case/run, т.к. `Blocked` есть в
каждой машине). `blocked_reason` обязателен по смыслу при `status: Blocked`
(validate_frontmatter — WARN, не ERROR: человек с борды может не успеть заполнить
сразу); детерминированные factory-переходы (D8/D4 pingpong в sla_sweep, конфликт
борда↔артефакта в board_inbound) проставляют `product_decision` автоматически.

**B3/B4 (docs/09 Этап 2, добавлено 2026-07-07):** lifecycle автотеста и test debt.

- **B3 — машина `automation`** (`schemas/transitions.yaml`, поле `automation_status`
  на тест-кейсе, живёт только при `status: Automated`): `active → quarantined`
  (FLAKY, оформляет failure-analyst с обязательными `quarantine_reason/since`) →
  `active` (ТОЛЬКО test-maintainer, 3 зелёных прогона подряд); `needs_maintenance`
  (детерминированная поломка); `deprecated` (решение человека/стратегии) → `retired`
  (терминал). Карантин не бесконечен: sla_sweep, правило `quarantine_expired` —
  дедлайн `quarantine_expiry`, а без него `quarantine_since + sla.quarantine_max`
  (дефолт 336 ч). Правило конвейера «Починить автотест в карантине» стоит ВЫШЕ
  новой автоматизации (сначала гасим долг, потом наращиваем покрытие).
**F1 (docs/09 Этап 2, добавлено 2026-07-07):** гейт ревью нового автотеста.
`Approved → Automated` переводит ТОЛЬКО **test-reviewer** (матрица C3) после
чек-листа: arch_check (C1), traceability (`allure.id`/маркеры/traceability.md),
соответствие теста GWT кейса по смыслу, фикстуры (ownership/cleanup), flake-риск,
независимый прогон. test-automator заполняет `automated_by` и добивается 3 зелёных,
но кейс остаётся `Approved`; замечания — `review: changes_requested` + раздел
«## Ревью автотеста» в кейсе → доработка автором → повторное ревью. Ревьюер не
правит тестовый код (иначе он автор). Два круга по одному пункту — эскалация.

- **B4 — test debt** живёт в `bugs/` с `type: test_debt` (+ `debt_kind:
  flaky_test|slow_test|missing_fixture|weak_locator|obsolete_test_case|
  missing_evidence|broken_environment`). Отличия от app_bug: чинит фабрика —
  guard-переходы `Open|Reopened → Fixed by test-maintainer/test-automator
  (guard: type=test_debt)` в матрице; `Fixed` не ждёт сборку приложения
  (fix-verifier верифицирует сразу, sla-правило `bug_fixed_waiting_build` молчит);
  severity-SLA `bug_open_*` не шумит — долг виден в отдельной секции
  `factory-status.md`. Для app_bug `Fixed` по-прежнему ставит только человек.

Новый вердикт триажа: `APP_CHANGED` (D9) — добавить в реестр вердиктов
failure-analyst рядом с `SITE_CHANGED`.

## 3. Обратный канал борды (board-inbound) `[план]`

`board/` генерируется из артефактов, но TrackState умеет и записывать (переходы,
комментарии — коммитами в `board/`). Правила обратного канала:

- **Whitelist переходов от человека** (всё остальное игнорируется с варнингом):
  - bug: `Open|Reopened → Fixed | Rejected | Intended | Blocked`; `любой → Open` (ручное переоткрытие);
  - test-case: `Draft|Review → Approved`; `любой → Review` (вернуть на доработку);
  - run: только комментарии.
- Комментарии карточек переносятся в `## Обсуждение` артефакта (и обратно —
  `board_sync.py` рендерит обсуждение в карточку).
- **Конфликт** (человек и агент изменили статус по-разному между проходами):
  артефакт → `Blocked`, причина в `state/escalations.md`. Правда не «затирается
  последним писавшим» — конфликт всегда решает человек.
- После применения inbound-переходов — обычный `board_sync.py`, борда снова
  консистентна артефактам.
- Реализация: `scripts/board_inbound.py`, вызывается оркестратором на шаге 0.
  Пока не реализовано — переходы D1–D12 человек делает правкой frontmatter
  артефакта (как сейчас), матрица §2 действует уже сегодня.

Чтобы разработчик мог двигать карточки удалённо, борду надо опубликовать
(GitHub Pages, docs/05) — иначе обратный канал доступен только на этой машине.

## 4. SLA и эскалации

Пороги — в `state/sla.yaml` (создан). sla-sweep выполняется на шаге 0 каждого
прохода: артефакты с просроченным `status_since` попадают в `state/escalations.md`.

Каналы варнинга (по нарастающей, настраивается в sla.yaml):

1. **`state/escalations.md`** — реестр активных варнингов (id, артефакт, причина,
   с какого времени, что нужно от человека). Проекция на борду: метка `attention`
   на карточке — человек видит проблемы первым взглядом на доску.
2. **Дневной дайджест** — один сводный файл/комментарий в день (`digest:` в
   sla.yaml), а не десять пингов: «3 бага ждут фикса > 4 суток, TC-021 заблокирован
   9 дней, спайк B не движется». `[план]` доставка наружу (push-уведомление CC /
   почта / мессенджер — решение владельца).
3. **Немедленный алерт** — только для: blocker-баг найден; конвейер сам встал
   (ENV_ISSUE дважды подряд, protected-ресурс недоступен); конфликт борды.

Правило дедупликации: варнинг по одному артефакту и одной причине поднимается один
раз, дальше живёт в escalations.md до разрешения; повторный алерт — только при
ухудшении (severity выросла, второй порог пробит).

## 5. Автономность: расписание, самовосстановление, разрешения

### Расписание `[план: включить после Фазы 3]`
- **Heartbeat**: `/qa-loop` каждые 2–4 часа (Windows Task Scheduler → `claude -p "/qa-loop"`
  или scheduled task Claude Code). Каждый проход начинается с pre_steps (E1–E3, E5) —
  поэтому отдельные «вотчеры-демоны» не нужны, опрос дешёвый.
- **Ночь**: полная регрессия. **День**: canary на живом AO3.
- Проходы не накладываются: лок-файл `state/loop.lock` с PID/временем; живой лок
  моложе `lock_stale` → новый проход выходит сразу.

### Самовосстановление
- `ENV_ISSUE` (эмулятор умер, Appium не отвечает, порт занят): один автоматический
  retry с пересозданием окружения (`Stop-NodeProcesses` → `Start-Emulator` →
  `Start-Appium`). Второй подряд — стоп + немедленный алерт (§4.3). Не долбить среду
  в цикле.
- **Протухшие локи**: lock старше `lock_stale` без записи о завершении в
  orchestrator-log = упавший агент → снять лок, залогировать, артефакт вернуть в
  прежний статус. (Сейчас упавший агент навсегда блокирует артефакт — см. TC-021.)
- `FLAKY`: после N подряд флапов — карантин (`@pytest.mark.quarantine`) без участия
  человека, запись в дайджест.

### Разрешения — «permission-окно = дефект конвейера»
Действующие практики (HANDOFF, `/permission-audit`) остаются; добавляются принципы:

1. **Никогда не ждать интерактивного подтверждения.** Если инструмент/команда не
   покрыта allowlist — агент НЕ вызывает её «на удачу», а деградирует: пропускает
   шаг, пишет причину в orchestrator-log и (если шаг критичен) поднимает эскалацию.
   Висящее окно в пустой комнате = сутки простоя.
2. Каждое всплывшее окно за автономный период — инцидент: `/permission-audit`
   запускается регулярно (входит в еженедельную ревизию), причина устраняется
   правкой allowlist или инструкции агента, а не «Allow always» руками.
3. Новые команды сначала обкатываются в присутствии человека, затем вносятся в
   версионируемый `.claude/settings.json`; `settings.local.json` периодически
   чистится (шаг 4 permission-audit).
4. Запрещённые манёвры (не «как обойти запрет», а «как жить без него») перечислены
   в инструкциях агентов: не прятать sleep-циклы в файлы, не поллить асинхронные
   Task, Grep/Read вместо bash-циклов.

## 6. Что остаётся человеку (ревизия docs/03 §4)

Сужаем до продуктовых решений — всё операционное забирает фабрика:

- Чинит баги приложения; двигает карточки `Fixed / Rejected / Intended`; отвечает
  в `## Обсуждение`.
- Утверждает P0/P1-кейсы (`→ Approved`) — можно с борды (D12). P2/P3 —
  авто-approve (уже допущено docs/03 §2).
- Разбирает `Blocked` и позиции из `state/escalations.md`.
- Утверждает изменения стратегии и большие рефакторинги фреймворка.
- Раз в день читает дайджест; раз в неделю — ревизия permission-audit и flaky-карантина.

Всё остальное — включая пересборку APK, перезапуск окружения, обновление борды,
ответы на типовые вопросы по багам, обновление тестов под intended-изменения и
обновление документации QA — фабрика делает сама.
