# 12 — Runbook: приём нового проекта фабрикой

Статус: **шаблон** (живёт при Этапе 5 docs/09 «подготовка к
мультипроектности»; правки — по мере накопления опыта второго
проекта). **НЕ бут-документ** — не входит в перечень Session Start
(docs/HANDOFF.md), boot-бюджет его не считает (docs/09 «boot-диета»).

Назначение: повторяемый заход фабрики на новый SUT (system under
test) — что из текущей машинерии переносится «как есть», что
переписывается заново, и в каком порядке. Источник — фазовая карта
docs/09-improvement-plan.md Часть I (пройдена AO3 целиком 2026-07-02
→ 07-21) и разведка полноты фабрики `recon-new-project-readiness-0721`
(routing-log, принята Lead'ом 2026-07-21).

## 1. Принцип разделения: ядро vs SUT-слой

Ядро фабрики — статусные машины, журналы, гейты, политика ярусов —
не знает, что тестируется. SUT-слой — всё, что описывает КОНКРЕТНОЕ
приложение (экраны, локаторы, сборка, устройство). Перенос нового
проекта = скопировать ядро без изменений, переписать SUT-слой с нуля.

Страж шва — `scripts/tests/test_rules_agent_parity.py`: держит
паритет `state/rules.yaml` ↔ агентов ↔ схем; если правка ядра ломает
этот паритет, тест красный независимо от того, какой SUT подключён.

| Компонент | Переносимость | Что делать при новом проекте |
|---|---|---|
| `state/rules.yaml` (машина правил конвейера) | Полная — синтаксис и структура (правила/pre_steps/skip_statuses) не знают об Android/AO3 | Копировать как есть; SUT-специфика может появиться только в `when`-условиях НОВЫХ правил, если проект их заведёт |
| `schemas/transitions.yaml` + `schemas/*.schema.yaml` | Полная — статусные машины и поля артефактов (bug/test-case/run/charter) универсальны | Копировать; расширять поля только по факту нехватки, не заранее |
| `scripts/log_append.py` и журналы (routing-log, orchestrator-log) | Полная | Копировать без изменений |
| pre_steps `stale_locks.py`, `sla_sweep.py`, `board_inbound.py` | Полная — работают с фронтматтером/статусами артефактов (test-cases/bugs/runs/charters), заголовки не содержат Android/AO3-упоминаний (сверено заголовочно) | Копировать без изменений |
| pre_step `build_watch.py` (сборка новой версии SUT) | Нет — жёстко Android/Gradle/APK (см. §4) | Переписать шаг целиком под механизм сборки/публикации нового SUT |
| Скилл `/qa-loop` и политика маршрутизации CLAUDE.md (ярусы scout/builder/critic/Lead) | Полная — про модели и роли, не про приложение | Копировать; CLAUDE.md-канон КОМАНД внутри — частично проектный, см. §4 |
| Борда (проекция `board/` из артефактов) | Полная — `board_sync.py` проецирует артефакты, не приложение | Копировать; известный баг sync-виджета (кириллица, upstream IstiN/trackstate, docs/09 п.9) — общий для любого проекта, не блокирует |
| `.claude/agents/*.md` — роли конвейера | Частичная — структура промпта (DoD, witness, эскалация) переносима; device-канон внутри — нет | См. §4 «SUT-агенты» |
| `framework/config`, `framework/core` | Частичная (~40% по разведке, см. §4) | Core почти не трогать (docs/02 §3: «core ничего не знает о приложении»); config — переписать capabilities/settings |
| `framework/screens`, `framework/web`, `framework/steps`, `data/`, `tests/` | Нет — SUT-слой (~60% по разведке) | Писать с нуля под новый SUT |
| `scripts/env.ps1`, `scripts/tasks.ps1`, `scripts/doctor.py`, ~7 доп. скриптов обвязки | Нет | См. §4 «env-скрипты» и «doctor» |

## 2. Данные разведки (принять как карту, не как факт без проверки)

`recon-new-project-readiness-0721` оценил: ~11 скриптов обвязки
проектной специфики; 4–6 агентов с device-каноном в промптах;
~60% фреймворка — SUT-слой. Точечная перепроверка этим раном
(builder, 2026-07-21):

- **doctor.py — 13 проверок подтверждено** (`scripts/doctor.py:82-164`):
  5 класса `core` (venv-python, `state/rules.yaml`, `state/sla.yaml`,
  `state/app-under-test.yaml`, `schemas/transitions.yaml`) + 8 класса
  `run` (java, adb, AVD, node/npx, appium, gradlew.bat,
  app-under-test git-глубина, APK). Оценка «13 проверок» точна.
- **Агенты с device-каноном — фактически ШИРЕ оценки «4–6».** Grep по
  `.claude/agents/*.md` на `adb|emulator|AVD|Get-Device` даёт 10 файлов
  с совпадением, но бо́льшая часть — это ОДИН и тот же переносимый
  абзац (правило 6 CLAUDE.md, «env-негатив требует сверки»,
  продублированный в промпт каждого агента дословно — сам этот абзац
  портируется без изменений, меняется только путь к `env.ps1`/
  `tasks.ps1`). Агенты с ГЕНУИННО SUT-специфичным содержимым сверх
  этого абзаца: `test-runner.md`, `test-automator.md`,
  `test-maintainer.md` (ссылки на `framework/web/selectors.py` и
  `app-under-test/.../ao3_bridge.js`, `.claude/agents/test-maintainer.md:26-27`),
  `test-reviewer.md`, `fix-verifier.md`, `exploratory-tester.md`,
  `failure-analyst.md` — это 7, не 4–6. Расхождение зафиксировано,
  не молчаливо принято (см. отчёт). Basis включения `failure-analyst`
  — НЕ по цитате абзаца (его grep-совпадения — тот же переносимый
  абзац, что у отсеянных `builder`/`critic`/`scout`), а по связности
  воркфлоу: его шаг 1 читает Allure-артефакты Android/Appium-прогона
  (logcat, page source, screenshot — `failure-analyst.md:21`), а
  вердикт `APP_CHANGED`/`APP_BUG` требует git-истории именно
  `app-under-test/` (`failure-analyst.md:24-30`) — оба привязаны к
  этому SUT и его пайплайну, переписываются при смене SUT так же, как
  у явно SUT-специфичных агентов.
- **~60% фреймворка — подтверждено структурой** docs/02-framework-architecture.md
  §2: слои `screens/`, `web/`, `steps/`, `data/`, `tests/` (SUT) против
  `core/`, `config/` (портируемое ядро framework); пропорция на глаз
  по числу файлов слоя, не пересчитывалась построчно — это оценка
  разведки, не измеренный факт.

## 3. Фаза 0 нового проекта — чек-лист риск-спайков

Аналог спайков A/B/C, которыми AO3 закрыла свою Фазу 0 (docs/09:82,
таблица «Открытые вопросы фазовой карты»: «~~WEBVIEW-контекст~~,
~~mitm replay~~, ~~run-as сидинг~~ — ✅ спайки A/B/C (Фаза 0)»).
Решения этого раздела — входные данные для Фазы 1 (каркас) и
`framework/config/`.

1. **Тип SUT и automation-интерфейс.** Нативный / гибрид / web /
   API-only? Для AO3 выбор зафиксирован в docs/02-framework-architecture.md:5-12
   («Appium 2 + UiAutomator2», обоснование — единственный зрелый стек
   для гибрида Compose+WebView). Другой SUT — другой стек (Espresso,
   Playwright, чистый REST-клиент) — решение владельца, не по
   умолчанию Appium.
2. **Эталон истины: PROJECT.md-аналог или код.** Урок класса BUG-001
   (docs/09-history.md:31: расхождение подписей вкладок Library с
   PROJECT.md) — документ-эталон устаревает, истина = код
   приложения; при новом проекте закладывать это правило В стратегию
   с первого дня, не ждать первого расхождения.
3. **Доступность интеграционного/API-слоя** (принцип Этап 5 п.3,
   docs/09:179-182) — решается ЗДЕСЬ, не позже: есть ли у SUT
   API-слой ниже UI, доступен ли он для тестов в обход интерфейса.
   Для AO3 — закрыт запретом менять код приложения (docs/01, таблица
   исключений); для нового проекта может открыться (например,
   тестовый бэкенд с seed-эндпоинтом) — если так, слой встраивается
   уже в Фазу 0, не пристраивается потом.
4. **Скоуп security/privacy smoke.** Решение 07-14 для AO3 —
   минимальный (статическая инспекция манифеста + adb, docs/09:178).
   Новый проект переиспользует методику AO3 как образец (цель
   Этапа 5 п.2), но конкретный скоуп — снова решение владельца per
   project, не автокопия.
5. **Live vs replay стратегия** (mitm-класс). Для AO3 — mitmproxy
   record/replay (docs/02:10, §core/mitm.py); для проекта без
   HTTP-слоя (чисто нативное приложение с локальной БД) эта спайка
   может не понадобиться вовсе — фиксировать явным «н/п» в отчёте
   Фазы 0, не молчать.
6. **Риски среды.** Аналог R-03 Cloudflare bot-check AO3 (docs/09:81,
   открытый вопрос фазовой карты — «жёсткий бот → live = canary с
   ручной сессией»): у нового SUT — свой риск среды (rate-limit,
   капча, платный API-лимит), искать его в Фазе 0, не находить в
   проде.

Выход Фазы 0 — по образцу Части I плана: все спайки пройдены (или
явно помечены «н/п»), гипервизор/окружение выбраны и задокументированы
(для AO3 — WHPX, docs/09:22).

## 4. Чек-лист адаптации по компонентам

Каждый пункт: что заменить → где лежит → чем детектится забытая
адаптация.

### env-скрипты

- **`scripts/env.ps1`** — пути `JAVA_HOME`/`ANDROID_HOME`/
  `ANDROID_AVD_HOME`/`PATH` (`scripts/env.ps1:2-7`) целиком
  проектные (жёсткий `$root = "D:\AO3_tests"`, Android SDK). Новый
  проект без Android — файл переписывается под свой рантайм
  (например, только `PATH` для Node/Playwright).
  Детектор забытой адаптации: `doctor.py` (класс `core`/`run`)
  падает FAIL на первом же прогоне, если пути ведут не туда — не
  тихая деградация.
- **`scripts/tasks.ps1`** — функции `Start-Emulator`,
  `Clear-EmulatorStaleLocks` (жёсткий путь
  `tools\avd\ao3_test_api34.avd`, `scripts/tasks.ps1:21`),
  `Install-App`, `Wait-PackageServiceReady`, `Get-Device`
  (`scripts/tasks.ps1:10-233`, полный список функций) — все про
  Android-эмулятор конкретно. `Invoke-Pytest`/`Show-Report` (обвязка
  над pytest/Allure) переносимы почти без правок — меняется только
  `$venv`/рабочий каталог.
  Детектор: `Get-Device` даёт однозначный DEVICE/NO DEVICE — если
  канон CLAUDE.md (правило 6 «Дисциплина команд») не адаптирован под
  новый проект, agent-промпты продолжат ссылаться на несуществующую
  `tasks.ps1`-функцию и это всплывёт первым же `command not found`.
- **`scripts/board.ps1`** — механизм (`Sync-Board`, `scripts/board.ps1:8-15`)
  переносим; проектный только `$root`.

### doctor-проверки (13 штук, `scripts/doctor.py:82-164`)

SUT-специфичные (переписать/удалить при смене стека):
`AVD ao3_test_api34` (`doctor.py:43,113-116` — имя AVD захардкожено),
`gradlew.bat (app-under-test)` (`doctor.py:127-128` — предполагает
Android/Gradle сборку), `app-under-test git-глубина` (`doctor.py:135-151`
— предполагает git-репозиторий приложения рядом), `APK по apk_path`
(`doctor.py:153-162` — Android-артефакт).

Условно-специфичные (часть общего Android+Appium стека, а не только
AO3; при другом мобильном стеке на Appium — переносимы, при смене
драйвера — нет): `java (JAVA_HOME)`, `adb (ANDROID_HOME)`,
`node + npx.cmd`, `appium (tools/appium)`.

Переносимые без изменений (класс `core`): venv-python + pytest/yaml,
`state/rules.yaml`, `state/sla.yaml`, `state/app-under-test.yaml`,
`schemas/transitions.yaml` — эти проверки не знают о приложении.

Детектор забытой адаптации: `doctor.py` эскалирует в
`state/escalations.md` при FAIL (`doctor.py:167-176,192-195`) —
непереписанная SUT-проверка на новом проекте либо ложно зеленеет
(проверяет несуществующий путь как отсутствующий — WARN, не молчание),
либо честно падает FAIL с понятной причиной в detail-строке.

### build_watch (сборка новой версии SUT)

`scripts/build_watch.py` — целиком SUT-специфичный pre_step (в отличие
от трёх остальных pre_steps — см. §1): `git fetch origin` в
`app-under-test/` (`build_watch.py:122`), затем `gradlew.bat
assembleDebug` (`build_watch.py:223`), путь артефакта
`APK_REL = "app/build/outputs/apk/debug/app-debug.apk"`
(`build_watch.py:47`), запись `apk_sha256` в `state/app-under-test.yaml`
(`build_watch.py:193`). Для нового SUT переписывается ВЕСЬ шаг «откуда
взять свежую сборку и что считать её артефактом/хэшем» — это не
Android-специфика внутри общего механизма, это и есть механизм: у
web-проекта «сборка» может значить `git pull` + деплой на staging,
у API-проекта — публикацию образа контейнера с тегом-хэшем.
Детектор забытой адаптации: первый прогон pre_steps `/qa-loop` на
новом проекте — `build_watch` либо падает на `git fetch`/`gradlew.bat`
для несуществующего `app-under-test/`, либо (в `--dry-run`) молча не
находит новых коммитов и не обновляет `state/app-under-test.yaml`
— второе опаснее (тихое отсутствие сборок), поэтому детектор требует
явного смок-прогона с `--no-escalate` снятым.

### SUT-агенты (device-канон в промптах)

По перепроверке §2 — 7 агентов несут содержимое сверх переносимого
абзаца о env-негативе: `test-runner.md`, `test-automator.md`,
`test-maintainer.md`, `test-reviewer.md`, `fix-verifier.md`,
`exploratory-tester.md`, `failure-analyst.md`. Менять в них: любые
пути к экранам/локаторам/сборке SUT (пример — `test-maintainer.md:26-27`
ссылается на `framework/web/selectors.py` и `app-under-test/.../ao3_bridge.js`
— оба имени существуют только в этом проекте). НЕ менять: структуру
промпта (DoD-секция, требование witness, правило эскалации, класс-а-не-
-экземпляр) — она из ядра.
Детектор: `.claude/agents/*.md` не имеет собственного парного теста
(в отличие от `rules.yaml`) — забытая ссылка на несуществующий файл
всплывёт только на первом реальном диспатче агента (сломанный Read/Grep
в его логе). Кандидат будущей работы — не решать здесь (non-goal этого
runbook): шаблонизация самих агентных промптов.

### framework/requirements.txt

Часть пинов — выбор стека, не универсальный набор: `Appium-Python-Client>=5.3.1`
(`framework/requirements.txt:1`) и `mitmproxy>=11` (`framework/requirements.txt:5`)
привязаны к Appium/Android+live-replay стеку — web-SUT на Playwright не
несёт первую строку вовсе, SUT без HTTP-перехвата (см. §3 п.5) — вторую.
`pytest`/`allure-pytest`/`pytest-rerunfailures`/`pyyaml` (строки 2-4,6)
универсальны для этого класса фреймворка независимо от SUT. Правку
пина — только вместе с обновлением этого файла тем же ходом
(CLAUDE.md «Дисциплина команд» п.5, CLAUDE.md:442-443). Детектор
забытой адаптации: `doctor.py` проверка `venv-python + pytest/yaml`
(class `core`) ловит несовпадение venv/requirements при первом
прогоне, но НЕ ловит лишний непронумерованный Appium-пин на проекте
без Appium — это ловится только ревью самого файла.

### Расписание и удалённая публикация борды

- **Heartbeat-задача.** Имя `AO3-QA-Heartbeat` (docs/09:65-66) и её
  интервал (2 ч) — проектные: заведена Windows Task Scheduler-задачей
  под этим именем, создана ВЫКЛЮЧЕННОЙ. Новый проект заводит свою
  задачу под своим именем (коллизия имён между проектами на одной
  машине иначе перезапишет/спутает расписание). Детектор: `scheduled-tasks`
  инструментарий (`list_scheduled_tasks`) не найдёт задачу под старым
  именем на новом проекте — либо забытая адаптация всплывает явным
  отсутствием, либо (хуже) старая задача с прежним именем случайно
  продолжает дёргать СТАРЫЙ репозиторий — сверять цель задачи, не
  только факт её существования.
- **Публикация борды наружу.** GitHub Pages на публичном `origin`
  (docs/05-board.md:80-83, «Как смотреть — по URL (GitHub Pages)»,
  PAT `contents:write`) — remote и права проектные. Детектор: `Sync-Board`
  коммитит и пушит (`scripts/board.ps1`) — на новом репо без
  настроенных Pages/PAT пуш либо упадёт по правам, либо уйдёт молча в
  репозиторий без включённого Pages (борда закоммичена, но URL не
  отдаёт страницу) — при адаптации проверять оба конца, не только
  локальный коммит.

### allowlist / permissions

Пост-адаптационная сверка — механизм `/permission-audit` (скилл в
реестре сессии): прогнать после первого полного захода на новом
проекте, чтобы новые пути/команды (другой `env.ps1`, другой AVD-имя
или его отсутствие) не плодили волну permission-запросов молча.
Не проектируется здесь — это отдельный запуск скилла, не пункт
однократной адаптации.

### Борда

Проекция (`board_sync.py` строит `board/` из `test-cases/`, `bugs/`,
`runs/`, `state/`) переносима без изменений — она не знает содержимого
артефактов, только их статусные поля из схем. Известный дефект
sync-виджета (падение на кириллице в коммитах, upstream
IstiN/trackstate, docs/09:139-142 «п.9») — общий для любого проекта,
использующего тот же форк борды; не блокер переноса, контент борды
работает и без sync-виджета.

## 6. Порядок захода (= фазовая карта docs/09 Часть I)

1. **Фаза 0 — окружение и риск-спайки.** Чек-лист §3 этого документа.
   Выход: спайки пройдены/помечены «н/п», гипервизор и окружение
   выбраны и задокументированы.
2. **Фаза 1 — каркас фреймворка.** Слои по docs/02 §2 созданы (пустые
   или минимальные), зависимости слоёв (docs/02 §3:
   `tests → steps → screens|web → core → config`) соблюдены с первого
   коммита. Выход (по образцу AO3, docs/09:23-24): P0 smoke-набор
   дважды подряд зелёный.
3. **Фаза 2 — тест-дизайн.** Стратегия (аналог docs/01) + реестр фич +
   фичевая трассабилити. Выход: все P0/P1-области с Approved-кейсами.
4. **Фаза 3 — автоматизация P0/P1.** Конвейер в бою (F1-гейт,
   lifecycle B3/B4 — уже перенесены из ядра, §5). Выход: P0 100%,
   P1 ≥ 80%, полный regression в согласованном тайминге.
5. **Фаза 4 — агентная оркестрация.** Агенты конвейера адаптированы
   по §4 «SUT-агенты»; параллельно не изобретаются новые роли без
   нужды — реестр `.claude/agents/` переносится, редактируются только
   те 7 (или больше — как выяснится по факту), что несут device-канон.
6. **Фаза 4.5 — тёмная фабрика.** Из четырёх pre_steps (docs/09:61-62)
   три переносятся из ядра как есть: `stale_locks`, `sla_sweep`,
   `board_inbound` (см. §1) — специфичная часть только в порогах SLA и
   расписании, не в самом механизме. Четвёртый, `build_watch`, —
   SUT-специфичен и переписывается целиком (см. §4 «build_watch»); то

   см. §4 «doctor-проверки») — переносятся только его `core`-проверки.
7. **Фаза 5 — эксплуатация.** Регрессия/canary по расписанию, ревизии
   стратегии, `/permission-audit`.

Критерий выхода каждой фазы — по образцу Части I плана docs/09: явный
измеримый пункт, не «на глаз готово».

## 7. Открытые вопросы мультипроектности

Секция честности — кандидаты решений владельца, НЕ решаются этим
runbook:

- **Один репозиторий на проект vs монорепо фабрики.** Текущий AO3 —
  один репо на одну фабрику+SUT. Второй проект: отдельный клон этого
  репо (дублирование ядра) или общий репо-механизм с несколькими
  SUT-слоями внутри? Оба варианта имеют разную цену сопровождения
  ядра (дубли рассинхронизируются; общий репо усложняет `rules.yaml`
  условиями по проекту) — решение вне скоупа этого документа.
- **Шаблонизация ядра** (вынос переносимой части §1/§5 в отдельный
  репо-шаблон, из которого проекты форкаются) — снизила бы дублирование
  при варианте «один репо на проект», но сама требует поддержки и
  версионирования шаблона. Не спроектировано.
- **Шаблонизация SUT-агентов** (упомянуто в §4 как некандидат этой
  работы) — если device-канон вынести в общий сниппет с параметрами
  вместо копирования абзаца в 7+ промптов, паритет-тест можно
  расширить и на агентов; сейчас нет.

Ни один из трёх вопросов не имеет ответа «по умолчанию» — каждый
ждёт слова оператора при заведении второго проекта.
