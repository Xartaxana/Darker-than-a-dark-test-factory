# AT-BUG-021 — эмулятор крашится mid-test на live Sort&Filter форме (кандидат-сиблинг AT-BUG-016)

## Окружение
- `emulator-5554`, WHPX-гипервизор, Appium 2 + UiAutomator2.
- Live-навигация на `archiveofourown.org/tags/Fluff/works` (реальная тяжёлая
  страница AO3) через `open_live_sort_filter_form_relationship_ready` /
  `open_live_listing` — та же страница, что фигурирует в `bugs/AT-BUG-016.md`
  (TC-040, теперь Fixed) и её эксплицитно названный, но НЕ подтверждённый
  сиблинг `browser_screen.py` живая work-страница.

## Суть долга

2 идентичных env-класс отказа подряд на одном и том же шаге (docs/06 §5
критерий fail-fast достигнут):
1. `WebDriverException: disconnected: not connected to DevTools` внутри
   `driver.get()`/`switch_to.context()` (`open_live_listing`).
2. Немедленно вслед — `adb: device 'emulator-5554' not found` для ЛЮБОГО
   дальнейшего действия той же сессии.
3. `Get-Device` -> `NO DEVICE` (позитивная сверка, не голый env-негатив).
4. Appium `/status` -> 200 ОБА раза — сам Appium-сервер жив, деградация
   именно на уровне устройства/эмулятора.
5. `Start-Emulator` поднимал эмулятор заново успешно оба раза — но
   следующий же live-навигационный вызов ронял его снова (не единичный
   сбой, воспроизводимо на одном и том же классе действия).

Ни одного РЕАЛЬНОГО теста-фейла за всю сессию batch C — вся написанная
логика/локаторы (include/exclude relationship-чекбоксы через `<dd>`, не
`<ul>` как предполагало имя переменной в `ao3_bridge.js`; идемпотентность
save-filter-button) кода не задета, инфраструктура (`selectors.py`,
`sort_filter_form_page.py`, `browser_steps.py`, `test_ao3_selectors.py`)
оставлена в дереве рабочей, `arch_check.py` 0/0.

`bugs/AT-BUG-016.md` (Fixed) диагностировал и устранил КОНКРЕТНЫЙ путь
qemu-краша (`0xc0000005`) для ОДНОЙ фикстуры (`sort_filter_form.mitm`,
TC-040) — форвардящей live-контент из-за неполной self-contained записи.
Этот баг (AT-BUG-021) бьёт по ТОЙ ЖЕ странице, но через ДРУГОЙ путь
(`open_live_listing`/`open_live_sort_filter_form_relationship_ready` —
`@pytest.mark.live` navigate, не replay-фикстура вообще, форвард не
применим как причина). Сигнатура отказа тоже ИНАЯ (DevTools disconnect +
adb device not found, не qemu Windows Event Log `0xc0000005` — не
подтверждено НИ намеренной сверкой Event Log, ни отсутствием). Общее —
только сама тяжёлая live-страница и WHPX-эмулятор под нагрузкой её
рендера. Это может быть (а) ТОТ ЖЕ класс системной хрупкости под тяжёлым
live-рендером, не до конца устранённый узким фиксом AT-BUG-016, ЛИБО (б)
независимая причина (напр. Appium DevTools-сессия отваливается по

_Witness (Windows Application Event Log, метод AT-BUG-016; система TZ
Romance Standard Time = UTC+2, Event Log в локальном времени)._

**Сверка часов (важно — окно в манифесте было помечено «местного
времени» ошибочно).** Routing-журнал ведётся в UTC (naive): моё
собственное событие `delegated` записано `09:53:21`, тогда как текущий
UTC на момент диагностики `09:54:43` — совпадает. Artefact-таймстемпы
«Z» = UTC (краши AT-BUG-016 процитированы как локальные
`03:04:32`/`03:14:33` = UTC `01:04:32`/`01:14:33`, согласуются с
нарративом того бага, закрытым `01:52:53Z`). Окно инцидента batch C
в routing-журнале: `delegated 03:54:02` → `rejected 04:15:28` — это
**UTC** (реальное локальное окно 05:54–06:15). В общей шкале (UTC) в
это окно попадают РОВНО ДВА qemu-краша.

**Два краша В ОКНЕ (Event ID 1000, Application Error):**
- local `06:07:01` / **UTC `04:07:01`** — `qemu-system-x86_64.exe`,
  faulting module `unknown`, exception `0xc0000005`, offset
  `0x...1dc7726dcb4`, PID `0x23BC`, путь
  `D:\AO3_tests\tools\android-sdk\emulator\qemu\windows-x86_64\qemu-system-x86_64.exe`,
  report `726b69ca-f5f7-4df8-9c8e-48f899bcbe01`.
- local `06:09:33` / **UTC `04:09:33`** — тот же образ и сигнатура,
  faulting module `unknown`, exception `0xc0000005`, offset
  `0x...11c031a1cb4`, PID `0x94C` (ДРУГОЙ процесс), report
  `37af11c5-d400-4402-8e58-53aee2f67e5a`.

Оба внутри окна `03:54:02–04:15:28` UTC. Разные PID и разные
start-time процесса = это ДВА отдельных qemu-процесса: первый упал →
`Start-Emulator` поднял новый qemu (новый PID) → второй упал. Это
буквально совпадает с нарративом бага («Start-Emulator поднимал заново
оба раза, следующий live-навигационный вызов ронял снова»). `rejected`
записан `04:15:28` UTC — ~6 мин после второго краша (написание отчёта +
заведение бага). Класс атрибуции чист: параллельный AT-BUG-018 того
окна был уже escalated к `02:13:01` UTC (текущий его прогон — сейчас,
~09:xx UTC, не в этом окне); счёт совпадает (2 отказа = 2 краша).

**Ответ на вопрос сигнатуры (п.2 диспатча).** Дуэт «disconnected: not
connected to DevTools» + немедленный «adb: device not found» —
СЛЕДСТВИЕ смерти qemu, не признак живого процесса. Оба канала
обслуживаются гостём поверх adb-порта эмулятора: при исчезновении
процесса CDP/DevTools-эндпойнт и adb-устройство отваливаются
ОДНОВРЕМЕННО. `Appium /status 200` при этом ожидаем — Appium-сервер —
хостовый процесс, независим от qemu. `Get-Device: NO DEVICE`
(позитивная сверка) + необходимость `Start-Emulator` = процесса не
было, не «adb-сервер икнул при живом qemu». Живой-qemu класс (обрыв
adb-сервера, таймаут CDP-сессии) ОПРОВЕРГНУТ: при нём процесс бы жил,
а Event Log позитивно фиксирует его access-violation-краш дважды.

**Кластер (п.1 диспатча): 11 крашей `0xc0000005` за 4 дня**, все
`qemu-system-x86_64.exe` / module `unknown`, все разные PID:
07-15 ×2 (`16:07:05`/`16:10:06` local — запись `sort_filter_form.mitm`,
AT-BUG-006 инкремент 2), 07-18 ×5 (`07:22:05`..`07:41:57` local — сессия
AT-BUG-016 + воспроизведение критиком `07:41:57`), 07-19 ×2 (`03:04:32`/
`03:14:33` local — B4-ремедиация AT-BUG-016) + 07-19 ×2 (эти два, batch
C). Все привязаны к одной тяжёлой live-странице `/tags/Fluff/works`.

**Код-слой (п.3 диспатча) — единый класс подтверждён.**
`open_live_sort_filter_form_relationship_ready` → `open_live_listing`
(`browser_steps.py:410`) делает `driver.get(<живой archiveofourown.org>)`
в WebView-контексте — реальный рендер тяжёлой страницы целиком (внешние
CSS/JS/шрифты/картинки), да ещё в Cloudflare-retry-петле (повторные
`driver.get` = повторные тяжёлые рендеры). Replay-сиблинги TC-079/081/083
идут через `open_sort_filter_form_relationship_ready` → `open_listing`
на самодостаточной `.mitm` (после фикса AT-BUG-016 — без live-forward),
рендера живой страницы нет. «Та же тяжёлая страница, разные транспорты»
= в точности ось краша. Faulting module `unknown` + `0xc0000005` —
характерно для падения в динамически-сгенерированном коде TCG/GPU-
эмуляции (тот же вывод, что в AT-BUG-016).

**Связь с AT-BUG-016 (уточнение, важно для рычага фикса).** Закрытие
AT-BUG-016 заявляло, что qemu-хрупкость «пропала полностью» после
полной сетевой изоляции, а остаток — детерминированный локаторный баг
`_find_pill` (AT-BUG-019). Это верно для ТОГО пути, но узко: фикс
AT-BUG-016 не УСТРАНИЛ qemu-хрупкость под тяжёлым live-рендером — он
УБРАЛ триггер (заменил live-forward на self-contained запись). На пути
`open_live_listing` триггер убрать НЕЛЬЗЯ: TC-078/080/082 —
`@pytest.mark.live` ПО ДИЗАЙНУ (canary против реального AO3). Поэтому
исходный (первичный) механизм краша AT-BUG-016 здесь всплывает снова —
это тот же класс, а не новый.

**Хрупкость ВЕРОЯТНОСТНАЯ, не строго детерминированная (класс, D-0043).**
Тот же `open_live_listing` в batch A (TC-066/068/070) и batch B
(TC-072/074/076) отработал 3× зелёным каждый (witness routing-журнала)
на живых страницах — крашей не было. Batch C на `/tags/Fluff/works`
(самая тяжёлая — tag-листинг) упал дважды подряд и добил fail-fast.
Значит хрупкость поражает ВЕСЬ live-canary-класс вероятностно, с бóльшим
риском на самых тяжёлых страницах; batch C не уникально сломан — он
просто дважды подряд вытянул краш-лотерею. Это согласуется с наблюдением
B4 AT-BUG-016 «1 pass из 3, не строго детерминировано по шагу» и с
названной там (не расширенной) осью SIBLING_MAP «рендер РЕАЛЬНЫХ live
AO3-страниц в эмуляторном WebView». AT-BUG-021 — ПЕРВЫЙ подтверждённый
witness'ом инстанс на этой оси помимо самого AT-BUG-016.

**Причина: qemu-краш (`0xc0000005`, access violation в TCG/GPU-коде),
не DevTools/adb-класс — потому что** Event Log позитивно фиксирует ДВА
краша qemu-процесса (разные PID) внутри окна инцидента, `Get-Device: NO
DEVICE` подтверждает исчезновение устройства, а DevTools/adb-дуэт есть
ожидаемое совместное следствие смерти процесса, обслуживающего оба
канала.

**Рекомендация ремедиации (решение — за координатором должного яруса;
роль критика ограничена диагнозом, D-0043).** Класс — не
DevTools-таймаут, поэтому ретрай/переподключение/увеличение CDP-таймаута
НЕ помогут (к мёртвому процессу переподключиться нельзя — неверный

1. **Диагностический прогон под софт-рендером** `-gpu
   swiftshader_indirect` (вариант 3 AT-BUG-016, так и не проверенный —

   дизайну). Прямой рычаг против GPU-компоновочного краша: прогнать
   TC-078/080/082 несколько раз под софт-рендером и смотреть Event Log
   на `0xc0000005`. Это инфра-правка (`tasks.ps1`/параметры эмулятора)
   с широким blast radius — Lead-ярус, отдельный контролируемый прогон,
   НЕ в этом read-only диспатче (устройство занято AT-BUG-018).
2. **Если софт-рендер не убирает краш** — честный вывод: live-половина

   TC-079/081/083 несут ТЕ ЖЕ R-02 контракты детерминированно и зелены.
   Предложить оператору: replay-сиблинги — в штатные regression/p0
   проходы; live-версии (TC-078/080/082) — реже/вручную/вне
   блокирующего гейта (их ценность — ловля дрейфа реального AO3 —

3. Облегчение live-цели (тег с меньшим числом работ = легче листинг при
   той же форме Sort&Filter) — паллиатив: снижает вероятность, не
   устраняет класс; форма-контракт привязан к листинговой странице, так

Инфраструктура batch C (`selectors.py`/`sort_filter_form_page.py`/
