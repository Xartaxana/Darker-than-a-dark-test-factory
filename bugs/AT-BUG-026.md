# AT-BUG-026 — qemu краш (0xc0000005) под тяжёлым live-рендером AO3, sibling AT-BUG-016-live

## Окружение
Не зависит от сборки приложения: долг тестовой системы (`type: test_debt`,
`debt_kind: broken_environment`). Обнаружен при D1-верификации AT-BUG-024
(fix-verifier, mode=verify), диагностирован critic-входом (правило 3б,
ESC-006 в `state/escalations.md`, resolved).

## Суть долга

`qemu-system-x86_64.exe` крашится (`0xc0000005`, access violation, модуль
`unknown` — динамически генерируемый код, TCG-JIT/GPU-эмуляция) во время
исполнения теста, использующего ТЯЖЁЛЫЙ live-рендер реальной страницы
`archiveofourown.org` в эмуляторном WebView под UiAutomator2. Краш убивает
ВЕСЬ эмулятор целиком (`Get-Device` → `NO DEVICE`, без осиротевших
процессов) — не единичный тест падает, а рушится весь `-m p0` прогон.

**Подтверждённый крашер:** `test_save_filter_button_idempotent_live`
(TC-082, `framework/tests/canary/test_ao3_selectors.py:431-450`) —
`open_live_listing(driver, rb.SORT_FILTER_FORM_URL)` грузит
`archiveofourown.org/tags/Fluff/works` вживую. 2/2 независимых прогона
`-m p0` (fix-verifier, 2026-07-22) упали на этом тесте, оба раза с
идентичным крахом. Тот же exception (та же тяжёлая страница) уже
задокументирован в `state/escalations.md` ESC-002 / `bugs/AT-BUG-016.md`
(TC-040/TC-083) — тот фикс сузил ТОЛЬКО `.mitm`-replay-flow до
самодостаточного (устранил forward живого контента); LIVE-вариант теста
по определению не имеет replay-изоляции (обязан ходить в реальный AO3) —
этот краш вне охвата фикса AT-BUG-016, sibling, не новый корень
(правило 9 CLAUDE.md, ось уже названа в AT-BUG-016 Lead-записи
2026-07-18: «поверхность рендер РЕАЛЬНЫХ live AO3-страниц в эмуляторном
WebView»).

**Флейковость (не детерминированный регресс):** test-maintainer тем же
кодовым состоянием (фикс AT-BUG-024 уже применён) получил
`46 passed, PYTEST_EXIT=0` — включая этот самый тест — в отдельном
прогоне 2026-07-22T11:40. Т.е. краш не гарантирован каждый прогон;
исторический Get-WinEvent (critic, 60 событий) показывает рекуррентность
15.07–22.07 (16 крашей идентичного класса — тот же бинарник, тот же
exception code 0xc0000005, модуль `unknown`; РЕАЛЬНЫЕ offset'ы различны в
каждой записи, т.к. код в `unknown`-модуле генерируется динамически —
см. разбор critic в `bugs/AT-BUG-024.md` «Верификация» 2026-07-22T14:05).

## Критерий готовности (Fixed) — ПЕРЕСМОТРЕН полным Lead 2026-07-23

Прежний критерий «полный `-m p0` без краха ×3 подряд» ОТМЕНЁН как
статистически неинформативный для вероятностного отказа: при остаточной
частоте краша p за прогон вероятность пройти ×3 без реального фикса
велика (p=0.2 → 51%), а стоимость каждой попытки ~25 мин device-времени;
одновременно гейт ложно блокируется несвязанными флейками (прогоны 2-3
попытки (а) выше — RED без краша). Прецедент класса — урок D-0052
OS-репо: детерминированную проверку заменяет ИЗМЕРЕНИЕ, не повторение
дорогого интегрального прогона.

Новый двухчастный критерий:
1. **Baseline ДО ремедиации (диагностический цикл critic ESC-007):**
   replay-изолированный краш-рейт-цикл — только heavy-render итерации
   `/tags/Fluff/works` (Sort&Filter form) в петле, дефолтный GPU
   (`swiftshader_indirect`), подтверждённо достижимый прокси, детектор —
   Event Log `Id=1000, 0xc0000005` с позитивным контролем формы запроса
   (F-34). Даёт: узкий воспроизводитель (минуты вместо 25-минутного
   `-m p0`), измеренную частоту ДО. Если K=30 итераций (масштаб соука
   AT-BUG-021) крашей не дают — live-вариант того же цикла
   (`open_live_listing`), т.к. узкая гипотеза «только live» стоит
   непроверенной (ESC-007).
2. **Fixed = измеренное подавление на воспроизводителе:** 0 крашей за
   K итераций цикла, где K ≥ 3× средняя историческая наработка на краш
   этого класса (по Event Log: 16 крашей / плотность прогонов 15-22.07 —
   точное K фиксирует диспатч диагностики по факту baseline). Полный
   `-m p0` — ОДИН подтверждающий прогон (роль — смок на отсутствие
   побочек ремедиации, НЕ статистический тест).

Сопутствующие решения Lead (2026-07-23, разбор эскалации):
- **Контейнмент отделён от лечения:** вероятностный qemu-краш не должен
  убивать вердикт всего конвейерного прогона. Механизменный кандидат
  (очередь Lead, правило 10 при реализации): bounded auto-restart-resume
  p0-прогона при NO DEVICE mid-run с пометкой ENV_ISSUE в run-отчёте.
  Это НЕ критерий Fixed этого бага — отдельный механизм.
- **alt-класс `-gpu host`** (NO DEVICE без WER-сигнатуры, п.1 эскалации):
  named-not-covered — `host` не дефолт и после 1/4 не кандидат; детектор
  заведения отдельного test_debt — первый NO DEVICE без `0xc0000005` под
  ДЕФОЛТНЫМ GPU (2 инцидента test-automator 19:35 не квалифицированы:
  Event Log не проверялся).
- **Демотирование на `sort_filter_form.mitm`-семью НЕ расширять** —
  ратифицирую вердикт critic ESC-007 (replay-witness'а краша нет ни
  одного; setup-фейл до yield тяжёлый DOM не рендерил).
- П.3 эскалации (облегчение разметки .mitm) — остаётся преждевременным
  до подтверждённого replay-witness (позиция critic подтверждена).

**Попытка (а): полный `-m p0` под `-gpu host`.**

Дефект инструментовки (честно, не маскирую): прогоны 1-2 шли внутри
одного большого PowerShell-скрипта, логированного через `Start-Transcript`
— поймал факт вывода `Write-Host`/нативный stderr, но НЕ захватил
собственно STDOUT дочернего `python.exe`/`pytest` (progress-точки и
итоговую summary-строку) в фоновом (без реальной консоли) режиме запуска
— осталась только строка `PYTEST_EXIT=N`. Не значит, что прогон не
состоялся — `$LASTEXITCODE` считывается из того же процесса, надёжен;
значит только, что для прогонов 1-2 нет дословного текста summary (только
итоговый код). Прогоны 3-4 велись отдельными фоновыми job'ами БЕЗ этого
дефекта (харнесс сам захватывал `stdout` в файл) — их текст приведён выше
дословно.

**Побочная находка тулинга (не по коду репозитория, докладываю честно):
фоновый Bash-job имеет собственный жёсткий потолок времени жизни
(~50-60 минут по наблюдению), НЕ связанный с `Wait-Process`-ожиданием
самой сессии.** Первый большой скрипт (env-подъём + цикл до 6 попыток
`-m p0` внутри ОДНОГО фонового вызова) был принудительно убит харнессом
(`status: killed`) на середине 3-й попытки, хотя `Wait-Process`
блокировался штатно и процесс был жив по `Get-Process` до самого убийства
— это НЕ признак сломанной среды приложения (не по критерию docs/06 §5),
а ограничение среды ОРКЕСТРАЦИИ самой сессии агента. Переключился на
паттерн «один `Invoke-Pytest` = один фоновый job» (env-подготовка —
отдельными короткими foreground-вызовами), это устранило проблему
инструментовки: прогоны 3-4 дали полный дословный текст.

**Проверка Event Log для прогона 4 (позитивный контроль формы запроса,
CLAUDE.md п.6/класс F-34).** `Get-WinEvent -FilterHashtable
@{LogName='Application'; Id=1000; StartTime=(Get-Date).AddHours(-2)}` →
**0 событий** в окне прогона 4. Контрольный запрос БЕЗ сужения окна
(`-AddHours(-24)`) нашёл ровно 2 известных краша ИЗ ДО-сессионного периода
(14:59:24 и 15:32:37 — те самые, что уже задокументированы в
`bugs/AT-BUG-024.md` D1-верификации) — форма запроса доказанно рабочая,
пустой результат в узком окне не промах. Также проверены `System`-лог
(TDR/GPU-driver события — 0 совпадений) и `%LOCALAPPDATA%\CrashDumps`
(пусто) и список процессов (`qemu-system*`/`emulator*` — ни одного, не
завис, действительно исчез). **Вывод: устройство пропало БЕЗ
характерной сигнатуры `0xc0000005`, которую до сих пор давал КАЖДЫЙ
задокументированный краш этого класса (AT-BUG-016/021/024/026-found_in).
Это либо (i) редкая вариация того же класса без WER-отчёта (напр. крах
до попадания в отчёт, либо иной код исключения), либо (ii) отдельный,
НЕ качественно охарактеризованный побочный эффект бэкенда `host`
(GPU-passthrough на хост-видеокарту, в отличие от softwar-рендера
`swiftshader_indirect`).** Не берусь утверждать которое — недостаточно
данных для причинного заявления (калибровка №3 CLAUDE.md): «вклад `host`
в ЭТОТ конкретный отказ не исключён», не более.

**Оговорка по причинности (важно, честно).** 1 чистый из 4 прогонов под
`host` — ХУЖЕ, чем можно было ожидать по узкому соуку AT-BUG-021 (26/26
чистых на ИМЕННО TC-078/080/082). Это не противоречие: узкий соук
проверял ТОЛЬКО 3 теста на ОДНОЙ странице; полный `-m p0` прогоняет 46
тестов на РАЗНЫХ страницах/фикстурах в течение ~25 минут — экспозиция
шире, и по вероятностной модели самого класса (см. `AT-BUG-021`,
«хрупкость ВЕРОЯТНОСТНАЯ, поражает ВЕСЬ live-canary-класс, не только
конкретный тест») риск накапливается на весь прогон, не гасится тем, что
узкая тройка была стабильна отдельно. **Причинное заявление, которое
ДАННЫЕ этой сессии ПОДТВЕРЖДАЮТ:** `-gpu host` НЕ даёт статистической
гарантии «полный `-m p0` без краха» — 1/4 недостаточно для вывода
обратного намного меньше, чем нужно было бы для уверенности (тот же
недостаток, что честно называла сама сессия AT-BUG-021).
