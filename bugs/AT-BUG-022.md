# AT-BUG-022 — нет observability-примитива, различающего «switchTab сработал» от «switchTab no-op», когда цель — вкладка-0

## Окружение
- Не зависит от сборки приложения: долг тестовой системы (`type: test_debt`,
  `debt_kind: missing_fixture` — по конвенции этого репо категория «недостающая
  тестовая инфраструктура/фикстура», не обязательно данные; используется тем же
  образом, что и AT-BUG-004/005/006/010 для наблюдательных/инфраструктурных
  пробелов). `framework/screens/browser_screen.py:255-270` (докстринг класса),
  `framework/steps/browser_steps.py::switch_to_tab`/`assert_active_tab_url`/
  `swipe_close_tab`, фикстура `tab_markers.mitm`.

## Суть долга

`test-cases/tabs/TC-084.md` (P1, R-08, «тап по чипу неактивной вкладки
активирует её») в attempt 1 использовал established-приём TC-026
(«reduce-to-one»: `switch_to_tab` до цели → закрыть остальные вкладки свайпом
→ `assert_active_tab_url(target)`) для доказательства, что тап по чипу
вкладки 0 (Home) сделал её активной. Критик отклонил на приёмке (см.
«Обсуждение» ниже, находка полностью процитирована) — приём не различает
два состояния:

1. `switchTab(0)` реально сработал, вкладка 0 стала активной;
2. `switchTab(0)` — no-op (индекс активной вкладки не изменился, остался 2).

В ОБОИХ случаях итоговое наблюдение через reduce-to-one идентично: `swipe_
close_tab` закрывает вкладки 1 и 2 (жест бьёт по конкретной позиции чипа
независимо от того, какая вкладка помечена активной — это НАТИВНЫЙ свайп по
физическому View, не завязан на active-флаг), остаётся ровно вкладка 0; т.к.
chromedriver и без того НАВСЕГДА прилипает к вкладке-0 (первый когда-либо
созданный WebView — см. докстринг `browser_screen.py:255-270`,
`driver.current_url` всегда читает именно её), `assert_active_tab_url(HOME)`
пройдёт ЗЕЛЁНЫМ, даже если тап был бы полным no-op. Reduce-to-one доказывает
только «контент вкладки 0 существует и равен Home» — факт, тривиально истинный
и ДО тапа (вкладка 0 никогда не закрывалась, её URL не менялся сценарием), а
не «тап сделал вкладку 0 активной».

Сравнение с TC-026 (`test_tabs.py::test_long_press_link_opens_background_tab_
without_switching`, строки 313-315), который использует ТОТ ЖЕ приём для
ДРУГОГО утверждения: там цель — фоновая вкладка НЕ на позиции 0 (`switch_to_
tab(driver, 1)`, затем `swipe_close_tab(driver, 0)`, затем `assert_active_
tab_url(first_work.url)`), и заявляемый факт — «контент фоновой вкладки
корректно ЗАГРУЗИЛСЯ» (content-existence), а не «переключение сделало её
активной» (TC-026 отдельно и корректно проверяет обратное: активной остаётся
ИСХОДНАЯ вкладка, через ту же sticky-к-нулю особенность, где вкладка-0 и есть
исходная). Перенос приёма на утверждение «тап переключил активность» —
ошибка переноса контекста: конфаунд решает исход независимо от переключения.

**Класс шире одного кейса.** Reduce-to-one структурно не может отличить «X
стал активным в результате действия» от «X — единственный выживший после
закрытия остальных» ни для какой цели: закрытие вкладок идёт по позиции чипа
нативным жестом, не по active-флагу, поэтому вкладка-цель ВСЕГДА переживёт
свёртку независимо от того, была ли она когда-либо реально активной. Для
цели vs НЕ-0 (как в TC-026) это не проблема, т.к. утверждение TC-026 —
content-existence, а не active-статус. Для утверждения ИМЕННО «стала
активной» (какое требует TC-084) reduce-to-one в принципе неприменим — нужен
ДРУГОЙ observability-примитив, читающий текущую активность независимо от
закрытия соседних вкладок.

**Гипотеза направления (критик, НЕ подтверждена — только названа, требует
эмпирической проверки на эмуляторе с 2+ живыми вкладками, из которых цель НЕ
вкладка-0):** нативные жесты (тап/свайп/скролл) бьют по физически видимому
View (`activeContainer` — см. докстринг `browser_screen.py:255-270`), не
подвержены прилипанию chromedriver. Возможный различающий приём: после тапа
по чипу-цели выполнить НАТИВНЫЙ скролл-жест по видимой области контента (без
захода в WEBVIEW-контекст), затем прочитать `window.scrollY` вкладки-0 через
sticky WEBVIEW-контекст (`execute_script`). Если тап реально сделал
вкладку-0 активной — она физически отображается, нативный скролл сдвигает
ИМЕННО её DOM, `scrollY` вкладки-0 меняется. Если тап — no-op (вкладка-0
не видна) — нативный скролл бьёт по физически отображённой (другой) вкладке,
DOM вкладки-0 не трогается, `scrollY` вкладки-0 остаётся прежним (0). Это
единственный известный на сейчас способ читать «текущая активность» БЕЗ
свёртки числа вкладок к одной, но он НЕ проверен эмпирически — данная
сессия (test-designer) не располагала инструментом для запуска
эмулятора/pytest в этом ходе (только Read/Glob/Grep/Write/Edit), поэтому
гипотеза не могла быть подтверждена или опровергнута в attempt 2.

Заблокирован: TC-084 (P1) — единственный кейс на фичу
`browse-tab-switch-active`, центральное утверждение Then («тап по
неактивному чипу делает вкладку активной») не имеет доказанного способа
автоматической проверки.

## Критерий готовности (Fixed)

- Установлен и эмпирически подтверждён на эмуляторе (live прогон, минимум
  2 живые вкладки, из которых цель тапа НЕ вкладка-0 — иначе конфаунд не
  проверен) observability-механизм, реально различающий «switchTab сработал»
  от «switchTab no-op» — либо гипотеза со scrollY-асимметрией выше, либо
  другой найденный приём.
- `test-cases/tabs/TC-084.md`: «Заметки для автоматизации» и (при
  необходимости) операционализация Then обновлены на подтверждённый
  механизм; ссылка на этот баг снята/переведена в исторический факт.
- Хотя бы один прогон TC-084 на подтверждённом механизме зелёный (реальное
  доказательство различения — напр. контрольный прогон с намеренно
  «сломанным» no-op сценарием НЕ обязателен для Fixed, но механизм обязан
  логически различать оба состояния, не просто наблюдать один пример).
- Smoke без регресса.

## Анализ

Класс — «наблюдательный примитив для активности вкладки отсутствует»:
ближайший сосед по духу — AT-BUG-004/005/006/010 (инфраструктурный пробел,
блокирующий automation конкретных кейсов), отличие в том, что здесь не
хватает не данных/фикстуры, а именно СПОСОБА НАБЛЮДЕНИЯ состояния через
существующий стек Appium/chromedriver. Приоритет — P1 (блокирует P1-кейс,
ядровая навигация мультитаба), device-зависимый (эмпирическая проверка
гипотезы требует эмулятора).

## Обсуждение

**2026-07-19 — критик (приёмка TC-084, attempt 1, /qa-loop):** единственная
блокирующая находка (остальное принято без замечаний) — `assert_active_tab_
url(HOME)` как центральный Then для сценария «тап по чипу неактивной вкладки
0 (Home) активирует её» не различает рабочий `switchTab` от no-op, т.к.
Home и так вкладка-0 (сама точка chromedriver-прилипания, см. докстринг
`browser_screen.py:255-270`) — `current_url` ВСЕГДА читает вкладку-0
независимо от переключения в UI. TC-026 использует тот же приём корректно
для ДРУГОГО утверждения (content-existence фоновой вкладки), перенос на
«переключение сделало вкладку активной» — ошибка, конфаунд решает исход.
Указано возможное (не мандатное, не проверенное самим критиком) направление:
асимметрия нативных жестов vs. sticky chromedriver-контекст через
`window.scrollY`.

**2026-07-19 — test-designer (retry attempt 2, task_id
browse-tab-switch-active):** заведено по итогам разбора находки. Гипотеза
критика логически состоятельна (нативные жесты бьют по физически видимому
`activeContainer`, независимо от sticky-прилипания chromedriver к вкладке-0
— тот же факт из докстринга, что объясняет исходный конфаунд), но требует
эмпирической проверки на живом эмуляторе (минимум 2 живые вкладки, цель ≠
0), которую эта сессия не могла выполнить — доступный инструментарий хода
ограничен Read/Glob/Grep/Write/Edit, без Bash/adb/pytest. `docs/01-test-
strategy.md` §9 «tabs: переключение активной вкладки тапом по чипу»
возвращён в `needs-design` (ссылка на TC-084 снята из designed-записи, факт
проектирования и находка задокументированы прозой) — метка `designed` не
может стоять, пока центральное утверждение кейса не имеет проверенного
способа наблюдения. `TC-084.md` НЕ переведён из `Review`, «Заметки для
автоматизации» обновлены: снята ложная пометка «блокера нет», добавлена
явная blocking-пометка со ссылкой на этот баг. Следующий проход B4
(test-maintainer, эмулятор доступен) — эмпирически проверить гипотезу
scrollY-асимметрии на живом прогоне (2+ вкладки, цель ≠ 0, напр. Given
TC-084 буквально: активна 2, тап по чипу 0) и либо подтвердить механизм
(тогда обновить TC-084 и вернуть `designed`), либо найти другой
различающий приём, либо задокументировать здесь, что различающего
наблюдения в текущем стеке Appium/chromedriver для этого класса
утверждений не существует вовсе (в таком случае Then TC-084 потребует
пересмотра — решение выше test-designer, в очередь Lead).
