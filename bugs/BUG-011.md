# BUG-011 — Restore пропускает работы молча при гонке с async-сканом папки загрузок

## Окружение
- Эмулятор ao3_test_api34 (API 34), сборка debug 1.10 (versionCode 11)
- Версия APK: app-debug.apk от 2026-07-02T02:39:46

## Суть проблемы

`SettingsViewModel.setDownloadFolderUri` (SettingsScreen.kt:523–529) синхронна в UI-части
(обновляет `uiState`, preferences, `downloadRepo.customFolderUri`), но запускает асинхронную
корутину `scanForDownloads(silent=true)` на IO-диспетчере. Возврат из SAF-пикера (тап ALLOW
в DocumentsUI) НЕ гарантирует, что асинхронный скан уже завершился.

Если пользователь положит HTML-файл резервной копии с известным `ao3_id` в целевую папку
загрузок, а ЗАТЕМ СРАЗУ в Settings выберет эту же папку как «Download Folder» через SAF-пикер,
то:

1. **Синхронная часть** `setDownloadFolderUri` завершается немедленно → пользователь видит
   диалог подтверждения скана закрыться и вернуться на Settings.
2. **Асинхронный скан** продолжает работу в `Dispatchers.IO`, создавая stub-записи в Room
   для всех .html-файлов в папке (включая положенный HTML-файл резервной копии).
3. **Пользователь открывает** Restore → выбирает файл резервной копии → нажимает Restore.
4. **`importFromUri`** (в `BackupRepository`) обрабатывает импорт, но КОГДА ОН ПРОВЕРЯЕТ
   `ao3Id in existingIds`, асинхронный скан уже успел добавить stub-запись с этим `ao3Id`
   в базу (ответ на вопрос, где параллельное чтение базы).
5. **Результат**: Restore видит `ao3Id` уже существующим и молча **пропускает** эту работу как
   дубликат. Пользователь видит диалог вроде: **«Restored 0 works, 0 filters (1 works already existed)»**
   — вместо ожидаемого восстановления рейтинга, комментария и тегов из бэкапа.

Пользователь НЕ получает никакого предупреждения о том, что его данные из резервной копии
не были восстановлены; поведение выглядит как молчаливое игнорирование.

## Шаги воспроизведения (Given-When-Then)

**Given:**
- Library пуста
- Существует HTML-файл резервной копии, содержащий одну работу с известным `ao3_id` и непустыми
  рейтингом, комментарием или тегами

**When:**
1. Положить HTML-файл в целевую папку загрузок (например, `/sdcard/Download/backup.json`)
2. Открыть Settings → Download Folder → выбрать ту же папку через SAF-пикер → тап ALLOW
3. Немедленно (БЕЗ задержки) открыть Settings → Backup & Restore → Restore from backup
4. Выбрать положенный HTML-файл → тап Restore

**Then (ожидается):**
- Диалог показывает: **«Restored 1 works, 0 filters. Also relinked 0 and added 0 downloaded file(s).»**
- Работа появляется в Library с восстановленным рейтингом, комментарием и тегами

**Actual (факт):**
- Диалог показывает: **«Restored 0 works, 0 filters (1 works already existed).»**
- Работа НЕ появляется в Library; рейтинг/комментарий/теги НЕ восстановлены

## Класс проблемы

Асинхронная гонка (race condition) между:
- Асинхронной корутиной `scanForDownloads(silent=true)`, запущенной при смене папки загрузок
- Синхронным импортом данных в `Restore from backup`

Скан успевает создать stub-запись в Room ДО того, как Restore начнёт обрабатывать файл.
Это НЕ фликер (не редкий, не таймаут) — воспроизводится детерминированно при таком порядке действий.

## Повлиянные компоненты

- `SettingsViewModel.setDownloadFolderUri` (SettingsScreen.kt:523–529)
- `SettingsViewModel.scanForDownloads` (запущенная асинхронно)
- `BackupRepository.importFromUri` (проверяет `ao3Id in existingIds`)

## Классовая полнота

Проблема распространяется на любой сценарий, где:
- Файл размещается в SAF-папке загрузок **рядом по времени** со смены самой папки
- И этот файл должен быть обработан **независимо** от async-скана папки

Ремонт на уровне инфраструктуры: `setDownloadFolderUri` могла бы дождаться завершения
скана перед возвратом (например, дождаться Idle/Done на `scanDownloadsState` перед
завершением функции), но это требует изменения контракта функции и может нарушить UI-отзывчивость.

Обход на уровне приложения: явно синхронизировать Restore с сканом или сбросить базу перед
восстановлением (но это не штатное поведение для пользователя).

## Анализ (рекомендация разработчику)

Класс: **асинхронная гонка при составе операций**, не ошибка одного компонента.

Severity: **minor** — условие репродукции требует предварительного размещения файла и немедленного
выполнения Restore (маловероятно в обычном сценарии использования, где пользователь обычно:
(а) выбирает папку один раз и забывает, или (б) восстанавливает резервную копию, которую
положил туда заранее, минут за несколько до того, как попросить скан).

Однако, если пользователь попытается восстановить работу ИЗ ТОЙ ЖЕ папки, где она уже лежит как
скачанный файл, результат будет молчаливо потеряными данными (рейтинг, комментарий, теги).
